# 知识体系建设与文档工作流

## 1. 文档目标

这份文档不是介绍某一项技术，而是定义你后续写技术文档、整理旧笔记、分析项目架构时的一套统一工作流。

核心目标：

- 职责分离
- 结构清晰
- 长期可维护
- 适合全栈开发持续积累

---

## 2. 你真正需要沉淀的，不是“文章”，而是“知识资产”

一个程序员的知识库，至少应区分下面五类资产：

| 类型 | 作用 | 典型内容 |
|------|------|----------|
| 概念型文档 | 建立术语与基础认知 | 定义、背景、适用场景 |
| 原理型文档 | 建立底层理解 | 机制、状态变化、数据流、设计动机 |
| 架构型文档 | 建立系统观 | 分层、模块关系、关键链路、边界 |
| 工程型文档 | 指导落地实施 | 配置、部署、开发、排障、优化 |
| 复盘型文档 | 积累经验与决策 | 问题分析、取舍原因、踩坑总结 |

如果这五类内容混在一篇文档里，笔记会越来越乱。  
所以后续每次写文档，都要先判断：**这篇文档属于哪一类主职责**。

---

## 3. 建议的知识分层

建议以后围绕同一个主题，按下面四层理解与沉淀：

### 3.1 概念层

回答：

- 它是什么
- 它解决什么问题
- 它属于哪一类技术

常见文档名：

- `xxx 基础概念.md`
- `xxx 快速入门.md`
- `xxx 是什么.md`

### 3.2 原理层

回答：

- 它为什么这样设计
- 它内部怎么运转
- 核心机制如何协同

常见文档名：

- `xxx 原理解析.md`
- `xxx 核心机制.md`
- `xxx 数据流详解.md`

### 3.3 架构层

回答：

- 它由哪些模块组成
- 各层职责边界是什么
- 模块之间如何协作

常见文档名：

- `xxx 架构分析.md`
- `xxx 模块拆解.md`
- `xxx 全链路流程.md`

### 3.4 工程层

回答：

- 怎么落地
- 怎么开发
- 怎么部署
- 怎么排查问题

常见文档名：

- `xxx 实战指南.md`
- `xxx 部署指南.md`
- `xxx 问题排查.md`

## 4. 一篇高质量技术文档的标准结构

无论是技术分析、项目分析还是原理解析，都尽量遵循同一套骨架：

1. 背景与问题定义
2. 核心概念
3. 原理或架构总览
4. 关键模块与职责
5. 核心流程 / 数据流 / 调用链
6. 工程实践或落地方式
7. 常见问题 / 局限 / 风险
8. 总结与延伸阅读

这套骨架的好处是：

- 读起来稳定
- 后续容易补充
- 适合拆成多篇子文档
- 适合让 AI 接着补写

---

## 5. 主题级文档组织建议

围绕一个主题，建议优先形成“1 个主文档 + 若干子文档”。

### 5.1 主文档职责

主文档负责：

- 给出总览
- 定义主题边界
- 提供阅读顺序
- 链接到子文档

例如：

- `Kafka核心原理与应用指南.md`
- `Dify项目深度分析参考规范.md`
- `RAG全链路优化设计与实现指南.md`

### 5.2 子文档职责

子文档负责深挖某一方面：

- 单独的原理
- 单独的模块
- 单独的问题排查
- 单独的对比分析

例如：

- `kafka消息机制原理图_v1.md`
- `redis-假断线问题分析.md`
- `Dify应用管理与发布机制完整解析.md`

### 5.3 判断规则

如果内容回答的是“整体是什么”，放主文档。  
如果内容回答的是“某个局部怎么工作”，放子文档。

## 6. 旧笔记重构 SOP

当你手上有一篇写得比较乱的旧笔记，按这个顺序处理。

### 6.1 第一步：诊断

先判断它的问题属于哪类：

- 内容重复
- 结构跳跃
- 概念和原理混写
- 原理和工程实践混写
- 缺少标题层级
- 缺少结论和总结
- 信息有效，但表达太口语化

### 6.2 第二步：切块

把原文拆成若干块：

- 背景块
- 概念块
- 原理块
- 架构块
- 实践块
- FAQ 块

### 6.3 第三步：归位

将每个内容块放回标准结构里。

### 6.4 第四步：补缺

补充最常缺的四样东西：

- 为什么这样设计
- 整体结构图
- 关键流程图
- 工程中的意义

### 6.5 第五步：定归属

最后再判断它应该放到哪个目录、叫什么名字、是否需要拆成多篇。

---

## 7. 多篇笔记合并 SOP

当同一个主题已经写散了，不要继续往旧文档里硬塞，先做归并。

### 7.1 合并顺序

1. 找出主题相同的文档
2. 标记重复内容
3. 按“概念 -> 原理 -> 架构 -> 工程 -> FAQ”重排
4. 冲突观点先保留并标注
5. 形成一份主文档
6. 把太长的章节拆成子文档

### 7.2 合并后的结果应该是

- 主文档更稳
- 子文档更专
- 重复内容减少
- 阅读路径更清楚

## 8. 推荐的文档命名规则

### 8.1 文件名尽量表达“主题 + 职责”

推荐：

- `Kafka核心原理与应用指南.md`
- `FastAPI项目架构分析.md`
- `Redis持久化机制深度解析.md`
- `Dify工作流域深度分析.md`
- `OpenClaw网关服务部署指南.md`

不推荐：

- `1.md`
- `notes.md`
- `整理一下.md`
- `学习记录.md`

### 8.2 常见职责后缀

| 后缀 | 用途 |
|------|------|
| `技术分析` | 面向整体技术认知 |
| `原理解析` | 面向机制与底层逻辑 |
| `架构分析` | 面向系统结构与模块关系 |
| `实战指南` | 面向落地实施 |
| `部署指南` | 面向安装部署 |
| `问题分析` | 面向故障或异常现象 |
| `选型对比` | 面向方案比较 |
| `FAQ` | 面向问答式复习 |

## 9. 推荐的写作与整理工作流

### 9.1 当你遇到一个新主题

建议顺序：

1. 先做一篇总览文档
2. 再拆原理文档
3. 再拆架构或模块文档
4. 最后补工程实践和 FAQ

### 9.2 当你阅读一个新项目

建议顺序：

1. 先写项目总览
2. 再写模块拆解
3. 再写关键流程
4. 最后写二次开发指南

### 9.3 当你排查一个线上问题

建议顺序：

1. 先记录现象
2. 再记录排查路径
3. 再记录根因和修复
4. 最后回填到对应主题文档

你的目标不是单点突破，而是形成跨层认知。

## 11. AI 协作时的职责分离

以后你和 AI 协作，建议明确区分下面四种任务，不要一口气全混在一个提示词里。

| 任务类型 | AI 角色 | 输出物 |
|----------|---------|--------|
| 分析类 | 架构师 / 研究者 | 技术分析、架构图、模块说明 |
| 整理类 | 编辑 / 信息架构师 | 重构后的 Markdown、目录、提纲 |
| 写作类 | 技术写作者 | 正式文档、FAQ、教程 |
| 复盘类 | 事故分析者 / reviewer | 排查文档、根因分析、改进建议 |

这就是你说的“职责分离”。  
本质上不是人为搞复杂，而是让 AI 每次只做好一件事。

## 13. 一周一次的知识库维护动作

建议每周做一次轻量整理，而不是等到很乱再重构。

### 每周固定动作

1. 新增的零散笔记统一归类
2. 主题重复的文档做合并标记
3. 把临时记录补成正式标题
4. 给重点主题补 FAQ
5. 给核心文档补 Mermaid 图
6. 标注“待补充 / 待验证 / 待拆分”的内容

### 每月固定动作

1. 选一个主题做一次主文档重构
2. 清理命名混乱的文件
3. 给重点领域补一份知识地图
4. 检查哪些文档已经不适合继续追加，应该拆分

## 14. 文档质量自检清单

每次写完或整理完一篇文档，至少问自己下面这些问题：

- 这篇文档的主职责是什么
- 它更偏概念、原理、架构还是工程
- 是否存在重复内容
- 是否说明了“为什么这样设计”
- 是否说明了“它怎么运转”
- 是否说明了“工程上有什么意义”
- 是否有清晰标题结构
- 是否适合下次继续追加
- 是否应该拆为主文档和子文档

如果这几个问题答不上来，说明文档还没整理到位。

---

## 15. 最终落地建议

你后续不需要再追求“每次都写一篇大而全的文章”。  
更有效的做法是：

1. 先确定文档职责
2. 再选合适模板
3. 先产出主结构
4. 再逐步补原理、架构、实践、FAQ
5. 最后把零散内容持续回填到知识体系

这样做的结果是：

- 新技术能更快看懂
- 旧笔记能持续变干净
- 项目分析会更像工程文档
- 你的知识库会逐渐从“记录”变成“体系”

这才是真正能帮助你在全栈开发路上扎实前进的方式。
