## 项目分析
分析项目代码 {项目路径}，请编写一份技术分析文档，涵盖以下内容：
1. **项目定位**：一句话说清楚这是什么项目，解决什么问题
2. **整体架构**：架构风格（分层/DDD/微服务等）、技术栈、核心模块及职责
3. **核心执行链路**（重点）：选取最典型的核心功能，用完整执行链路将所有模块串联起来（后端描述请求流，前端描述交互流，CLI/SDK 描述调用流）；若使用 DDD，需额外说明各层职责边界与领域模型设计
4. **关键设计决策**：为什么这样设计，而不仅是描述是什么
5. **FAQ 12个以上**：每题详细解答，覆盖基本原理、设计决策、实际应用、性能优化四类问题

质量要求：文档结构清晰，分章节组织；内容详实准确，便于读者理解；关键设计决策重点分析「为什么这样设计」而非仅描述「是什么」；流程图准确反映组件关系和关键流程

格式要求：Markdown，充分使用 Mermaid 图辅助说明。
**必须包含以下图（每项至少一张）**：
- 核心概念：关键术语/概念之间的关系或层级图
- 架构设计：整体系统架构图
- 关键组件：核心组件的结构及依赖关系图
- 功能模块：各模块的职责划分与协作关系图
- 核心执行链路：主流程/调用链的时序或流程图

**可选补充图（遇到以下情况时主动添加，无需逐一覆盖）**：
- 多入口 / 多协议并存（REST + gRPC、CLI + SDK 等） → 用图对比各入口的处理差异
- 插件 / 扩展 / 中间件机制 → 展示扩展点的挂载位置与调用时机
- 状态机 / 事件驱动 / 消息队列流转 → 文字难以清晰表达状态跃迁时
- 跨服务 / 跨层依赖（微服务拓扑、DDD 分层边界、monorepo 包依赖） → 有明显层级或服务边界时
- 部署与运行时环境（容器编排、环境变量注入、构建流水线） → 有具体步骤或配置依赖时
- 关键算法 / 数据结构的处理逻辑 → 步骤超过 3 步且存在分支判断时
- 其他任何「一张图能替代三段文字」的地方
