# 项目分析提示词模板

> 使用说明：替换所有 `{花括号}` 占位符，按需删除可选段落，然后将整段提示词发送给 AI。

---

## 快速替换清单

| 占位符 | 含义 | 示例 |
|--------|------|------|
| `{项目名称}` | 仓库/项目名 | `ollama` |
| `{本地路径}` | 项目在本机的绝对路径 | `E:/github_project/ollama` |
| `{GitHub URL}` | 原始仓库地址（可选） | `https://github.com/ollama/ollama` |
| `{一句话描述}` | 你对该项目的初步认知 | `本地运行大模型的管理工具` |
| `{主语言/技术栈}` | 你已知的技术信息 | `Go、CUDA` |
| `{分析深度}` | 本次重点 | 见下方三档选项 |
| `{关注模块}` | 你最想搞清楚的模块（可选） | `模型加载流程` |

---

## 提示词正文（复制此段使用）

```
## 任务：深度分析 GitHub 项目 {项目名称}

**本地路径**：`{本地路径}`
**仓库地址**：{GitHub URL}（可选）
**我的初步认知**：{一句话描述}
**已知技术栈**：{主语言/技术栈}

---

### 分析深度：{分析深度}

<!-- 三档选择一，删除其余两档 -->

<!-- 档位 A - 全局概览（适合首次接触） -->
> 目标：用最短路径建立完整的系统认知地图，不深入代码细节。

<!-- 档位 B - 架构深潜（适合已有初步认知） -->
> 目标：拆解核心模块职责、数据流向与关键设计决策，理解"为什么这样设计"。

<!-- 档位 C - 聚焦模块（适合专项攻关） -->
> 目标：对 **{关注模块}** 进行端到端的详细分析，追踪从入口到输出的完整执行链路。

---

### 请完成以下分析

#### 1. 项目定位（必须）
- 用 2～3 句话说明：解决什么问题 / 目标用户 / 与同类项目的核心差异

#### 2. 目录结构解读（必须）
- 列出一级、二级目录，标注每个目录的职责
- 用 Mermaid `graph TD` 或文字树形图表示整体模块边界

#### 3. 技术栈全景（必须）
- 列表形式：语言 / 框架 / 关键依赖 / 协议 / 存储方案
- 指出任何非主流或值得关注的技术选型，并说明原因

#### 4. 核心执行链路（必须）
- 选取 1～2 条最重要的业务流程（如：启动流程 / 核心功能调用路径）
- 用 Mermaid 时序图或流程图展示端到端过程
- 标注每一步"为什么这样设计"

#### 5. 关键设计决策（必须）
- 列出 3～5 个架构层面的重要决策
- 每条格式：**决策**：做了什么 → **原因**：为什么这样做 → **权衡**：放弃了什么

#### 6. 模块依赖关系（档位 A 可省略）
- 用 Mermaid 图展示主要模块之间的依赖方向
- 指出循环依赖、高耦合区域或扩展点

#### 7. 数据模型（档位 A 可省略）
- 核心数据结构（struct / class / schema）及其字段含义
- 数据在模块间的流转方式（同步/异步/消息队列/共享内存等）

#### 8. 扩展与插件机制（如有）
- 项目如何支持扩展：插件协议 / Hook / 事件总线 / 配置驱动
- 新增功能的最小改动路径是什么

#### 9. FAQ（至少 6 条）
覆盖以下四类，每条详细解答：
- **基本原理类**：核心机制的底层逻辑
- **设计决策类**：为什么不用 X，而选择 Y
- **实际应用类**：如何在真实场景中使用/二次开发
- **边界情况类**：失败路径、并发问题、资源限制

---

### 格式要求

- 输出 Markdown，适当使用 Mermaid 图辅助说明
- **每张图必须配套文字**：
  - 图前导语（1～3 句）：说明该图的关注焦点和阅读路径
  - 图后要点（视复杂度）：节点超过 8 个或存在复杂依赖时，用 2～4 条要点解释设计决策
- 建议图类型参考：
  - 架构图 / 流程图 → `@mermaid-A-系统认知层.md`
  - 数据模型 / 类层级 → `@mermaid-B-代码深潜层.md`
  - 状态机 / 时序图  → `@mermaid-C-运行时行为层.md`
  - 规划 / 时间线    → `@mermaid-D-规划可视化层.md`
- 语言：中文，术语保留英文原名
- 重点突出"为什么"，不只是描述"是什么"
```

## 按深度选择的精简版

### 档位 A — 5 分钟快速扫描

```
请分析本地项目 {项目名称}（路径：`{本地路径}`）。

我的背景：{一句话描述}，技术栈：{主语言/技术栈}。

输出：
1. 项目定位（3 句话以内）
2. 目录结构（带职责注释的树形图）
3. 技术栈列表
4. 最重要的 1 条执行链路（Mermaid 流程图）
5. 3 个最关键的设计决策

格式：Markdown，中文，术语保留英文。
```

---

### 档位 B — 架构深潜

```
请深度分析本地项目 {项目名称}（路径：`{本地路径}`）。

我的背景：{一句话描述}，技术栈：{主语言/技术栈}。

分析要求：
1. 项目定位（2～3 句，含与同类项目的核心差异）
2. 目录结构（带职责注释的树形图）
3. 技术栈全景（语言 / 框架 / 关键依赖 / 协议 / 存储，非主流选型请说明原因）
4. 核心执行链路 1～2 条（Mermaid 时序图或流程图，每步标注"为什么"）
5. 模块依赖关系（Mermaid 图，指出高耦合区域或扩展点）
6. 关键设计决策 3～5 条（格式：决策 → 原因 → 权衡）
7. FAQ 6 条（原理 / 设计 / 实践 / 边界各 1～2 条）

格式：Markdown + Mermaid，中文，术语保留英文，重点突出"为什么"。
```

---

### 档位 C — 聚焦模块深挖

```
请深入分析本地项目 {项目名称}（路径：`{本地路径}`）中的 **{关注模块}** 模块。

背景：{一句话描述}

分析要求：
1. 该模块的职责边界：做什么 / 不做什么
2. 入口函数 / 触发条件
3. 完整执行链路（Mermaid 时序图，标注每步的"为什么"）
4. 依赖的其他模块及依赖方向
5. 核心数据结构及字段含义
6. 错误处理与边界情况
7. 如果要在此基础上新增功能，最小改动路径是什么？
8. FAQ 6 条（原理 / 设计 / 实践 / 边界各 1～2 条）

格式：Markdown + Mermaid，中文，术语保留英文。
```
