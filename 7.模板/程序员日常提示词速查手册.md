# 程序员日常提示词速查手册

## 1. 文档目标

这份手册用于统一你在日常开发与学习中最常见的 AI 提问场景，重点覆盖以下任务：

- 技术分析
- 原理分析
- 项目架构分析
- 源码模块深潜
- 笔记整理
- 旧文档重构
- 技术选型对比
- 学习路线与知识体系沉淀

目标不是“写一堆提示词”，而是形成一套可长期复用的工作方法。

---

## 2. 使用原则

### 2.1 三段式输入法

每次提问尽量包含三类信息：

1. **对象**
   你要分析什么，例如 Kafka、FastAPI、Dify、某个仓库、某个模块、某篇旧笔记。
2. **任务**
   你要 AI 做什么，例如重构、分析、归纳、补全、对比、解释、生成目录。
3. **约束**
   你希望输出成什么样，例如 Markdown、Mermaid、面试导向、源码导向、适合新手、保留原信息、不虚构。

### 2.2 输出优先级

优先让 AI 输出下面这些“能直接沉淀为知识资产”的结果：

- 结构化文档
- 分层说明
- 对比表格
- Mermaid 图
- FAQ
- 落地建议
- 待补充项

### 2.3 通用要求

下面这段可以作为多数任务的公共前置约束。

```text
你是一名资深全栈工程师兼技术写作者。
请基于我提供的材料完成任务，要求：

1. 输出为结构清晰的 Markdown 文档
2. 先给整体框架，再展开关键细节
3. 明确区分：背景、概念、原理、架构、流程、实现、问题、总结
4. 不要空泛描述，不要只给定义，要解释为什么这样设计
5. 如果存在多层结构，请明确每一层职责与边界
6. 如果适合，补充 Mermaid 图、表格、FAQ、最佳实践
7. 如果输入材料不完整，请明确标出“已知信息 / 推断信息 / 待确认信息”
8. 保持术语准确，面向程序员，避免营销化表达
```

## 4. 模板 A: 技术架构分析

### 4.1 适用场景

- “帮我分析 Kafka / Redis / FastAPI / Ray / Dify / AgentScope”
- “我想快速建立对某项技术的整体认知”
- “我要写一篇技术分析文档”

### 4.2 提示词

```text
请对【技术名】做一份面向程序员的技术架构分析文档。

目标：
帮助我快速建立对这项技术的整体认知，理解它解决什么问题、核心架构如何设计、关键模块如何协作、适合在什么场景使用。

输出要求：
1. 用 Markdown 输出
2. 文档结构包含：
   - 技术背景与定位
   - 它解决的核心问题
   - 核心概念说明
   - 整体架构分层
   - 关键组件职责
   - 核心数据流 / 请求流 / 调用链路
   - 关键设计权衡
   - 优势、局限与适用场景
   - 与相近技术的区别
   - 实际项目中的落地建议
   - FAQ
3. 如果适合，请补充 Mermaid 架构图
4. 内容要解释“为什么这样设计”，而不仅是“它是什么”
5. 如果有版本差异，请单独说明
```

## 5. 模板 B: 原理深度解析

### 5.1 适用场景

- “Kafka 为什么高吞吐”
- “Redis 持久化原理是什么”
- “Transformer 的注意力机制怎么工作”
- “为什么某项技术能做到某种效果”

### 5.2 提示词

```text
请对【技术 / 机制 / 概念】做一份原理深度解析。

我关注的不是表面介绍，而是：
1. 设计动机
2. 内部机制
3. 数据如何流动
4. 为什么这么设计
5. 这样设计带来的收益与代价

输出结构：
- 背景与问题定义
- 原理的核心目标
- 相关前置概念
- 内部组成
- 运行流程分步骤解析
- 关键算法 / 规则 / 状态变化
- 与其他实现方案的差异
- 常见误区
- 工程中的实际影响
- 面试常问问题

附加要求：
1. 如果涉及数据流或状态变化，请使用 Mermaid 流程图
2. 如果涉及公式或规则，请用程序员能理解的方式解释
3. 不要只讲结论，要讲推导逻辑
```

## 6. 模板 C: 项目架构分析

### 6.1 适用场景

- “帮我分析这个 GitHub 项目”
- “帮我看懂这个仓库的架构”
- “帮我写项目分析文档”

### 6.2 提示词

```text
请基于我提供的项目材料，对【项目名 / 仓库】做一份项目架构分析。

分析目标：
帮助我快速看懂这个项目的定位、架构、模块划分、关键流程、技术栈、目录组织，以及二次开发切入点。

输出结构：
- 项目背景与目标
- 核心问题域
- 整体架构风格
- 项目目录结构解读
- 关键模块拆解
- 模块依赖关系
- 核心调用链 / 请求链 / 数据链
- 技术栈与中间件作用
- 配置、扩展点、插件机制
- 二次开发建议
- 潜在风险与复杂点
- 阅读源码的推荐顺序

要求：
1. 明确区分“业务模块”“基础设施模块”“公共能力模块”
2. 给出 Mermaid 架构图或模块关系图
3. 如果我提供了源码目录，请结合目录一起分析
4. 尽量指出：入口文件、核心抽象、主干流程、扩展机制
```

## 7. 模板 D: 源码模块深潜

### 7.1 适用场景

- “只分析 `core/workflow` 模块”
- “帮我看懂这个 service 层”
- “分析这个类、这个函数、这条调用链”

### 7.2 提示词

```text
请对【模块 / 目录 / 类 / 函数】做源码级深度分析。

我希望你重点回答：
1. 这个模块的职责是什么
2. 它在整个系统中的位置是什么
3. 它依赖谁、被谁调用
4. 关键对象、关键函数、关键数据结构是什么
5. 核心执行流程是什么
6. 哪些地方最值得二次开发时重点关注

输出结构：
- 模块定位
- 输入与输出
- 内部组成
- 关键类 / 函数职责表
- 主流程分步讲解
- 上下游依赖关系
- 设计亮点与隐含约束
- 容易误解的点
- 二次开发建议

要求：
1. 优先用“职责 + 调用链 + 数据流”来解释源码
2. 尽量避免逐行翻译代码
3. 如果合适，请补充 Mermaid 时序图或流程图
```

## 8. 模板 E: 从零生成学习笔记

### 8.1 适用场景

- “给我写一篇 xx 技术学习笔记”
- “我要沉淀成长期可复习的文档”
- “面向初学到进阶整理一份笔记”

### 8.2 提示词

```text
请围绕【主题】写一篇适合程序员长期积累的学习笔记。

目标：
不是一次性科普，而是形成可复习、可扩展、可继续追加的知识文档。

输出结构：
- 一句话定义
- 这项技术解决什么问题
- 核心概念
- 原理说明
- 架构 / 模块 / 流程
- 实际开发中的应用方式
- 常见问题与易错点
- 与相近技术的区别
- 面试问答
- 延伸阅读建议

要求：
1. 风格偏“技术笔记”，不是营销文章
2. 每一节尽量有清晰小标题
3. 关键内容可表格化
4. 重要链路可加 Mermaid 图
5. 最后附“3 分钟速记版总结”
```

## 9. 模板 F: 旧笔记重构

### 9.1 适用场景

- “帮我整理这篇旧笔记”
- “内容有点乱，帮我重组”
- “不要丢信息，但要更清晰”

### 9.2 提示词

```text
请对我提供的旧笔记进行重构整理。

整理目标：
1. 不改变原始主题
2. 尽量保留原有有效信息
3. 删除重复、无序、跳跃表达
4. 补齐缺失的层次结构
5. 把它整理成一篇清晰、可复习、可继续迭代的 Markdown 文档

输出要求：
- 先给出“原笔记存在的问题”
- 再输出“重构后的正式版本”

重构时请按以下规则处理：
- 相同意思的内容合并
- 口语化但有价值的信息可转成正式表述
- 模糊表述尽量改成更准确的技术语言
- 缺少过渡时补充必要的小标题
- 原文信息不足处标注“待补充”
- 不要凭空编造具体事实

建议结构：
- 主题概述
- 核心概念
- 原理 / 架构 / 流程
- 工程实践
- 常见问题
- 总结
```

## 10. 模板 G: 多篇笔记归并为一份体系文档

### 10.1 适用场景

- “这几篇笔记帮我合并一下”
- “同一个主题写散了”
- “我想整理成一份主文档”

### 10.2 提示词

```text
请将我提供的多篇笔记合并为一份主题统一、结构完整的知识文档。

任务要求：
1. 识别重复内容并合并
2. 按合理顺序重排章节
3. 区分基础概念、核心原理、工程实践、扩展讨论
4. 对冲突内容进行标注，不要强行混为一谈
5. 输出一份适合长期维护的主文档

输出结构：
- 文档总览
- 合并后的正式版本
- 重复内容合并说明
- 仍待确认的问题
- 后续可拆分出的子主题

附加要求：
请给出建议的文件命名和目录归属，方便我放入知识库。
```

## 11. 模板 H: 技术选型与对比分析

### 11.1 适用场景

- “Kafka 和 RabbitMQ 怎么选”
- “FastAPI vs Django”
- “Dify / AgentScope / LangGraph 对比”

### 11.2 提示词

```text
请对【方案 A】与【方案 B / C】做一份程序员视角的技术对比与选型分析。

我的关注点：
- 解决的问题是否一致
- 架构思想是否不同
- 使用门槛
- 扩展性
- 性能与复杂度
- 团队适配度
- 适用场景

输出结构：
- 对比结论摘要
- 各方案定位
- 核心能力对比表
- 架构与实现思路差异
- 优势与局限
- 适用团队 / 项目阶段
- 选型建议
- 哪些情况下不建议使用

要求：
1. 避免泛泛而谈
2. 结论必须对应具体场景
3. 最后给出“如果是我当前场景，该怎么选”
```

## 12. 模板 I: 学习路线与知识地图

### 12.1 适用场景

- “我要系统学 xx”
- “如何从原理走到工程实践”
- “帮我规划知识体系”

### 12.2 提示词

```text
请围绕【主题】为一名程序员设计一份系统学习路线。

我的目标是：
不仅知道概念，还要理解原理、看懂架构、会做工程实践，并最终形成自己的知识文档。

请输出：
- 学习目标分层
- 前置知识
- 核心知识地图
- 建议学习顺序
- 每个阶段应该产出什么笔记
- 每个阶段应该完成什么实践
- 容易卡住的点
- 判断是否学会的标准

请按四层来设计：
1. 概念层
2. 原理层
3. 架构层
4. 工程层

最后补充：
- 7 天速学版
- 30 天系统版
- 后续进阶方向
```

## 13. 模板 J: 问题排查与复盘沉淀

### 13.1 适用场景

- “Redis 假死怎么排查”
- “线上问题如何复盘成文档”
- “故障处理后要整理成知识资产”

### 13.2 提示词

```text
请将下面这个技术问题 / 故障案例整理为一份问题排查与复盘文档。

文档目标：
帮助我沉淀问题现象、定位过程、根因分析、修复方案和后续预防措施，便于团队复用。

输出结构：
- 问题背景
- 现象描述
- 影响范围
- 初步怀疑点
- 排查路径
- 根因分析
- 修复方案
- 为什么之前没有发现
- 如何避免再次发生
- 可复用的排查清单

要求：
1. 不只写“发生了什么”，还要写“为什么会这样”
2. 如果存在多种可能原因，请按优先级排序
3. 最后提炼成一份简短经验卡片
```

## 17. 一句话心法

你的提示词不要只问“这是什么”，而要持续逼近下面五个问题：

1. 它解决什么问题
2. 它为什么这样设计
3. 它内部是怎么运转的
4. 它在真实项目里怎么落地
5. 我应该把它沉淀成哪一类知识文档

如果这五个问题能持续问清楚，你的知识体系会越来越扎实，而不是只积累零散结论。
