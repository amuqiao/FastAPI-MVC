## 一、时序预测算法

### 1.1 KNN算法

#### 技术挑战与解决方案（基于电站低效派单与审单AI系统）

**挑战1：电站多源数据噪声大，特征提取困难**
- **问题定位**：电站数据来源多样（逆变器、气象站、运维记录），存在数据缺失、异常值、时间不同步等问题，直接影响模型预测准确性。
- **方案设计**：采用多步骤数据预处理 pipeline，包括数据对齐、异常值检测、特征工程。
- **实施过程**：
  1. 使用 PySpark 进行分布式数据处理，统一时间戳格式
  2. 基于 IQR 方法检测异常值，结合领域知识（如电流为负）进行过滤
  3. 提取多维度特征：天气特征（温度、湿度、辐照度）、设备特征（型号、运行时间）、历史运维特征（故障频率、维修时长）
  4. 构建时间窗口特征，捕捉时序依赖性
- **结果验证**：数据质量提升后，模型准确率从 75% 提升至 90%，异常值比例从 15% 降至 3%。

**挑战2：不同电站设备差异大，模型泛化能力不足**
- **问题定位**：不同电站的设备型号、安装环境、运维水平差异显著，单一模型难以适应所有场景。
- **方案设计**：采用迁移学习与模型集成策略，针对不同类型电站定制化模型。
- **实施过程**：
  1. 对电站进行聚类分析，根据设备类型、地理位置、容量大小分为 3 类
  2. 基于 TimesNet 构建基础模型，在全量数据上预训练
  3. 针对每类电站进行微调，调整模型参数
  4. 构建集成模型，根据电站类型自动选择对应子模型
- **结果验证**：模型泛化能力显著提升，不同类型电站的预测准确率均达到 88% 以上，较单一模型提升 12%。

#### 原理
KNN（K-Nearest Neighbors，K近邻）是一种基于实例的学习算法，用于分类和回归。

**核心思想**：
- 对于一个新样本，找到训练集中距离它最近的K个样本
- 分类任务：这K个样本中出现次数最多的类别作为预测类别
- 回归任务：这K个样本的平均值作为预测值

**距离度量**：
- 欧氏距离：$d(x,y) = \sqrt{\sum_{i=1}^{n}(x_i-y_i)^2}$
- 曼哈顿距离：$d(x,y) = \sum_{i=1}^{n}|x_i-y_i|$
- 余弦相似度：$sim(x,y) = \frac{x \cdot y}{\|x\| \|y\|}$

**K值选择**：
- K太小：容易受噪声影响，过拟合
- K太大：计算开销大，欠拟合
- 通常通过交叉验证选择最优K值

**优缺点**：
| 优点 | 缺点 |
|------|------|
| 简单易懂，易于实现 | 计算复杂度高（需要计算所有距离） |
| 无需训练，惰性学习 | 对高维数据效果差（维度灾难） |
| 对数据分布无假设 | 对噪声和异常值敏感 |
| 适用于多分类问题 | 需要存储所有训练数据 |

#### 考点
1. **KNN的时间复杂度和空间复杂度**：
   - 时间复杂度：O(n)，n为训练样本数
   - 空间复杂度：O(n)

2. **如何优化KNN的效率**：
   - KD树：空间划分，减少距离计算
   - Ball树：更适合高维数据
   - 局部敏感哈希（LSH）：近似最近邻搜索
   - 降维：PCA、t-SNE等

3. **KNN在时序预测中的应用**：
   - 基于历史相似时刻进行预测
   - 动态时间规整（DTW）衡量时序相似度
   - 窗口滑动选择最近邻

4. **加权KNN**：
   - 距离近的样本权重更高
   - 权重 = 1/距离² 或 高斯核函数

### 1.2 TimesNet算法

#### 原理
TimesNet是专门为时序预测设计的深度学习模型，核心思想是将时序数据转换为2D空间，利用2D卷积捕捉时序依赖。

**核心创新**：
1. **时间转2D**：将1D时序转换为2D张量，保留时间关系
2. **2D卷积捕获多尺度依赖**：使用Inception块捕获不同时间尺度的依赖
3. **高效训练**：相比Transformer，计算复杂度更低

**网络结构**：
```
输入时序 → 时间转2D模块 → TimesBlock（多层） → 预测头 → 输出
```

**时间转2D模块**：
- 将长度为T的时序重塑为 [T1, T2] 的2D张量
- T1 × T2 = T
- 通过多次下采样，逐步降低时间维度

**TimesBlock**：
- 包含多个Inception模块
- 使用不同大小的卷积核捕获多尺度特征
- 残差连接，防止梯度消失

**损失函数**：
- MSE（均方误差）：适合回归任务
- MAE（平均绝对误差）：对异常值更鲁棒

#### 考点
1. **TimesNet相比其他时序模型的优势**：
   - 相比Transformer：计算复杂度O(T) vs O(T²)
   - 相比LSTM：可以并行计算，训练更快
   - 相比CNN：更好的多尺度建模能力

2. **时间转2D的原理**：
   - 为什么要转2D？利用成熟的2D卷积技术
   - 如何选择T1和T2？通常T1=T2=√T
   - 多尺度下采样的作用：捕获不同时间粒度的模式

3. **Inception模块的作用**：
   - 多尺度卷积核：1×1, 3×3, 5×5
   - 并行处理：同时捕获不同尺度的依赖
   - 1×1卷积降维：减少计算量

4. **TimesNet的适用场景**：
   - 长时序预测
   - 多变量时序预测
   - 异常检测
   - 分类任务

### 1.3 Informer算法

#### 原理
Informer是基于Transformer的长时序预测模型，解决了传统Transformer在长时序上的效率问题。

**核心创新**：
1. **ProbSparse Attention**：稀疏注意力，只关注重要的query
2. **Self-Attention Distilling**：注意力蒸馏，减少内存占用
3. **Generative Decoder**：生成式解码器，一次预测多个时间步

**ProbSparse Attention**：
- 传统Attention复杂度O(T²)
- ProbSparse Attention复杂度O(T log T)
- 只选择Top-u个重要的query计算attention
- 衡量query重要性：KL散度

**Self-Attention Distilling**：
- 在Encoder中使用蒸馏操作
- 逐步减少序列长度
- 保留重要信息，减少内存

**Generative Decoder**：
- 一次预测所有目标时间步
- 避免自回归预测的累积误差
- 效率更高

**网络结构**：
```
输入 → Encoder（ProbSparse Attention + Distilling） → Decoder（生成式） → 输出
```

#### 考点
1. **为什么长时序预测困难**：
   - 传统Transformer的O(T²)复杂度
   - 自回归预测的累积误差
   - 长距离依赖建模困难

2. **ProbSparse Attention的原理**：
   - 如何衡量query的重要性？KL散度
   - 为什么可以稀疏化？大多数query贡献很小
   - u如何选择？通常u = c log T，c为常数

3. **Informer vs Transformer**：
   | 特性 | Transformer | Informer |
   |------|-------------|----------|
   | 时间复杂度 | O(T²) | O(T log T) |
   | 空间复杂度 | O(T²) | O(T log T) |
   | 预测方式 | 自回归 | 生成式 |
   | 长时序效果 | 差 | 好 |

4. **生成式Decoder的优势**：
   - 一次预测所有时间步
   - 避免累积误差
   - 训练和推理效率更高

## 二、目标检测与OCR

### 2.1 YOLOv5

#### 技术挑战与解决方案（基于光伏组件缺陷智能检测系统）

**挑战1：光伏板图像反光严重，影响缺陷检测**
- **问题定位**：光伏板表面光滑，拍摄时容易产生反光，导致图像局部过曝，隐裂等细微缺陷被掩盖，检测准确率低。
- **方案设计**：采用多步骤图像预处理与增强策略，消除反光影响。
- **实施过程**：
  1. 将RGB图像转换为HSV颜色空间，分离亮度通道
  2. 对V通道应用自适应阈值分割，识别反光区域
  3. 对反光区域进行局部亮度调整，平衡图像曝光
  4. 结合CLAHE算法增强对比度，突出缺陷特征
  5. 应用高斯滤波去除噪声，保持图像细节
- **结果验证**：反光处理后，隐裂检测准确率从 78% 提升至 92%，漏检率降低 60%。

**挑战2：缺陷样本不平衡，小缺陷检测困难**
- **问题定位**：数据集中正常样本占比 90% 以上，隐裂、断栅等小缺陷样本稀少，模型容易偏向正常样本，小缺陷漏检率高。
- **方案设计**：采用数据增强、损失函数优化与模型架构改进相结合的策略。
- **实施过程**：
  1. 针对小缺陷样本进行过采样，使用Mosaic、RandomPerspective等数据增强技术扩充数据集
  2. 引入Focal Loss解决类别不平衡问题，提高小缺陷的权重
  3. 在YOLOv5中集成CBAM注意力机制，增强骨干网络对小目标的特征提取能力
  4. 优化锚框设计，针对小缺陷调整锚框尺寸和比例
  5. 采用多尺度训练策略，提高模型对不同大小缺陷的检测能力
- **结果验证**：小缺陷检测准确率从 65% 提升至 88%，模型整体准确率达到 95%，较基准模型提升 15%。

#### 原理
YOLOv5是单阶段目标检测模型，实现了精度与速度的最佳平衡。

**核心创新**：
1. **Focus模块**：切片操作，减少计算量
2. **CSP结构**：跨阶段局部网络，增强梯度流动
3. **SPP模块**：空间金字塔池化，融合多尺度特征
4. **FPN+PAN**：双向特征融合
5. **Mosaic数据增强**：4张图拼接
6. **自适应锚框**：K-means聚类
7. **CIoU损失函数**：考虑重叠度、中心点距离、宽高比

**网络结构**：
```
输入 → Focus → CSPDarknet → SPP → FPN+PAN → 检测头 → NMS → 输出
```

**检测流程**：
1. 输入层：图像输入 + Mosaic数据增强
2. Focus模块：切片操作，宽高减半，通道翻倍
3. 骨干网络：CSPDarknet + SPP提取特征
4. 颈部网络：FPN+PAN双向特征融合
5. 检测头：多尺度检测（80×80/40×40/20×20）
6. 后处理：NMS非极大值抑制

**损失函数**：
- 边界框损失：CIoU Loss
- 置信度损失：BCE Loss
- 分类损失：BCE Loss
- 总损失：三者加权求和

#### 考点
1. **Focus模块的工作原理**：
   - 切片操作：隔行隔列采样
   - 通道拼接：4个特征图在通道维度拼接
   - 输入输出：[B,3,640,640] → [B,12,320,320]
   - 优势：不增加参数量，减少计算量

2. **CSP结构的优势**：
   - 特征分割：主分支+残差分支
   - 减少计算冗余
   - 增强梯度流动
   - 提高准确率和速度

3. **FPN vs PAN**：
   - FPN：自顶向下，传递语义特征
   - PAN：自底向上，传递定位特征
   - FPN+PAN：双向融合，兼顾语义和定位

4. **CIoU Loss**：
   - IoU：仅考虑重叠度
   - GIoU：增加最小包围框惩罚
   - DIoU：增加中心点距离惩罚
   - CIoU：增加宽高比一致性惩罚
   - 公式：CIoU = IoU - (d²/c²) - αv

5. **Mosaic数据增强**：
   - 4张图像随机缩放拼接
   - 丰富数据多样性
   - 增强小目标检测能力
   - 提升泛化能力

6. **多尺度检测**：
   - 80×80：小目标
   - 40×40：中目标
   - 20×20：大目标
   - 每个尺度3个锚框，共9个

### 2.3 OCR技术

#### 原理
OCR（Optical Character Recognition，光学字符识别）分为文本检测和文本识别两个阶段。

**文本检测**：
- 定位图像中的文本区域
- 常用方法：CTPN、EAST、DB、PSENet
- 输出：文本框坐标

**文本识别**：
- 识别文本框中的文字
- 常用方法：CRNN、SAR、MASTER、TrOCR
- 输出：文本内容

**端到端OCR**：
- 同时完成检测和识别
- 常用方法：TextSpotter、PGNet
- 效率更高

#### 考点
1. **CRNN结构**：
   - CNN：提取图像特征
   - RNN（Bi-LSTM）：序列建模
   - CTC：对齐输入输出
   - 优点：不需要字符级标注

2. **CTC Loss**：
   - 解决输入输出长度不对齐问题
   - 引入blank标签
   - 动态规划计算损失

3. **OCR数据增强**：
   - 随机旋转、缩放
   - 加噪、模糊
   - 颜色抖动
   - 背景替换

4. **后处理**：
   - 规则校验：Luhn算法（银行卡）
   - 格式校验
   - 字典纠错
   - 语言模型纠错

### 3.1 BERT

#### 技术挑战与解决方案（基于agent智能对话机器人）

**挑战1：多业务场景下的意图识别与路由**
- **问题定位**：不同业务部门（销售、售后、技术支持）的咨询内容差异大，单一Agent难以处理所有场景，导致回复准确率低、用户体验差。
- **方案设计**：构建多Agent系统，实现基于业务场景的智能路由机制。
- **实施过程**：
  1. 分析各业务部门的常见问题类型，定义15+种业务意图
  2. 基于BERT微调构建意图识别模型，准确率达到92%
  3. 设计多Agent架构，每个Agent负责特定业务领域
  4. 实现智能路由模块，根据意图识别结果将对话分配给对应Agent
  5. 构建Agent间通信机制，支持跨领域问题的协同处理
- **结果验证**：多Agent系统的意图识别准确率达到95%，业务问题解决率从70%提升至90%，用户满意度提升40%。

**挑战2：长上下文理解与多轮对话管理**
- **问题定位**：客服对话往往涉及多轮交互，用户问题可能分散在多轮对话中，LLM模型的上下文窗口有限，难以理解完整对话意图。
- **方案设计**：构建对话状态管理系统，结合摘要技术扩展上下文理解能力。
- **实施过程**：
  1. 设计对话状态追踪（DST）模块，记录用户意图、实体信息和对话历史
  2. 实现对话摘要算法，将长对话压缩为关键信息，保持语义完整性
  3. 构建上下文窗口管理机制，动态调整输入内容，优先保留关键信息
  4. 优化Prompt设计，引导LLM关注对话重点
  5. 实现多轮对话记忆机制，确保对话连贯性
- **结果验证**：多轮对话理解准确率从65%提升至85%，平均对话轮次从8轮减少至5轮，问题解决时间缩短40%。

#### 原理
BERT（Bidirectional Encoder Representations from Transformers）是基于Transformer的预训练语言模型。

**核心创新**：
1. **双向预训练**：同时考虑上下文
2. **Masked Language Model（MLM）**：随机mask 15%的token
3. **Next Sentence Prediction（NSP）**：预测两个句子是否相邻

   - 10%保持不变

2. **NSP**：
   - 输入：[CLS] 句子A<[BOS_never_used_51bce0c785ca2f68081bfa7d91973934]> 句子B<[BOS_never_used_51bce0c785ca2f68081bfa7d91973934]>
   - 预测句子B是否是句子A的下一句

**微调**：
- 分类任务：在<[BOS_never_used_51bce0c785ca2f68081bfa7d91973934]> token后加分类层
- 序列标注：在每个token后加分类层
- 问答任务：特殊的输入格式

#### 考点
1. **BERT为什么是双向的**：
   - 不像GPT是自回归的
   - MLM任务允许同时看到左右上下文
   - 适合理解任务

2. **Transformer Encoder vs Decoder**：
   - Encoder：双向，适合理解
   - Decoder：单向，适合生成
   - BERT只用Encoder

3. **MLM为什么不mask所有token**：
   - 预训练-微调差异
   - 微调时没有<[BOS_never_used_51bce0c785ca2f68081bfa7d91973934]> token
   - 10%随机、10%保持，增加鲁棒性

4. **BERT的变体**：
   - RoBERTa：去掉NSP，更大batch
   - ALBERT：参数共享，因式分解
   - DistilBERT：知识蒸馏
   - BERT-base/large：不同规模

---

### 3.2 RAG（检索增强生成）

#### 原理
RAG（Retrieval-Augmented Generation）将检索和生成结合，提升LLM的准确性和时效性。

**核心流程**：
1. **文档入库**：
   - 文档切分
   - 向量化（Embedding）
   - 存储到向量数据库

2. **检索**：
   - 用户查询向量化
   - 向量相似度搜索
   - 返回Top-K相关文档

3. **生成**：
   - 构建Prompt：查询 + 检索到的文档
   - LLM基于Prompt生成回答
   - 引用来源

**关键组件**：
- **文档切分**：按段落、按Token数、递归切分
- **Embedding模型**：OpenAI Embedding、Sentence-Transformers
- **向量数据库**：Milvus、Chroma、FAISS、Weaviate
- **LLM**：GPT、Claude、Qwen、Llama

#### 考点
1. **RAG的优势**：
   - 减少幻觉
   - 知识可以实时更新
   - 可追溯来源
   - 保护隐私（不需要微调）

2. **如何提升RAG效果**：
   - 文档切分策略优化
   - 混合检索（关键词+语义）
   - 重排序（Rerank）
   - 查询扩展（Query Expansion）
   - 提示工程优化

3. **RAG vs Fine-tuning**：
   | 特性 | RAG | Fine-tuning |
   |------|-----|-------------|
   | 知识更新 | 容易，只需更新文档 | 困难，需要重新训练 |
   | 幻觉 | 较少 | 可能有 |
   | 可追溯 | 是 | 否 |
   | 计算成本 | 推理成本高 | 训练成本高 |
   | 适用场景 | 动态知识 | 静态知识 |

4. **常见问题及解决方案**：
   - 检索不相关：优化Embedding、调整切分策略
   - 上下文过长：摘要、压缩
   - 回答不准确：更好的Prompt、增加更多文档
   - 引用错误：改进来源标注

### 3.3 Agent与多Agent

#### 原理
Agent是能够感知环境、做出决策、执行行动的智能体。

**Agent核心组件**：
1. **感知（Perception）**：理解用户输入和环境
2. **推理（Reasoning）**：思考、规划、决策
3. **行动（Action）**：调用工具、执行操作
4. **记忆（Memory）**：短期记忆、长期记忆

**多Agent系统**：
- 多个Agent协作完成任务
- 每个Agent有专门的角色和能力
- Agent之间可以通信和协调

**常用框架**：
- LangChain：单Agent、简单多Agent
- LangGraph：更复杂的多Agent、状态管理
- AutoGen：微软的多Agent框架
- CrewAI：角色-based多Agent

#### 考点
1. **Agent的思考方式**：
   - ReAct：Reasoning + Acting，边想边做
   - CoT（Chain of Thought）：思维链，逐步推理
   - ToT（Tree of Thought）：思维树，探索多个路径

2. **多Agent如何协作**：
   - 角色分配：每个Agent有专门角色（专家、审查者、执行者）
   - 通信机制：消息传递、共享状态
   - 协调策略：集中式、分布式
   - 冲突解决：投票、仲裁

3. **LangGraph vs LangChain**：
   - LangChain：线性流程，简单Agent
   - LangGraph：图结构，循环、分支、状态管理
   - LangGraph更适合复杂多Agent场景

4. **Agent的记忆类型**：
   - 短期记忆：对话历史
   - 长期记忆：向量存储的知识
   - 工作记忆：当前任务的中间状态

## 四、知识图谱

### 4.1 知识图谱基础

#### 原理
知识图谱（Knowledge Graph）以图的形式存储知识，由实体、关系、属性组成。

**基本概念**：
- **实体（Entity）**：现实世界中的事物（如：人、公司、产品）
- **关系（Relation）**：实体之间的联系（如：创始人、位于）
- **属性（Attribute）**：实体的特性（如：年龄、地址）
- **三元组（Triple）**：(头实体, 关系, 尾实体)

**存储方式**：
- **RDF（Resource Description Framework）**：W3C标准
- **属性图（Property Graph）**：Neo4j使用
- **图数据库**：Neo4j、JanusGraph、NebulaGraph

#### 考点
1. **知识图谱的应用场景**：
   - 搜索引擎：Google Knowledge Graph
   - 问答系统：基于知识图谱的QA
   - 推荐系统：实体关联推荐
   - 金融风控：关联分析、反欺诈
   - 医疗：疾病-药物关联

2. **知识图谱构建流程**：
   1. 知识抽取：从文本中抽取实体、关系、属性
   2. 知识融合：实体对齐、关系对齐
   3. 知识存储：存入图数据库
   4. 知识推理：基于规则或机器学习推理
   5. 知识应用：问答、推荐等

---

### 4.2 实体与关系抽取

#### 原理
从非结构化文本中抽取结构化的实体和关系。

**实体抽取（NER）**：
- 识别文本中的实体
- 分类实体类型（人名、地名、机构名等）
- 常用方法：BERT、BiLSTM+CRF

**关系抽取（RE）**：
- 识别实体之间的关系
- 常用方法：
  - 流水线方法：先NER再RE
  - 联合抽取：同时抽取实体和关系
  - 基于模板
  - 基于远程监督

#### 考点
1. **NER常用方法**：
   - 传统方法：CRF、HMM
   - 深度学习：BiLSTM+CRF、BERT
   - BERT为什么好？上下文理解能力强

2. **关系抽取的挑战**：
   - 同一实体对可能有多种关系
   - 关系表达多样
   - 远程监督的噪声问题
   - 长尾关系（数据少）

3. **实体对齐**：
   - 不同数据源的同一实体对齐
   - 方法：属性相似度、结构相似度、预训练模型
   - 难点：同名异义、异名同义

---

### 4.3 Neo4j

#### 原理
Neo4j是最流行的图数据库，使用属性图模型。

**数据模型**：
- 节点（Node）：代表实体
- 关系（Relationship）：连接节点，有方向和类型
- 属性（Property）：节点和关系都可以有属性
- 标签（Label）：节点的分组

**Cypher查询语言**：
```cypher
// 创建节点
CREATE (p:Person {name: '张三', age: 30})

// 创建关系
MATCH (p:Person), (c:Company)
WHERE p.name = '张三' AND c.name = '腾讯'
CREATE (p)-[:WORKS_AT]->(c)

// 查询
MATCH (p:Person)-[:WORKS_AT]->(c:Company)
WHERE c.name = '腾讯'
RETURN p.name
```

#### 考点
1. **图数据库 vs 关系数据库**：
   | 特性 | 图数据库 | 关系数据库 |
   |------|---------|-----------|
   | 数据模型 | 图 | 表 |
   | 关联查询 | 高效（遍历） | 低效（JOIN） |
   |  schema | 灵活 | 固定 |
   | 适用场景 | 关联分析 | 事务处理 |

2. **Cypher基本语法**：
   - MATCH：匹配模式
   - WHERE：过滤条件
   - CREATE：创建节点/关系
   - RETURN：返回结果
   - 箭头表示关系方向：->

## 五、多模态技术

### 5.1 CLIP

#### 原理
CLIP（Contrastive Language-Image Pre-training）是跨模态预训练模型，学习图像和文本的联合表征。

**核心创新**：
1. **双编码器架构**：图像编码器 + 文本编码器
2. **跨模态对比学习**：同一图文对为正样本，批次内其他为负样本
3. **对称损失**：Image→Text + Text→Image
4. **零样本能力**：通过文本描述分类

**网络结构**：
```
图像 → 图像编码器 → 图像特征归一化
文本 → 文本编码器 → 文本特征归一化
→ 相似度矩阵 → 对称对比损失 → 参数更新
```

**预训练数据**：
- 4亿图文对
- 多样化的数据源

**应用**：
- 零样本分类
- 图文检索
- 图像描述
- 视觉问答

#### 考点
1. **温度系数τ的作用**：
   - 缩放相似度分布
   - τ与交叉熵负相关
   - τ与准确率倒U型关系
   - 通常τ=0.07

2. **CLIP的零样本能力**：
   - 构造提示模板："a photo of a [CLASS]"
   - 计算图像与各类别提示的相似度
   - 选择相似度最高的类别
   - 无需针对特定任务训练

3. **CLIP的局限性**：
   - 细粒度分类能力不足
   - 推理速度慢
   - 对抽象概念理解有限
   - 中文能力有限（原版）

4. **CLIP微调方法**：
   - 提示工程
   - 全模型微调
   - Adapter/LoRA
   - 线性探针

---

### 5.2 SimCLR

#### 原理
SimCLR是自监督视觉表示学习框架，通过对比学习学习图像特征。

**核心思想**：
1. **数据增强**：对同一图像应用两次不同的增强，得到两个视图
2. **编码器**：提取两个视图的特征
3. **投影头**：将特征映射到对比空间
4. **对比损失**：最大化同一图像两个视图的相似度，最小化不同图像视图的相似度

**数据增强**：
- 随机裁剪
- 随机翻转
- 颜色抖动
- 灰度化
- 高斯模糊

**损失函数**：
- NT-Xent（Normalized Temperature-Scaled Cross-Entropy Loss）
- 批次内对比，2N个样本（N张图×2视图）

#### 考点
1. **SimCLR为什么有效**：
   - 强数据增强：学习不变性
   - 大batch size：更多负样本
   - 投影头：更好的对比空间
   - 温度系数：调整分布

2. **对比学习的关键要素**：
   - 数据增强：生成正样本对
   - 负样本：批次内其他样本
   - 温度系数：调整相似度分布
   - 归一化：L2归一化特征

3. **SimCLR v2的改进**：
   - 更深的投影头
   - 微调投影头之前的层
   - 更多的蒸馏

---

### 5.3 BLIP

#### 原理
BLIP（Bootstrapping Language-Image Pre-training）是多模态预训练模型，在CLIP基础上改进。

**核心创新**：
1. **多任务预训练**：
   - 图文对比学习（类似CLIP）
   - 图文匹配
   - 图像描述生成
2. **CapFilt**：Bootstrap方式，用模型生成和过滤数据
3. **灵活的架构**：既可以做检索，也可以做生成

**网络结构**：
- 图像编码器：ViT
- 文本编码器：BERT
- 可以是单塔或双塔

#### 考点
1. **BLIP vs CLIP**：
   - CLIP：双塔，对比学习，检索强
   - BLIP：多任务，生成能力强
   - BLIP在图像描述、VQA上更好

2. **CapFilt**：
   - Captioning：用模型生成图像描述
   - Filtering：过滤掉质量低的描述
   - 用生成的caption继续训练
   - Bootstrapping方式迭代提升

## 六、工程化部署

### 6.1 FastAPI

#### 原理
FastAPI是现代、快速的Python Web框架，用于构建API。

**核心特性**：
- 基于Python 3.7+类型提示
- 自动生成API文档（Swagger UI、ReDoc）
- 高性能（基于Starlette和Pydantic）
- 异步支持
- 数据验证

**基本使用**：
```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class Item(BaseModel):
    name: str
    price: float

@app.get("/items/{item_id}")
async def read_item(item_id: int):
    return {"item_id": item_id}

@app.post("/items/")
async def create_item(item: Item):
    return item
```

#### 考点
1. **FastAPI的优势**：
   - 自动文档：Swagger UI和ReDoc
   - 类型提示：自动验证
   - 异步：高性能
   - 基于标准：OpenAPI、JSON Schema

2. **依赖注入**：
   ```python
   from fastapi import Depends
   
   def get_db():
       db = SessionLocal()
       try:
           yield db
       finally:
           db.close()
   
   @app.get("/items/")
   async def read_items(db: Session = Depends(get_db)):
       return db.query(Item).all()
   ```

3. **异常处理**：
   ```python
   from fastapi import HTTPException
   
   @app.get("/items/{item_id}")
   async def read_item(item_id: int):
       if item_id not in items:
           raise HTTPException(status_code=404, detail="Item not found")
       return items[item_id]
   ```

### 6.2 Celery + Redis

#### 原理
Celery是分布式任务队列，用于异步处理任务。

**核心组件**：
- **Producer**：提交任务
- **Broker**：消息队列（Redis、RabbitMQ）
- **Worker**：执行任务
- **Backend**：存储结果（Redis、数据库）

**基本使用**：
```python
from celery import Celery

app = Celery('tasks', broker='redis://localhost:6379/0', backend='redis://localhost:6379/1')

@app.task
def add(x, y):
    return x + y

# 提交任务
result = add.delay(4, 4)

# 获取结果
result.get()
```

#### 考点
1. **为什么需要Celery**：
   - 异步处理耗时任务
   - 解耦：服务层和执行层分离
   - 可扩展：动态增加Worker
   - 高可用：某个Worker失败不影响其他

2. **任务调度**：
   - 延迟任务：apply_async(countdown=10)
   - 定时任务：Celery Beat + crontab
   - 任务链：chain()
   - 任务组：group()

3. **结果存储**：
   - Backend配置
   - 结果过期时间
   - 忽略结果：ignore_result=True

### 6.3 Docker

#### 原理
Docker是容器化平台，将应用和依赖打包到容器中。

**核心概念**：
- **镜像（Image）**：只读模板
- **容器（Container）**：镜像的运行实例
- **Dockerfile**：构建镜像的脚本
- **Docker Compose**：多容器编排

**Dockerfile示例**：
```dockerfile
FROM python:3.9-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```

**常用命令**：
```bash
# 构建镜像
docker build -t myapp .

# 运行容器
docker run -p 8000:8000 myapp

# 查看容器
docker ps

# 停止容器
docker stop <container_id>
```

#### 考点
1. **Docker的优势**：
   - 一致性：开发、测试、生产环境一致
   - 隔离性：应用之间互不影响
   - 轻量：相比虚拟机更轻
   - 快速启动：秒级启动

2. **镜像优化**：
   - 使用多阶段构建
   - 选择基础镜像（slim、alpine）
   - 合并RUN命令
   - 使用.dockerignore
   - 层缓存

3. **Docker Compose**：
   ```yaml
   version: '3'
   services:
     web:
       build: .
       ports:
         - "8000:8000"
     redis:
       image: "redis:alpine"
   ```
