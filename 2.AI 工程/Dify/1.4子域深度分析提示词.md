# Dify 子域深度分析提示词

> 使用方式：修改下方「子域变量」区块，其余内容不变，整体发送即可。
> 同一份提示词可分析 Dify 的全部 11 个子域。

---

## 子域变量

```
子域名称：[如 知识库域（Knowledge / RAG）]
DDD 类型：[核心域 / 支撑域 / 边缘域]
主模型文件：[如 api/models/dataset.py]
核心领域模块：[如 api/core/rag/]
核心服务文件：[如 api/services/dataset_service.py]
专项聚焦（可选）：[填写该域特有的关键设计问题，如 索引状态机 / draft-published 双轨 / 凭据加密]
```

## 分析任务

深度分析 Dify 1.13.0「{子域名称}」，从数据模型、代码架构、业务流程三个维度完整还原设计意图。

**源文件读取顺序**：先逐类读取主模型文件（不跳过任何 Model 类），再读核心领域模块理解抽象结构，最后读服务层理解字段的实际使用语义。

---

## 分析维度

### 一、子域定位

- 在全局子域图（核心域 / 支撑域 / 边缘域）中的位置，一句话说清职责
- **数据主权**：该域独占哪些数据的写入权，哪些数据其他域不能直接写
- **边界约束**：importlinter 有哪些隔离规则；与其他子域的协作方式（ID 引用 / 快照引用 / 外键）

### 二、数据模型

- 表清单：所有表名 + 一句话职责
- 核心表（3-5 张）的字段分析：聚焦字段背后的**设计意图**，而非逐字段枚举
- 2 个关键设计决策，格式：`场景描述 → 选择方案 → 设计理由 → 代价与权衡`
- 跨域引用：该域通过哪些字段引用其他域，哪些是跨域逻辑关联（无外键约束）

### 三、代码架构（core/ 模块）

- 抽象层级：ABC / 接口 → 能力类型分支 → 具体实现的继承结构
- 关键扩展点：新增一个具体类型（如节点 / 工具 / 供应商）需要实现哪些抽象、遵守哪些注册约束
- 若填写了专项聚焦，在此深入展开

### 四、典型业务场景

选取 1-2 个最能体现该域设计意图的典型场景（如「上传文档并索引」「发布工作流」），说明：
- 经过哪些层（controller → service → core → model）
- 数据在哪个阶段写入、哪些字段发生变化
- 哪些步骤是异步（Celery 任务介入点）

### 五、核心实体状态机（如适用）

- 聚合根的生命周期：状态名与源码枚举值严格对应，不使用自创别名
- 哪些转换是不可逆的；Celery 任务在哪个状态转换处介入
- 关联字段变化（对应数据库字段赋值）

### 六、跨域协作边界

- **上游**：通过哪些字段（tenant_id / model_id 等）接受哪些域的注入
- **下游**：向哪些域暴露什么接口或 ID 引用
- **不拥有**：明确列出该域边界内不负责的能力（防止边界模糊）

### 七、运行时调度链路（如适用）

> 适用条件：该域向上层域暴露「可调用实例」而非仅 ID 引用时（如模型供应商域的 `ModelInstance`、工具域的 `ToolInvoker`）。纯配置存储型边缘域可跳过。

- **统一入口**：上层域通过什么对象或方法获取该域的运行时能力（类名 + 调用签名）
- **调度层次**：从入口到最终执行，经过哪几层对象，每层解决什么问题（凭据绑定 / 负载均衡 / 错误转换 / 插件代理）；凭据在哪一层被解密、在哪一层被传入
- **故障处理**：若有重试 / Cooldown / 降级机制，说明触发条件、时长与终止条件

---

## 质量标准

- 每个结论必须关联到具体字段名、类名或代码路径，不接受无出处的泛泛概括
- 设计决策部分写「为什么这样设计，而非其他方案」，而非仅描述「是什么」

---

## 格式规范

Mermaid 图按《Mermaid 作图风格完整指南》规范绘制，每张图前写 1-2 句关注焦点，图后写 2-3 条非显而易见的设计要点。

必须包含的图类型：

| 图 | 语法 | 分析目标 |
|----|------|---------|
| 数据模型关系图 | `flowchart TB` | 表关系、外键强约束与逻辑关联、NOTE 要点节点、linkStyle 着色 |
| 类层级关系图 | `flowchart TB` | core/ 抽象层级、继承树与 Mixin 组合、NOTE 要点节点 |
| 状态机图 | `stateDiagram-v2` | 核心聚合根生命周期（无状态驱动实体可省略） |
| 端到端流程图或时序图 | `flowchart LR` 或 `sequenceDiagram` | 典型场景执行链路；若有 Celery 异步链路，优先用时序图 |
| 子域模块架构图（条件） | `flowchart TB` | 该域内 controller → service → core → model 的模块调用关系；仅核心域或模块层次复杂的支撑域出图，边缘域可跳过 |
| 运行时调度图（条件） | `flowchart LR` | 从统一入口到实际执行的调度层次；有 LB / 重试时标注循环结构（适用于七时必须出图） |

> 图的选用逻辑参见《Mermaid 作图风格完整指南》一、六种图表的区别与选择。
