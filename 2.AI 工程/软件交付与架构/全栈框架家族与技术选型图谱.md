> **文档职责**：梳理主流全栈技术栈的框架家族与选型地图。
> **适用场景**：用于学习技术版图、做技术选型预研、理解不同技术的层次关系。
> **阅读目标**：快速建立“前端、后端、数据、AI、工程体系”各层主流技术的家族认知。

# 全栈框架家族与技术选型图谱

## 1. 使用说明

这份文档不是某个项目的落地方案，而是一个**主流全栈技术栈家族图谱**。  
目标是先帮助你建立“技术分层”和“技术家族”认知，再去做具体项目选型。

原则：

- 只收录主流、成熟、常见方案
- 偏大而全，但不收太末流的技术
- 每个节点只给一句简短说明

## 4. 分层速记

这一节是对第 2 节“全栈技术家族树”的补充说明。  
第 2 节回答“有哪些主流技术家族”，第 4 节回答“这些技术家族分别处在哪一层、主要解决什么问题、彼此如何配合”。

### 4.1 前端

- 语言层：`JavaScript / TypeScript`，负责写前端业务逻辑；其中 `TypeScript` 是对 `JavaScript` 的工程化增强
- UI 层：`React / Vue / Angular / Svelte / Solid`，负责组件化页面开发
- 应用框架层：`Next.js / Nuxt / Remix / SvelteKit / Astro`，负责路由、数据获取、SSR/SSG、项目组织
- 构建层：`Vite / Webpack / 框架内建构建系统`，负责本地开发、热更新、打包构建
- 运行层：`Node.js`，负责运行前端开发工具、构建工具，以及部分 SSR 服务
- 状态管理层：负责管理页面状态、远程数据缓存和共享数据
- 样式与视觉层：负责样式表达、动效和视觉呈现

前端里最容易混淆的一组关系可以记成：

```text
一个现代前端项目
├── 基础能力：HTML / CSS / JavaScript
├── 工程增强：TypeScript
├── UI 开发：React 或 Vue
├── 应用框架：Next.js 或 Nuxt
├── 构建工具：Vite / Webpack / 框架内建体系
└── 运行支撑：浏览器 + Node.js
```

也就是说，`React`、`Next.js`、`Vite`、`Node.js` 往往会同时出现，但它们不是同一层的替代关系。

### 4.2 后端

- 语言层：`Python / Node.js(TypeScript) / Go / Java`，决定主要开发语言和生态
- Web 框架层：`FastAPI / Django / Flask / NestJS / Express / Gin / Spring Boot`，负责路由、接口、请求处理
- 运行层：`Uvicorn / Gunicorn / Node runtime / JVM`，负责让后端程序真正运行起来
- 数据建模层：`Pydantic / Prisma / SQLAlchemy / TypeORM`，负责请求响应结构、数据对象和 ORM 映射
- 业务编排层：`Service / Pipeline / Workflow Engine / Job Worker`，负责组织业务流程和异步任务
- API 形态层：`REST / GraphQL / WebSocket / gRPC`，负责系统对外暴露能力的方式

可以把后端理解为：先选语言和框架，再确定运行方式、数据建模方式，以及 API 设计风格。

### 4.3 数据与存储

- 关系型数据库：`PostgreSQL / MySQL / SQLite`，负责存储结构化业务数据
- 缓存与消息：`Redis / RabbitMQ / Kafka`，负责提速、解耦、异步通信
- 对象存储：`OSS / S3 / MinIO`，负责存储图片、视频、文档等文件
- 向量数据库：`Milvus / pgvector / Weaviate`，负责语义检索、Embedding 检索等 AI 场景

这一层的核心不是“页面怎么写”，而是“数据放在哪里、怎么更快拿到、怎么支撑 AI 检索”。

### 4.4 AI 与多媒体

- 文本与推理：负责通用对话、文本生成、推理任务
- 图像与视频：负责图片生成、视频生成和多模态内容生产
- 语音能力：负责语音识别、语音合成、声音克隆
- 多模型接入：负责统一接多个模型供应商，做路由、鉴权、限流和抽象
- 媒体处理：负责视频转码、时间线处理、字幕生成与烧录

这一层更像“能力层”，常常被上层产品功能调用，而不是直接承担传统 Web 路由职责。

### 4.5 工程体系

- 依赖管理：`uv / Poetry / npm / pnpm / Yarn`，负责安装和管理项目依赖
- 测试体系：`pytest / Vitest / Jest / Playwright`，负责单测、集成测试和端到端测试
- 容器与部署：`Docker / Kubernetes / Vercel / Nginx`，负责打包、发布、托管和流量转发
- CI/CD：`GitHub Actions / GitLab CI / Jenkins`，负责自动化构建、测试、发布
- 可观测性：`Prometheus / Grafana / ELK / Sentry`，负责监控、日志、告警和异常追踪

这一层不直接产出业务功能，但决定项目是否容易开发、测试、发布和维护。

### 4.6 阅读这张图时的原则

- 先看技术属于哪一层，再看它和谁比较
- 同一层的技术更适合横向比较，不同层的技术更适合理解配合关系
- 做选型时，先确定主干层，再补配套层
- 不要把“语言、框架、运行时、构建工具、部署平台”混成一类

## 5. 适合怎么用这份图谱

- 做项目选型时，先确定每一层要解决什么问题
- 不要把“语言、框架、运行时、构建工具”混成一层比较
- 先选主干方案，再补配套方案
- 新项目优先看主流路线，不要一开始追求小众技术

## 6. 结论

如果你是从 Python 后端视角进入前端和全栈世界，这份图谱最重要的价值不是“记住所有技术名词”，而是：

- 先分清技术属于哪一层
- 再理解同层技术怎么比较
- 最后再做项目选型

这样看技术栈，就不会再把 `React / Next.js / Vite / Node.js` 混成一类了。
