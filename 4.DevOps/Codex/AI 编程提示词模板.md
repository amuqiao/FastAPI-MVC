> **文档职责**：收录可直接复用的 AI 编程提示词模板，覆盖后端全栈开发日常场景，兼顾代码工作和知识整理两类需求。
> **适用场景**：开发过程中需要 AI 协助时，直接查找对应场景，复制模板填写上下文后使用。
> **目标读者**：偏后端的全栈开发者，熟悉服务端开发，前端和部分新技术仍在学习中。
> **维护规范**：新增模板归入已有章节，不随意新开章节；模板效果差时直接修改模板内容，在"说明"里补充注意事项；[占位符] 风格保持统一；不收录一次性 prompt，只收录可复用的通用模板。

---

## 如何使用本文档

每条模板的结构：

- **适用时机**：什么情况下用这个模板（一句话）
- **模板**：可直接复制，`[XX]` 为需要填写的部分，`@文件` 为 Claude Code 文件引用
- **说明**：使用技巧或注意事项

填写时替换 `[XX]` 内容，`@文件` 在 Claude Code 交互模式中输入 `@` 后通过路径补全引用。

## 一、代码理解与分析

### 1.1 文件职责速览

**适用时机**：接手陌生代码，快速了解一个文件的作用。

```
请阅读 @[文件路径]，用 3-5 句话说明：它的职责是什么、输入输出是什么、核心逻辑是什么。不需要逐行解释。
```

**说明**：加"不需要逐行解释"可以避免 AI 输出过长。如果想更聚焦，可以追加"重点说明 [具体关注点，如错误处理 / 并发逻辑]"。

---

### 1.2 函数或方法分析

**适用时机**：看不懂某个函数的用途或调用方式。

```
解释 @[文件路径] 中 [函数名] 的作用，说明：参数的含义、返回值的结构、调用方需要注意什么、有哪些边界情况。
```

**说明**：如果函数有副作用（如修改全局状态、写数据库），AI 通常会主动提及，但也可以显式要求："特别说明是否有副作用"。

---

### 1.3 模块依赖梳理

**适用时机**：排查耦合问题，或理解某个模块在系统中的位置。

```
梳理 @[文件路径] 的依赖关系：它依赖哪些模块，哪些模块依赖它，是否存在循环依赖风险。用列表或简短说明呈现，不需要画图。
```

---

### 1.4 代码逻辑追踪

**适用时机**：需要理解一个请求或数据从入口到出口的完整流程。

```
追踪 [请求/数据/事件] 从 [入口，如 API 路由 / 消息队列消费者] 到 [出口，如数据库写入 / 响应返回] 的完整流程，列出经过的关键函数和处理步骤。相关文件：@[文件1] @[文件2]。
```

## 二、调试与问题排查

### 2.1 错误分析

**适用时机**：遇到报错，需要定位根本原因。

```
以下报错来自 [服务名 / 环境]：

[粘贴完整错误信息]

结合 @[相关文件]，分析根本原因，给出修复步骤。如果有多种可能原因，按可能性从高到低列出。
```

**说明**：粘贴完整错误信息（含 stack trace）比截图描述有效得多。如果报错与特定数据有关，也一并提供数据结构或样例。

---

### 2.2 行为异常排查

**适用时机**：程序没报错，但输出结果与预期不符。

```
[描述预期行为] 但实际上 [描述实际行为]。

复现条件：[输入数据 / 触发步骤]

相关代码：@[文件路径]

帮我找出可能的原因，从最可能的开始，不要直接修改代码，先列出排查思路。
```

**说明**："不要直接修改代码，先列出排查思路"可以防止 AI 直接给出一个可能错误的修复，先确认方向再动手。

---

### 2.3 性能问题分析

**适用时机**：接口慢、内存高、CPU 异常等性能类问题。

```
@[文件路径] 中的 [函数名 / 接口] 在 [场景，如高并发 / 大数据量 / 频繁调用] 下出现性能问题。

帮我找出主要瓶颈，给出优化建议，优先考虑 [时间复杂度 / 内存占用 / 数据库查询次数]，说明每项改动的预期收益。
```

## 三、功能开发与实现

### 3.1 功能实现

**适用时机**：需要 AI 直接写出某个功能的代码。

```
在 @[文件路径] 中实现 [功能描述]。

约束条件：
- [如：不引入新的外部依赖]
- [如：保持现有代码风格]
- [如：需要处理 XX 错误情况]

参考现有实现：@[参考文件]（可选）
```

**说明**：约束条件越明确，输出越贴合需求。不写约束条件，AI 可能引入你不熟悉或不兼容的库。

---

### 3.2 API 接口设计

**适用时机**：新功能需要先确定接口规范再开发。

```
设计一个 [功能描述] 的 REST API，包含：

- 路由和 HTTP 方法
- 请求体结构（JSON 示例）
- 响应结构（成功和失败）
- 错误码定义

风格参考 @[现有接口文件]（可选）。用 [JSON / Markdown 表格] 格式呈现。
```

---

### 3.3 数据库表设计

**适用时机**：新业务需要设计数据库表结构。

```
为 [业务场景描述] 设计数据库表结构，需要支持的核心查询：[列出 2-3 个主要查询场景]。

数据库：[MySQL / PostgreSQL]，数据规模预估：[量级]。

给出：建表 SQL、索引建议、简要说明设计理由。
```

**说明**：提供查询场景比只描述业务更有效，AI 可以根据查询模式反推索引设计。

---

### 3.4 单元测试生成

**适用时机**：为已有函数补充测试用例。

```
为 @[文件路径] 中的 [函数名] 生成单元测试。

测试框架：[如 Jest / pytest / JUnit]
要求覆盖：正常路径、边界值、常见异常情况。
不需要 mock 的模块：[如果有]
```

## 四、重构与优化

### 4.1 代码重构

**适用时机**：代码能跑，但可读性差、职责混乱或重复过多。

```
重构 @[文件路径] 中的 [函数名 / 模块名]，目标：[提高可读性 / 拆分职责 / 消除重复逻辑]。

约束：保持现有对外接口不变，不修改 [列出不希望改动的部分]。

先给出重构思路，确认后再写代码。
```

**说明**："先给出重构思路"很重要，避免 AI 直接输出一大段你可能不想要的代码结构。

---

### 4.2 代码审查

**适用时机**：自己写完代码后，想让 AI 提意见。

```
审查 @[文件路径] 的代码，重点关注：[安全性 / 性能 / 可维护性 / 错误处理]。

给出具体改进建议，指出行号或函数名，说明每条建议的理由。忽略风格类问题（格式、命名偏好）。
```

## 五、学习与笔记整理

### 5.1 原理讲解

**适用时机**：遇到不熟悉的技术概念，想快速理解核心原理。

```
解释 [技术 / 概念] 的工作原理。

我的背景：[如"熟悉关系型数据库，第一次接触时序数据库"]
讲解要求：用类比帮助理解，说明它解决了什么问题，核心设计是什么，不需要从零讲基础知识。
```

**说明**：提供背景能让 AI 跳过你已知的部分，直接切入陌生内容。

---

### 5.2 对比学习

**适用时机**：需要在两个相似技术间做选择，或理解它们的差异。

```
对比 [技术A] 和 [技术B] 的核心区别，从以下维度分析：[适用场景 / 性能特点 / 运维复杂度 / 生态成熟度]。

我更熟悉 [技术A]，可以用它作为参照来解释 [技术B] 的不同之处。
```

---

### 5.3 整理学习笔记

**适用时机**：和 AI 探讨完某个技术后，想把对话整理成可留存的笔记。

```
根据我们刚才的对话，整理一份关于 [主题] 的学习笔记，结构如下：

1. 核心概念（2-3 个关键词 + 一句话定义）
2. 工作原理（用简洁的步骤或流程描述）
3. 适用场景（什么时候用、什么时候不用）
4. 注意事项（常见坑或误解）
5. 延伸阅读（如果有值得了解的相关概念）

语言简洁，适合日后快速回顾。
```

---

### 5.4 陌生代码转化为文档

**适用时机**：读完某个陌生模块后，想留下一份可维护的说明文档。

```
阅读 @[文件路径]，为它生成一份模块说明文档，包含：

- 模块职责（一段话）
- 对外暴露的接口（函数名、参数、返回值）
- 主要内部逻辑（分点描述，不要逐行）
- 使用示例（如果能推断）
- 已知限制或注意事项

格式：Markdown，可以直接放入项目文档。
```

---

### 5.5 技术选型分析

**适用时机**：面对一个技术决策，需要结构化地分析选项。

```
我需要为 [具体需求] 选择技术方案，背景：[系统规模 / 团队规模 / 现有技术栈]。

候选方案：[方案A、方案B，或"你来推荐"]

给出 2-3 个方案，每个说明：适用场景、主要优势、主要限制、与我现有技术栈的兼容性。最后给出推荐和理由。
```

## 六、Claude Code 专属技巧

> 本节面向 Claude Code 新用户，结合工具特性给出提示词配合建议。

### 6.1 先规划后执行

**适用时机**：任务涉及多个文件改动，或不确定改动范围。

```
/plan

[描述任务]，相关文件：@[文件1] @[文件2]。

先给出改动计划和影响范围，不要直接修改文件。
```

**说明**：进入 `/plan` 模式后 Claude 只分析不执行。确认计划合理后，按 `Shift+Tab` 切换到 `default` 或 `acceptEdits` 模式再执行。

---

### 6.2 给任务提供充分上下文

**适用时机**：任务涉及项目特定结构或约定，怕 AI 猜错。

```
@[入口文件] @[相关模型文件] @[相关路由文件]

我们项目的约定：[如"所有错误通过统一的 AppError 类抛出""路由文件只做参数校验，业务逻辑在 service 层"]

现在需要：[任务描述]
```

**说明**：多引用几个相关文件，并说明项目约定，比只引用一个文件效果好得多。

---

### 6.3 边开发边记录知识

**适用时机**：开发中遇到陌生技术，想顺手记下来。

```
我们刚才解决了关于 [技术点] 的问题。在继续开发前，请先整理一份简短笔记，说明：这个问题的根本原因、解决思路、下次遇到类似问题的排查方向。格式：Markdown，100 字以内。
```

**说明**：利用 Claude Code 已有的对话上下文，直接从解决过程生成笔记，比事后回忆效率高。

---

### 6.4 大任务拆分

**适用时机**：需求较大，一次性给 AI 做容易出错或超出上下文。

```
我需要实现 [大功能描述]，请先把它拆分成独立的子任务，每个子任务说明：做什么、依赖哪些其他子任务、预计改动哪些文件。

拆分完成后不要开始执行，等我确认顺序和优先级。
```

**说明**：大任务拆分后，每次只做一个子任务，上下文更清晰，也方便中途调整方向。
