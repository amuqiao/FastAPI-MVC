# Codex CLI 安装与配置指南

> **文档职责**：只解决 Codex CLI 的安装、登录、模型选择、权限与基础配置。  
> **适用场景**：你已经具备 `brew` / `node` / `npm` 环境，现在要把 Codex CLI 真正装好并跑起来。  
> **不包含内容**：不讲代码开发流程、不讲提示词工程、不讲复杂团队协作。  
> **阅读顺序**：读完本文后，继续看 `3.Codex_CLI_代码开发快速上手.md`；如果你要把可复用工作流沉淀成 Skill，再看 `5.Codex_Skill_构建安装与使用指南.md`。

## 1. 先明确这篇文档的边界

这个目录里，安装与配置只保留这一篇主文档。

本文只回答 5 个问题：

1. 安装前需要什么环境
2. 怎么安装 Codex CLI
3. 怎么登录或配置 API Key
4. 怎么处理模型、沙箱、审批和基础配置
5. 怎么验证当前环境已经可用于开发

像“如何描述需求”“如何让 Codex 读懂仓库”“如何做代码开发闭环”，统一放到后面的文档，不再和安装文档混在一起。

## 2. 安装前准备

### 2.1 本机基础依赖

你至少需要：

- `node`
- `npm`
- 一个可用终端

如果你在 macOS 上还没装好这些，先看：

- `1.Brew_Node_npm_安装与使用指南.md`

### 2.2 网络与账号

你至少要满足下面二选一：

1. 可以通过 ChatGPT 账号登录 Codex
2. 可以通过 `OPENAI_API_KEY` 使用兼容 OpenAI 的接口

如果你的网络环境不能直接访问 OpenAI，也没关系。对 Codex CLI 来说，核心不是你用了哪个网关产品，而是下面两项是否正确：

- `OPENAI_API_KEY`
- `OPENAI_BASE_URL`

也就是说，只要你的网关兼容 OpenAI API，Codex CLI 本质上只是在读取这两个配置。

---

## 3. 安装 Codex CLI

在终端执行：

```bash
npm install -g @openai/codex
```

如果你安装 npm 包时需要镜像，可以按你本机的 npm 配置或镜像源处理。

安装后验证：

```bash
codex --version
```

如果这一步失败，先排查：

1. `node -v` 是否正常
2. `npm -v` 是否正常
3. 全局 npm bin 目录是否已加入 `PATH`

---

## 4. 登录与认证

### 4.1 直接登录

最直接的方式是启动：

```bash
codex
```

然后按产品界面提示完成登录。

### 4.2 使用 API Key

如果你走 API Key 或代理网关，一般这样配置：

```bash
export OPENAI_API_KEY="sk-xxx"
```

如果还需要指定兼容 OpenAI 的网关地址，再补：

```bash
export OPENAI_BASE_URL="https://your-gateway.example.com/v1"
```

然后重新执行：

```bash
codex
```

### 4.3 不要把文档写死在某个模型版本

旧文档里最容易过时的内容，就是把某个具体模型版本单独拆成一篇“配置教程”。

更稳的做法是：

- 安装文档只讲“怎么查看和切换模型”
- 不把目录结构绑定在某个短期版本号上
- 当前可用模型以 CLI 实际显示结果为准

## 5. 第一次启动时你会遇到什么

第一次进入 Codex，通常会遇到下面几类配置动作：

- 当前目录是否可信
- 是否启用或切换沙箱/权限模式
- 当前使用哪个模型

这些动作不要孤立理解，它们本质都在回答一个问题：

**Codex 这次会在什么边界内工作。**

你至少要看懂 3 个概念：

- `模型`：决定能力、速度和成本
- `沙箱`：决定它能读写和执行到什么程度
- `审批策略`：决定它什么时候停下来问你

## 6. 最小必要配置

### 6.1 临时切模型

命令行启动时可以直接指定：

```bash
codex --model gpt-5.4
```

如果你不确定该用哪个模型，更稳的策略不是把文档记成“永远用某一个版本”，而是：

1. 先启动 Codex
2. 用 `/model` 查看当前可选模型
3. 按任务复杂度切换

### 6.2 会话内切权限

进入 Codex 后，可以通过：

```text
/permissions
```

切换权限模式。

日常本地开发通常优先选择“能在工作区内低摩擦改代码，但不直接完全放权”的模式。

### 6.3 用配置文件固化默认行为

如果你不想每次手动重复设置，可以把默认模型、沙箱和审批策略写进配置文件。

建议把它理解为：

- 临时实验：在 CLI 里切
- 长期习惯：写进配置文件

## 7. 一个实用的配置思路

你不一定一开始就要把配置写得很复杂，下面这种思路已经够用了：

### 7.1 本地人工协作场景

- 默认模型：按你最常用的开发模型设置
- 默认权限：工作区内可写
- 默认审批：需要越界时再问

适合：

- 你边看边改
- 你会手动审查 diff
- 你不希望 Codex 在系统范围内乱跑

### 7.2 自动化或脚本场景

如果后面要接脚本或 CI，再单独做一套更明确的自动化配置。

这部分统一放到 `4.Codex_CLI_开发进阶指导.md`，不在安装文档里展开。

## 8. 国内或代理环境下的推荐理解方式

不要把“国内可用”理解成某一个具体工具名。

更本质的理解是：

1. npm 安装阶段解决包下载问题
2. Codex 运行阶段解决 OpenAI 接口访问问题
3. Codex 只关心最终能否拿到正确的 `API Key` 和 `Base URL`

也就是说：

- npm 镜像解决的是“包装得下来”
- 网关或代理解决的是“模型调得通”

这两个问题不要混在一起排查。

## 9. 安装完成后的验证清单

如果你想确认当前机器已经可以稳定使用 Codex CLI，至少做下面这些检查：

### 9.1 命令层

```bash
codex --version
```

### 9.2 登录层

```bash
codex login status
```

如果你不是走登录态，而是走 API Key，也至少确认环境变量已经生效。

### 9.3 运行层

进入任意一个测试仓库执行：

```bash
codex
```

然后给一个最简单任务，例如：

```text
先阅读当前目录，告诉我这是一个什么项目，先不要修改文件。
```

如果这一步能够正常返回，说明基础链路已经通了。

## 10. 常见问题

### 10.1 `codex: command not found`

通常是 npm 全局安装目录没有进 `PATH`，先确认：

- `which node`
- `which npm`
- `npm root -g`

### 10.2 能安装但不能登录或不能调用模型

优先排查：

1. 认证方式是否正确
2. `OPENAI_API_KEY` 是否正确
3. `OPENAI_BASE_URL` 是否正确
4. 终端是否继承了最新环境变量

### 10.3 文档里写死某个模型，但 CLI 里看见的是别的模型

这是正常问题。

模型版本更新很快，所以目录文档不再单独保留“某个具体版本的配置截图篇”。以后统一以 CLI 当前可选项和官方文档为准。
