# Codex 上下文添加指南

这份文档说明在 Codex 中，怎样给模型补充你希望它重点关注的上下文。可以把方式分成两类：

- 临时上下文：只对当前这次对话或当前任务生效。
- 持久上下文：会在后续会话里持续生效。

下面重点讲 4 类：文件上下文、目录上下文、代码片段上下文、其他常见上下文。

## 1. 文件上下文

### 1.1 在 IDE 里直接引用文件

如果你在 Codex IDE Extension 里工作，最直接的方式是用 `@文件名` 引用文件。

示例：

```text
参考 @src/pages/Home.tsx 的风格，帮我新增一个 FAQ 页面
再从 @src/data/faq.ts 里读取数据
```

这种方式适合：

- 明确指定“请看这个文件”
- 同时关联多个文件
- 减少“你先自己找文件”的来回沟通

### 1.2 在 CLI 里用 `/mention` 附加文件

如果你在 Codex CLI 里工作，可以用：

```text
/mention src/lib/api.ts
```

选择匹配结果后，Codex 会把这个文件加入当前对话上下文。后续继续追问时，它会更直接地基于这个文件回答。

这种方式适合：

- 命令行里快速指定目标文件
- 让当前会话聚焦某几个关键文件

### 1.3 利用 IDE 自动跟踪当前打开的文件

如果你同时使用 Codex App 和 IDE Extension，并且它们连接的是同一个项目，那么可以开启 IDE context 的 `Auto context`。开启后，Codex 会跟踪你当前正在查看的文件。

这时你可以直接问：

```text
这个文件是做什么的？
这里为什么会报错？
```

即使你没有手动再写一次文件名，Codex 也能更容易理解你在指哪一个文件。

## 2. 目录上下文

### 2.1 从正确的工作目录启动 Codex

在 CLI 中，最基础的目录上下文就是当前工作目录。直接运行：

```bash
codex
```

或者：

```bash
codex "解释这个代码库"
```

Codex 会读取当前工作目录，并据此理解项目结构。

如果你想指定另一个目录，可以显式切换工作根目录：

```bash
codex --cd ./services/payments "检查这里的支付流程"
```

### 2.2 用 `--cd`、`--add-dir` 控制目录范围

如果任务不只依赖当前目录，你可以显式告诉 Codex 应该以哪个目录为主，或者额外开放哪些目录。

示例：

```bash
codex --cd services/payments
codex --cd apps/frontend --add-dir ../backend --add-dir ../shared
```

适合这种场景：

- “把这个子目录当成主工作区”
- “当前项目之外，还要读取另一个目录”
- “前后端或 shared 目录要一起看”

### 2.3 用 `AGENTS.md` 给目录挂持久说明

如果你希望某个目录长期携带额外说明，最合适的是 `AGENTS.md`。

Codex 会从项目根目录一路读取到你当前所在目录；越靠近当前目录的 `AGENTS.md` 或 `AGENTS.override.md`，优先级越高。

例如：

```text
repo/
  AGENTS.md
  services/
    payments/
      AGENTS.override.md
```

你在 `services/payments/` 里运行 Codex 时，它会同时拿到：

- 全局 `~/.codex/AGENTS.md`（如果有）
- 仓库根目录的 `AGENTS.md`
- `services/payments/` 下更近的覆盖说明

这非常适合写：

- 目录职责
- 测试命令
- 代码规范
- 特殊风险提示
- “修改这里必须同步更新哪些文件”

### 2.4 跨多个目录时，优先拆成独立项目或 worktree

官方建议是：如果任务要跨多个仓库或多个目录，不要让 Codex 无限制地到处游走，而是优先打开单独项目，或者使用 worktree。

这比“让一个会话同时理解很多不相关目录”更稳。

## 3. 代码片段上下文

### 3.1 在 IDE 里直接选中代码

Codex IDE Extension 会利用当前打开的文件和你选中的代码。也就是说，你可以先框选一段函数、组件或报错附近代码，再发问。

适合：

- 解释某段逻辑
- 只修改局部实现
- 让 Codex 聚焦一个函数而不是整个文件

常见提问：

```text
这段选择的代码为什么会有竞态问题？
只重构我选中的这部分，不要改接口
给这段代码补测试
```

### 3.2 直接把片段贴进对话

如果你不在 IDE Extension 里，或者只是想讨论一小段代码，最简单的方法仍然是直接把代码粘贴到 prompt 里。

建议这样写：

```ts
function loadUser(id) {
  return fetch(`/api/users/${id}`).then((r) => r.json());
}
```

然后补一句明确要求：

```text
这段代码有什么错误处理问题？
只分析，不改文件
```

### 3.3 片段要带最少必要背景

单独贴代码片段时，最好再补 1 到 3 句背景，不然上下文很容易不完整。建议补充：

- 这段代码在哪个文件
- 它属于哪个模块
- 当前 bug / 目标是什么
- 是否允许改接口、改类型、改测试

例如：

```text
下面这段代码在 src/lib/auth.ts 里。
现在的问题是刷新 token 时偶发 401。
请只给出修复思路，不要直接改文件。
```

## 4. 其他添加上下文的方式

### 4.1 图片上下文

如果问题和 UI、报错截图、设计稿有关，可以把图片加入上下文。

支持方式：

- Codex App 中把图片拖进输入框
- IDE Extension 中把图片拖进 prompt
- CLI 中用 `-i` 或 `--image`

CLI 示例：

```bash
codex -i screenshot.png "解释这个报错"
codex --image img1.png,img2.jpg "对比这两个页面的差异"
```

适合：

- 页面截图
- 设计稿
- 流程图
- 控制台错误截图

### 4.2 终端输出上下文

在 Codex App 里，内置终端的当前输出也能被 Codex 读取。这样你可以先运行测试、构建或开发服务，再让 Codex 基于终端结果继续分析。

适合：

- 构建失败排查
- 测试失败定位
- 查看 dev server 启动日志

### 4.3 对话历史本身就是上下文

当前线程里的历史消息，本身就是最重要的上下文之一。所以在同一个线程里连续追问，通常比不断开新会话更有效。

如果会话已经很长，可以用：

```text
/compact
```

让 Codex 对已有对话做压缩总结，尽量保留关键信息、释放上下文空间。

如果你想延续以前的工作，可以用：

```text
/resume
```

重新打开之前的会话。

如果你想在当前上下文基础上分叉尝试另一种方案，可以用：

```text
/fork
```

### 4.4 MCP 工具上下文

如果本地文件还不够，你可以通过 MCP 给 Codex 接入外部工具和外部知识源，比如：

- 文档站
- 浏览器
- Figma
- 内部平台工具

示例：

```bash
codex mcp add context7 -- npx -y @upstash/context7-mcp
```

配置完成后，Codex 就能在会话里调用这些外部工具，把额外上下文带进来。

### 4.5 Apps / Connectors

在 CLI 里可以用 `/apps` 浏览可接入的应用，并把它们插入 prompt。这样 Codex 不只看本地代码，也能结合接入应用中的数据或工具完成任务。

### 4.6 Web 搜索

如果你问的是会变化的信息，比如最新文档、报错解决方案、外部版本差异，Codex 也可以用内置 web search 获取额外上下文。

这个能力更适合：

- 查最新 API 文档
- 查近期变更
- 查外部依赖的行为差异

## 5. 实战建议

如果你想让 Codex 更稳定、更少猜测，可以按这个优先级给上下文：

1. 先把工作目录切对。
2. 再明确附加关键文件或关键目录。
3. 如果只讨论局部逻辑，就选中代码片段或直接贴片段。
4. 如果涉及规则、约束、流程，写进 `AGENTS.md`。
5. 如果涉及外部系统，再接 MCP、Apps、Web 搜索、图片等补充上下文。

## 6. 一句话总结

最常用的做法其实只有几种：

- 看某个文件：IDE 里用 `@文件`，CLI 里用 `/mention 文件路径`
- 看某个目录：在目标目录启动 Codex，或使用 `--cd`、`--add-dir`
- 看某段代码：在 IDE 里选中代码，或者直接把片段贴进 prompt
- 长期生效的规则：写 `AGENTS.md`
- 额外信息源：加图片、终端输出、MCP、Apps、Web 搜索
