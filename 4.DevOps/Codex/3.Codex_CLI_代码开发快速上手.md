# Codex CLI 代码开发快速上手

> **文档职责**：只解决“Codex 已经能启动，接下来怎么拿它做日常代码开发”。  
> **适用场景**：第一次在真实仓库里用 Codex CLI 改代码。  
> **不包含内容**：不讲安装配置细节，不展开审批规则、`AGENTS.md`、CI 和团队协作。  
> **阅读顺序**：先读 `2.Codex_CLI_安装与配置指南.md`，再读本文；需要更强控制力时再读 `4.Codex_CLI_开发进阶指导.md`。

## 1. 先建立正确认知

Codex CLI 最适合执行的不是“零碎补全”，而是一轮完整的小型开发闭环：

1. 读仓库和定位代码
2. 分析问题或方案
3. 修改文件
4. 运行必要检查
5. 审查改动结果

所以你给 Codex 的任务，不要只写“帮我写代码”，而要尽量包含下面四类信息：

- `Goal`：你要它做什么
- `Context`：它应该重点看哪些文件、目录、报错或示例
- `Constraints`：不允许改什么
- `Done when`：什么条件满足才算完成

## 2. 第一次进入项目应该怎么做

### 2.1 从正确目录启动

```bash
cd your-project
codex
```

如果你只想让它聚焦某个子目录，也可以直接：

```bash
codex --cd services/payments
```

### 2.2 先确认两个关键开关

进入 CLI 后，优先确认：

```text
/model
/permissions
```

你要至少知道两件事：

1. 当前模型是否符合任务复杂度
2. 当前权限是否允许它在工作区内执行这次改动

### 2.3 第一句不要直接让它盲改

第一轮建议这样问：

```text
先阅读当前仓库，告诉我：
1. 这是一个什么项目
2. 主要目录结构
3. 启动方式
4. 测试方式
先不要修改文件。
```

这样可以让 Codex 先进入“理解项目”的状态，而不是一上来瞎写。

## 3. 一套最稳的基础开发流程

### 3.1 先分析

```text
定位订单创建相关代码。
请告诉我：
1. 入口在哪
2. service 在哪
3. 数据写入在哪
4. 可能的并发风险点在哪
先分析，不改代码。
```

### 3.2 再限定改动范围

```text
只修改订单创建链路和相关测试。
不要改公共鉴权逻辑，不要引入新依赖，不要改 API 返回结构。
```

### 3.3 最后再下发实现任务

```text
修复订单重复创建问题。

要求：
1. 先定位根因
2. 在最小改动范围内修复
3. 补充必要测试
4. 改完后运行相关测试并总结结果
```

这比一句“修一下 bug”稳定得多。

## 4. 你最常用的上下文补充方式

旧文档里“提示词笔记”和“上下文指南”最容易和快速上手重复。现在只保留最常用、最有价值的几种方式。

### 4.1 指定关键文件

如果你已经知道目标文件，直接用：

```text
/mention src/order/service.py
```

这通常比让 Codex 自己在整个仓库里游走更稳。

### 4.2 贴代码片段

如果只想讨论一段逻辑，直接贴代码，再补一句约束：

```text
下面这段代码在 `src/auth/token.py`。
问题是刷新 token 时偶发 401。
先分析原因，不要直接改文件。
```

### 4.3 用正确目录启动

如果问题本来只在子模块，不要默认让 Codex 看整个仓库：

```bash
codex --cd apps/api
```

如果任务确实跨目录，再补：

```bash
codex --cd apps/frontend --add-dir ../shared
```

## 5. 最常用的几类任务模板

### 5.1 新功能

```text
新增邮件验证码登录能力。

上下文：
1. 复用现有用户模型
2. 尽量复用认证中间件
3. 保持接口风格与现有登录接口一致

约束：
1. 不引入新依赖
2. 不修改数据库主表结构

完成标准：
1. 有发送验证码接口
2. 有验证码登录接口
3. 有最小必要测试
4. 已运行相关测试
```

### 5.2 Bug 修复

```text
修复“并发请求下偶发重复创建订单”的问题。

已知信息：
1. 现象集中在订单创建流程
2. 怀疑与事务边界或幂等控制有关

要求：
1. 先定位根因
2. 不要扩大重构范围
3. 说明复现方式、修复点、验证方式
```

### 5.3 重构

```text
重构支付模块，目标是降低 controller 和 service 的耦合。

约束：
1. 不改变对外接口
2. 不修改返回字段
3. 不要顺手重构无关模块

完成后：
1. 说明重构前后结构
2. 列出关键设计变化
3. 运行相关测试
```

### 5.4 只分析，不落地

```text
分析当前仓库的缓存设计问题。
不要修改文件，只输出：
1. 当前缓存使用点
2. 可能的穿透/击穿风险
3. 最值得优先处理的 3 个问题
```

## 6. 你每天最常用的 CLI 动作

### 6.1 `/mention`

给当前会话明确挂文件。

### 6.2 `/compact`

长会话压缩上下文，防止任务聊散。

### 6.3 `/new`

新任务开新线程，不要把多个需求混到一个会话。

### 6.4 `/resume`

隔一段时间继续旧任务时恢复原线程，而不是从头重新解释。

### 6.5 `/fork`

保留原思路，同时分叉一条新实现路线。

### 6.6 `/review`

改动完成后做一次审查，帮助你发现回归风险、遗漏测试或危险改法。

## 7. 三个最重要的使用习惯

### 7.1 先分析，再修改

先让 Codex 理解现状，再下发改动任务，稳定性会明显更高。

### 7.2 明确“不许做什么”

比“帮我优化一下”更有效的是：

- 不要改接口协议
- 不要新增依赖
- 不要改数据库结构
- 不要碰指定模块之外的文件

### 7.3 明确“改完必须验证什么”

最少补一句：

```text
改完请运行相关测试或检查，并总结结果。
```

否则很容易停留在“代码看起来像改好了”的假完成状态。

## 8. 一个可直接复用的通用模板

```text
目标：
实现/修复 ________

上下文：
1. 相关目录/文件：________
2. 当前报错/现象：________
3. 参考实现：________

约束：
1. 不要改 ________
2. 保持 ________ 兼容
3. 尽量复用现有 ________

完成标准：
1. ________
2. ________
3. 已运行相关测试/检查

输出要求：
1. 先分析
2. 再改代码
3. 最后总结改动文件、验证结果、剩余风险
```

## 9. 新手最常见的误用

### 9.1 任务太大

反例：

```text
把这个项目全部优化一下
```

正确做法是拆成单模块、单 bug、单需求。

### 9.2 没给约束

你不说不能改什么，它就只能自己猜。

### 9.3 没给完成标准

没有验收条件，就容易出现“改了，但不知道算不算完成”。

### 9.4 一个线程塞太多事

上下文一混，结果就开始飘。

### 9.5 不做验证和审查

只看代码，不跑检查，不看 review，是最常见的质量问题来源。

## 10. 10 分钟上手清单

如果你只想最快进入状态，照着做：

1. 在目标仓库执行 `codex`
2. 用 `/model` 和 `/permissions` 看当前会话状态
3. 先让它分析项目，不改代码
4. 第二轮定位目标模块
5. 第三轮再下发明确改动任务
6. 用 `/mention` 挂关键文件
7. 改完让它运行相关检查
8. 最后执行 `/review`
