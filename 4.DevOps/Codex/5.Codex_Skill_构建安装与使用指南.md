# Codex Skill 构建安装与使用指南

> **文档职责**：只解决 Codex Skill 的设计、落库、安装、调用和维护。  
> **适用场景**：你已经会用 Codex CLI，希望把高频工作流沉淀为可复用 Skill。  
> **不包含内容**：不重复讲 Codex CLI 安装，不展开普通代码开发流程。  
> **阅读顺序**：先读 `2.Codex_CLI_安装与配置指南.md` 和 `3.Codex_CLI_代码开发快速上手.md`，再读本文。

## 1. 先明确什么情况下应该做成 Skill

不是所有提示词都值得做成 Skill。

更适合做成 Skill 的场景：

- 你会反复做同一类任务
- 这类任务需要一套稳定步骤
- 任务里有少量专有约束，靠模型常识不够稳
- 你希望以后直接调用，而不是每次重写 prompt

不适合做成 Skill 的场景：

- 只是一次性问题
- 只有一句临时提示词
- 没有稳定工作流
- 没有可复用的边界或参考材料

可以把 Skill 理解为：

- 不是“更长的 prompt”
- 而是“可复用的任务协议 + 少量参考材料”

---

## 2. 一个实用 Skill 最小应该长什么样

推荐最小结构：

```text
skill-name/
├── SKILL.md
├── agents/
│   └── openai.yaml
└── references/
    └── xxx.md
```

各部分职责：

- `SKILL.md`
  负责触发条件、默认工作流、约束、输出契约
- `agents/openai.yaml`
  负责 UI 展示名、简介、默认调用提示
- `references/`
  负责按需加载的补充规则，不把长说明全塞进 `SKILL.md`

可选部分：

- `scripts/`
  当任务存在确定性步骤时再加
- `assets/`
  当 Skill 需要模板、图标或其他产出资源时再加

核心原则：

- `SKILL.md` 保持短而硬
- 参考材料按需拆到 `references/`
- 不要把 Skill 做成一堆给人看的制度文档

---

## 3. 构建 Skill 的推荐流程

### 3.1 先定义 Skill 的边界

先回答 4 个问题：

1. 这个 Skill 解决什么任务
2. 什么请求应该触发它
3. 什么请求不该触发它
4. 它每次工作时必须遵守什么约束

如果这 4 个问题说不清，先不要写 Skill。

### 3.2 再写 `SKILL.md`

`SKILL.md` 至少应包含：

- `name`
- `description`
- 任务目标
- 默认工作流
- 文件职责或输出契约
- 禁止事项

其中最重要的是：

- `description` 要能触发正确场景
- 工作流要稳定
- 约束要能防止任务失控

### 3.3 把细节拆到 `references/`

适合拆出去的内容：

- 判断维度
- 模块库
- 更新规则
- 示例
- 领域参考

不适合继续塞在 `SKILL.md` 里的内容：

- 长篇理论说明
- 大量样例
- 很长的表格
- 多领域细节堆叠

### 3.4 再补 `agents/openai.yaml`

至少补这 3 项：

- `display_name`
- `short_description`
- `default_prompt`

它的作用不是影响推理，而是让 Skill 更容易被发现和调用。

### 3.5 最后用真实任务验证

最少验证 3 类任务：

- 新建
- 更新
- 维护或重构

如果一个 Skill 只能处理“新建”，但一更新就失控，那它还不算成熟。

---

## 4. 推荐把 Skill 放在哪里

如果你希望 Skill 跟着仓库版本走，优先放到仓库内：

```text
.trae/skills/
└── skill-name/
```

优点：

- 可以纳入版本控制
- 可以和项目文档一起维护
- 团队或未来的你更容易理解它为什么这样设计

如果你希望在多个项目里复用，再安装到全局目录：

```text
~/.codex/skills/
└── skill-name/
```

更稳的做法不是二选一，而是：

1. 仓库里保留源码版
2. 全局目录安装可调用版

这样你既能版本管理，又能直接复用。

---

## 5. 本仓库关于自定义 Skill 的说明

本仓库曾用一个“知识库目录规划”类 Skill 作为示例，但这类能力现在已经下沉为 Prompt-first 资产，不再作为独立 Skill 维护。

如果某个能力更适合通过可见 Prompt 模板和少量 reference 复用，而不是通过 Skill 自动触发，就没必要强行保留为 Skill。

这类下沉后的资产更适合放在知识库可见目录里，便于长期维护、直接复制和按需引用。

## 6. 如何安装一个 Skill

如果 Skill 已经在仓库里，最直接的安装方式就是复制到全局目录。

以一个通用自定义 Skill 为例：

```bash
mkdir -p ~/.codex/skills/your-skill-name
cp -R .trae/skills/your-skill-name/. ~/.codex/skills/your-skill-name/
```

安装后的检查方式：

1. 看全局目录里是否已有该 Skill
2. 检查 `SKILL.md` 是否存在
3. 检查 `agents/openai.yaml` 是否存在
4. 检查 `references/` 是否齐全

如果你后面更新了仓库内 Skill，记得重新复制一次到全局目录。

## 7. 如何使用 Skill

最直接的使用方式，是在请求里显式点名 Skill：

```text
$your-skill-name ...
```

这个 Skill 的具体行为取决于它自己的设计与约束。

推荐把请求写成这 4 段：

- 任务目标
- 主题或目录范围
- 重点关注点
- 约束条件

### 7.1 先规划，不改文件

```text
$your-skill-name 帮我先做结构诊断，再给最小修改建议。不要直接执行改动。
```

### 7.2 只做主题结构规划

```text
$your-skill-name 先判断当前任务类型，再给出目录或文件职责建议。不要修改文件。
```

### 7.3 更新已有主题时也只给建议

```text
$your-skill-name 基于新增材料分析应该改哪些文件以及原因。不要修改文件。
```

### 7.4 审计并输出重组建议

```text
$your-skill-name 检查这个目录是否已经失控，找出重复、错放和可合并内容，并给出最小重组方案。不要修改文件。
```

---

## 8. 什么时候该改 Skill，而不是继续补 prompt

出现下面这些情况时，优先改 Skill：

- 同一类任务你已经重复写了 3 次以上 prompt
- 模型经常在同一处犯错
- 你已经形成稳定的判断步骤
- 你希望把经验沉淀为长期资产

出现下面这些情况时，继续用普通 prompt 就够了：

- 任务很临时
- 没有稳定步骤
- 没有明显复用价值

---

## 9. 维护 Skill 的最小原则

维护 Skill 时，优先只改真正长期稳定的内容：

- 触发条件
- 默认工作流
- 关键约束
- 参考材料结构

不要频繁修改：

- Skill 名称
- 目录结构
- 文件职责

如果只是某个示例需要调整，优先改 `references/examples.md`，不要每次都重写 `SKILL.md`。

---

## 10. 一条最实用的判断标准

如果某类工作你希望未来直接说一句话就能让 Codex 进入正确工作状态，那么它就值得做成 Skill。

是否值得做成 Skill，取决于它是否真的需要“被动触发 + 稳定流程 + 独立维护”。如果更适合 Prompt-first 复用，就不必勉强保留成 Skill。
