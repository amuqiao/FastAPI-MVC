# Codex Skill 构建安装与使用指南

> **文档职责**：只解决 Codex Skill 的设计、落库、安装、调用和维护。  
> **适用场景**：你已经会用 Codex CLI，希望把高频工作流沉淀为可复用 Skill。  
> **不包含内容**：不重复讲 Codex CLI 安装，不展开普通代码开发流程。  
> **阅读顺序**：先读 `2.Codex_CLI_安装与配置指南.md` 和 `3.Codex_CLI_代码开发快速上手.md`，再读本文。

## 1. 先明确什么情况下应该做成 Skill

不是所有提示词都值得做成 Skill。

更适合做成 Skill 的场景：

- 你会反复做同一类任务
- 这类任务需要一套稳定步骤
- 任务里有少量专有约束，靠模型常识不够稳
- 你希望以后直接调用，而不是每次重写 prompt

不适合做成 Skill 的场景：

- 只是一次性问题
- 只有一句临时提示词
- 没有稳定工作流
- 没有可复用的边界或参考材料

可以把 Skill 理解为：

- 不是“更长的 prompt”
- 而是“可复用的任务协议 + 少量参考材料”

---

## 2. 一个实用 Skill 最小应该长什么样

推荐最小结构：

```text
skill-name/
├── SKILL.md
├── agents/
│   └── openai.yaml
└── references/
    └── xxx.md
```

各部分职责：

- `SKILL.md`
  负责触发条件、默认工作流、约束、输出契约
- `agents/openai.yaml`
  负责 UI 展示名、简介、默认调用提示
- `references/`
  负责按需加载的补充规则，不把长说明全塞进 `SKILL.md`

可选部分：

- `scripts/`
  当任务存在确定性步骤时再加
- `assets/`
  当 Skill 需要模板、图标或其他产出资源时再加

核心原则：

- `SKILL.md` 保持短而硬
- 参考材料按需拆到 `references/`
- 不要把 Skill 做成一堆给人看的制度文档

---

## 3. 构建 Skill 的推荐流程

### 3.1 先定义 Skill 的边界

先回答 4 个问题：

1. 这个 Skill 解决什么任务
2. 什么请求应该触发它
3. 什么请求不该触发它
4. 它每次工作时必须遵守什么约束

如果这 4 个问题说不清，先不要写 Skill。

### 3.2 再写 `SKILL.md`

`SKILL.md` 至少应包含：

- `name`
- `description`
- 任务目标
- 默认工作流
- 文件职责或输出契约
- 禁止事项

其中最重要的是：

- `description` 要能触发正确场景
- 工作流要稳定
- 约束要能防止任务失控

### 3.3 把细节拆到 `references/`

适合拆出去的内容：

- 判断维度
- 模块库
- 更新规则
- 示例
- 领域参考

不适合继续塞在 `SKILL.md` 里的内容：

- 长篇理论说明
- 大量样例
- 很长的表格
- 多领域细节堆叠

### 3.4 再补 `agents/openai.yaml`

至少补这 3 项：

- `display_name`
- `short_description`
- `default_prompt`

它的作用不是影响推理，而是让 Skill 更容易被发现和调用。

### 3.5 最后用真实任务验证

最少验证 3 类任务：

- 新建
- 更新
- 维护或重构

如果一个 Skill 只能处理“新建”，但一更新就失控，那它还不算成熟。

## 4. 推荐把 Skill 放在哪里

如果你希望 Skill 跟着仓库版本走，优先放到仓库内：

```text
.trae/skills/
└── skill-name/
```

优点：

- 可以纳入版本控制
- 可以和项目文档一起维护
- 团队或未来的你更容易理解它为什么这样设计

如果你希望在多个项目里复用，再安装到全局目录：

```text
~/.codex/skills/
└── skill-name/
```

更稳的做法不是二选一，而是：

1. 仓库里保留源码版
2. 全局目录安装可调用版

这样你既能版本管理，又能直接复用。

## 5. 本仓库里的 `knowledge-base-planner` Skill 是怎么落的

当前已经创建了仓库内源码版：

```text
.trae/skills/knowledge-base-planner/
├── SKILL.md
├── agents/
│   └── openai.yaml
└── references/
    ├── examples.md
    ├── module-library.md
    ├── topic-diagnosis.md
    └── update-rules.md
```

同时也已经安装到了全局目录：

```text
~/.codex/skills/knowledge-base-planner/
├── SKILL.md
├── agents/
│   └── openai.yaml
└── references/
    ├── examples.md
    ├── module-library.md
    ├── topic-diagnosis.md
    └── update-rules.md
```

这个 Skill 解决的问题不是“围绕某个具体主题写一篇知识库方法文档”，而是作为你的个人知识库规划与诊断 Skill，统一处理仓库级和主题级的结构设计问题。

它支持的不是单一主题，而是围绕整个个人知识库的主题生命周期：

- 路由主题到正确领域
- 规划主题目录
- 设计主题骨架
- 分析已有主题
- 审计主题结构
- 给出重组建议

它的核心思路是：

1. 先判断这是仓库级任务还是主题级任务
2. 再判断主题归属和主题类型
3. 再选择需要的模块
4. 再做最小必要结构
5. 最后只给方案和建议，不直接改文件

也就是说，它不强制每个主题都长成“基础概念 / 核心原理 / 工程实践”，也不局限于 `macOS` 这类操作型主题。

如果你后面还需要一个真正负责写入和更新文档的 skill，更合理的做法是另建一个执行型 skill，例如：

- `knowledge-base-editor`
- 或 `knowledge-base-maintainer`

## 6. 如何安装一个 Skill

如果 Skill 已经在仓库里，最直接的安装方式就是复制到全局目录。

以本仓库的 `knowledge-base-planner` 为例：

```bash
mkdir -p ~/.codex/skills/knowledge-base-planner
cp -R .trae/skills/knowledge-base-planner/. ~/.codex/skills/knowledge-base-planner/
```

安装后的检查方式：

1. 看全局目录里是否已有该 Skill
2. 检查 `SKILL.md` 是否存在
3. 检查 `agents/openai.yaml` 是否存在
4. 检查 `references/` 是否齐全

如果你后面更新了仓库内 Skill，记得重新复制一次到全局目录。

## 7. 如何使用 Skill

最直接的使用方式，是在请求里显式点名 Skill：

```text
$knowledge-base-planner ...
```

这个 planner skill 默认只负责“先规划，后诊断”，不负责直接落地修改。

推荐把请求写成这 4 段：

- 任务目标
- 主题或目录范围
- 重点关注点
- 约束条件

### 7.1 先规划，不改文件

```text
$knowledge-base-planner 我想新增“消息队列可靠性”这个主题。
先判断它应该归到哪个顶层领域，是否应该独立成主题目录，再给出最小可维护结构。不要修改文件。
```

### 7.2 只做主题结构规划

```text
$knowledge-base-planner 围绕“Transformer”规划一个适合长期维护的主题知识库。
先判断主题类型，再给出目录和文件职责，不要修改文件。
```

### 7.3 更新已有主题时也只给建议

```text
$knowledge-base-planner 基于新增材料更新“4.DevOps/Git”主题。
先说明该改哪些文件以及原因，不要修改文件。
```

### 7.4 审计并输出重组建议

```text
$knowledge-base-planner 检查“2.AI 工程/agentscope”目录是否已经失控。
找出重复、错放、可合并内容，并给出最小重组方案。不要修改文件。
```

## 8. 什么时候该改 Skill，而不是继续补 prompt

出现下面这些情况时，优先改 Skill：

- 同一类任务你已经重复写了 3 次以上 prompt
- 模型经常在同一处犯错
- 你已经形成稳定的判断步骤
- 你希望把经验沉淀为长期资产

出现下面这些情况时，继续用普通 prompt 就够了：

- 任务很临时
- 没有稳定步骤
- 没有明显复用价值

---

## 9. 维护 Skill 的最小原则

维护 Skill 时，优先只改真正长期稳定的内容：

- 触发条件
- 默认工作流
- 关键约束
- 参考材料结构

不要频繁修改：

- Skill 名称
- 目录结构
- 文件职责

如果只是某个示例需要调整，优先改 `references/examples.md`，不要每次都重写 `SKILL.md`。

---

## 10. 一条最实用的判断标准

如果某类工作你希望未来直接说一句话就能让 Codex 进入正确工作状态，那么它就值得做成 Skill。

`knowledge-base-planner` 就属于这一类，而且它面向的是“任意主题”的生命周期管理，不是围绕一个固定主题定制。

你以后不需要再为 `macOS`、`Git`、`Kafka`、`智能体平台` 分别维护一套专用知识库 prompt。  
你只需要维护好一个通用的知识库规划 Skill，然后在每次任务里把主题、范围和约束说清楚。
