# Codex 日常使用速查清单

> **文档职责**：提供 Codex 日常使用时最常用的命令、提问模板、任务流程和排错清单。  
> **适用场景**：你已经会启动 Codex，但不想每次都回忆“这一步该怎么说、那个命令是什么”。  
> **阅读方式**：把它当成工作台旁边的速查卡，需要时直接复制。

## 1. 每次开工前先过一遍

- 当前目录是不是正确项目目录
- 当前任务是“只分析”还是“要改代码”
- 改动范围是否已经明确
- 有没有不能动的目录、模块或接口
- 改完后要跑什么验证

如果你还没想清楚这 5 件事，不要直接让 Codex 开始改。

---

## 2. 最常用的启动方式

### 2.1 在当前项目根目录启动

```bash
cd your-project
codex
```

### 2.2 只盯一个子目录

```bash
codex --cd services/payments
```

### 2.3 主目录之外再加额外上下文

```bash
codex --cd apps/frontend --add-dir ../shared --add-dir ../backend
```

原则：

- 能缩小范围就缩小范围
- 能只看子目录，就别默认看整个仓库

---

## 3. 第一句最好怎么说

第一次进入一个项目，推荐先用这句：

```text
先阅读当前仓库，告诉我：
1. 这是一个什么项目
2. 主要目录结构
3. 启动方式
4. 测试方式
先不要修改文件。
```

这一步的作用是：

- 让 Codex 先理解项目
- 避免一上来盲改
- 顺手摸清测试和启动命令

## 4. 日常最常用的 4 类任务模板

### 4.1 只分析，不改代码

```text
分析当前问题，告诉我：
1. 相关代码入口
2. 核心调用链路
3. 可能根因
4. 最小修复方案
先不要修改文件。
```

### 4.2 修一个 bug

```text
修复“XXX”问题。

要求：
1. 先定位根因
2. 在最小改动范围内修复
3. 不要改无关模块
4. 补充必要测试
5. 改完后运行相关验证并总结结果
```

### 4.3 做一个新功能

```text
新增“XXX”能力。

上下文：
1. 复用现有模块
2. 保持现有接口风格

约束：
1. 不引入新依赖
2. 不修改数据库主表结构

完成标准：
1. 功能可用
2. 有最小必要测试
3. 已完成相关验证
```

### 4.4 做重构

```text
重构“XXX”模块，目标是降低耦合或简化结构。

约束：
1. 不改变对外接口
2. 不修改返回字段
3. 不顺手扩大重构范围

完成后：
1. 说明改动点
2. 说明风险点
3. 运行相关测试
```

## 6. 最常用的 CLI 动作

### 6.1 挂文件到上下文

```text
/mention path/to/file
```

适合：

- 你已经知道关键文件
- 不想让 Codex 在全仓库乱找

### 6.2 生成或补仓库规则

```text
/init
```

适合：

- 仓库还没有 `AGENTS.md`
- 你想先生成一版协作规则骨架

### 6.3 压缩长会话

```text
/compact
```

适合：

- 当前会话已经很多轮
- 准备从分析进入实施
- 你感觉上下文开始发散

### 6.4 继续旧任务

```text
/resume
```

适合：

- 隔天接着做
- 中途切走后回来继续

### 6.5 开分支线程试方案

```text
/fork
```

适合：

- 想保留原思路
- 想并行试另一种实现

## 7. 每次改代码前最好先说清楚

- 要改哪个模块
- 不能改哪个模块
- 是否允许改测试
- 是否允许新增文件
- 是否允许新增依赖
- 是否允许改接口、数据库、配置

如果你不说，Codex 只能自己猜。

## 8. 最稳的开发闭环

推荐顺序：

1. 先分析
2. 再收缩范围
3. 再实施改动
4. 再跑验证
5. 最后看变更总结

对应一句话模板：

```text
先定位问题和相关文件，再在最小改动范围内实现修复，补充必要测试，并运行相关验证，最后总结改动点和结果。
```

## 9. 日常最常见的 3 种模式

### 9.1 看代码模式

适合：

- 理解项目
- 查链路
- 找 bug
- 做 review

推荐说法：

```text
先分析，不要修改文件。
```

### 9.2 改代码模式

适合：

- 修 bug
- 新增小功能
- 小范围重构

推荐说法：

```text
在最小改动范围内处理，并运行相关验证。
```

### 9.3 规划模式

适合：

- 设计文档
- 目录规划
- Skill 设计
- Plugin 选型

推荐说法：

```text
先给方案，不要直接改文件。
```

## 10. Skill 什么时候用

当你发现某类任务会反复出现，而且有稳定步骤时，就该考虑 `Skill`。

典型信号：

- 你总在重复同类 prompt
- 总在反复强调同类约束
- 总在做一类相似任务

一句话判断：

**偏任务方法和工作流沉淀，用 `Skill`。**

---

## 11. Plugin 什么时候用

当你需要：

- 正式扩展能力
- 打包多个能力
- 安装和管理入口
- marketplace
- hooks / MCP / app 配置

这时才考虑 `Plugin`。

一句话判断：

**偏产品化扩展和能力打包接入，用 `Plugin`。**

---

## 12. Script 什么时候就够了

如果问题只是：

- 一个重复动作
- 一个确定性流程
- 不需要模型判断

那通常写 `Script` 就够了，不要先上 Skill 或 Plugin。

例如：

- 批量重命名
- 生成脚手架
- 扫描目录输出索引
- 一键跑检查并汇总结果

## 13. 最常见的提问误区

### 13.1 任务太模糊

反例：

```text
帮我优化一下这个项目。
```

更好的说法：

```text
分析当前缓存设计问题，找出最值得优先处理的 3 个问题。先不要修改文件。
```

### 13.2 没有约束

反例：

```text
修一下订单问题。
```

更好的说法：

```text
修复订单重复创建问题，只修改订单创建链路和相关测试，不要改公共鉴权逻辑。
```

### 13.3 没有验收标准

反例：

```text
把登录做好。
```

更好的说法：

```text
新增邮件验证码登录能力，要求接口可用、复用现有用户模型、补最小必要测试，并运行相关验证。
```

## 14. 日常排错速查

如果 Codex 用着不顺，优先检查：

- 当前工作目录对不对
- 当前上下文是不是太大
- 任务是不是同时混了分析、开发、文档、排障
- 约束是不是没说清楚
- 验证方式是不是没提前讲
- 会话是不是太长，该不该 `/compact`

很多“它怎么乱改了”的根因，其实不是模型问题，而是任务边界没立住。
