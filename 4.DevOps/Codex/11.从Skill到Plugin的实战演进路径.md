# 从 Skill 到 Plugin 的实战演进路径

> **文档职责**：讲清一个能力为什么通常应该先做成 `Skill`，以及在什么节点再升级成 `Plugin`。  
> **适用场景**：你已经知道 `Skill` 和 `Plugin` 不一样，但还拿不准工程上应该先做哪个、后做哪个。  
> **阅读目标**：看完后能给一个能力设计出合理的落地顺序，避免一开始就过度设计。

## 1. 先说结论

更稳的工程顺序通常不是：

- 一上来就做 Plugin

而是：

1. 先确认问题是不是高频且稳定
2. 能脚本化的先脚本化
3. 需要任务工作流时做 `Skill`
4. 需要正式接入、安装和组合能力时再升级成 `Plugin`

一句话说：

**先用最轻的结构验证需求，再决定要不要升级成更重的结构。**

---

## 2. 为什么不建议一开始就做 Plugin

很多人第一次扩展 Codex 能力时，容易直接想到：

- 做一个插件
- 补一堆目录
- 设计 marketplace
- 顺手把 hooks、MCP、脚本都带上

这样做的问题是：

- 需求其实还没稳定
- 任务边界还不清楚
- 很多结构只是“先摆着”
- 后面一改方向，整个扩展壳都要跟着重构

所以更常见的情况是：

你以为自己要的是 `Plugin`，实际上你先需要的是：

- 一个清晰的任务边界
- 一套稳定的处理流程
- 一点点必要的自动化

这通常正是 `Skill` 或 `Script` 更适合先解决的事。

---

## 3. 最推荐的四步演进法

### 第一步：先判断这是不是一个“稳定问题”

先问自己：

- 这类任务是不是会反复出现
- 处理步骤是不是基本稳定
- 约束是不是能说清楚
- 输出是不是有稳定形态

如果这些问题还答不上来，先不要做 `Plugin`，甚至先不要做 `Skill`。

这个阶段更适合：

- 先手工做几次
- 记录判断标准
- 观察哪些步骤是真正重复的

如果任务本身都不稳定，结构越重，后面返工越大。

### 第二步：能脚本化的先脚本化

当你发现有些步骤已经很明确，而且几乎不需要模型判断时，先抽成 `Script`。

典型例子：

- 批量重命名
- 批量生成脚手架
- 导出固定格式
- 扫描目录并输出索引
- 跑检查命令并汇总结果

这里不要一上来就做 Skill。

因为这类事情的核心不是“让模型更会推理”，而是：

**让动作稳定执行。**

### 第三步：需要任务工作流时做 Skill

当你发现问题已经不是“单个命令”能解决，而是需要：

- 判断任务属于哪一类
- 决定执行顺序
- 限制输出边界
- 在不同上下文下做一致处理

这时候做 `Skill` 最合适。

`Skill` 适合承接的是：

- 任务边界
- 默认工作流
- 约束规则
- 输出契约
- 与脚本或参考资料的协作方式

这个阶段的重点不是扩展产品化，而是：

**把做事方式固化下来。**

### 第四步：需要正式接入时再升级成 Plugin

只有在下面这些需求真的出现时，再考虑升级成 `Plugin`：

- 需要可安装、可展示、可管理
- 需要 marketplace 入口
- 需要把多个 skill / script / hooks / MCP 配置打包
- 需要作为团队统一扩展分发
- 需要一个正式的扩展边界

这个时候，`Plugin` 才是在解决真实问题，而不是增加结构负担。

---

## 4. 一个最现实的判断标准

如果你现在犹豫“该不该直接做 Plugin”，可以看这个标准：

### 4.1 如果你主要在定义“怎么做”

比如你在定义：

- 任务流程
- 判断逻辑
- 输出格式
- 触发条件
- 禁止事项

那你主要做的是：`Skill`

### 4.2 如果你主要在定义“怎么接进来”

比如你在定义：

- 插件入口
- marketplace
- 扩展包结构
- hooks
- MCP 配置
- 多能力打包关系

那你主要做的是：`Plugin`

这两个重心完全不一样。

很多人混乱，就是因为还在定义“怎么做”，却已经开始搭“怎么接进来”的壳。

---

## 5. 一个合理的演进示例

假设你要做一个“数据库审查助手”。

很多人一开始会想：

- 先建插件
- 再接数据库
- 再补 marketplace
- 再写 skill

但更稳的方式通常是：

### 阶段 1：先做 Script

先把确定性动作抽出来，比如：

- 执行 explain
- 收集索引信息
- 格式化 SQL
- 生成基础报告

这一步的目标是：

- 先确认哪些动作稳定
- 先把重复动作自动化

### 阶段 2：再做 Skill

然后做一个 `sql-reviewer` skill，负责：

- 什么时候该跑 explain
- 什么时候要关注索引
- 输出报告该按什么结构写
- 哪些风险要优先标出来

这一步的目标是：

- 把审查方法固定下来

### 阶段 3：最后再考虑 Plugin

只有在你真的需要下面这些时，再上 Plugin：

- 团队统一安装
- 做正式扩展
- 集成多个相关能力
- 接数据库工具或 MCP

这个时候，Plugin 里可以放：

- `skills/sql-reviewer`
- `scripts/`
- `.mcp.json`
- `.codex-plugin/plugin.json`

这时的 Plugin 是成熟能力的打包，不是空壳。

## 6. Skill 升级成 Plugin，通常意味着什么

从 `Skill` 升级成 `Plugin`，不是简单地“多建几个目录”。

它通常意味着这几件事发生了变化：

### 6.1 从个人工作流，变成了扩展能力

原来你只是自己用；
现在你希望别人也能装、能看、能统一使用。

### 6.2 从单一任务协议，变成了能力组合

原来只有一个 Skill；
现在可能要组合：

- 多个 Skill
- 脚本
- hooks
- MCP

### 6.3 从“会做事”，变成“能接入系统”

原来重点是流程和规则；
现在还要考虑：

- 如何安装
- 如何展示
- 如何管理入口
- 如何和其他能力并存

所以升级成 Plugin 的前提不是“我想更正规一点”，而是：

**能力边界真的变了。**

---

## 7. 什么时候不要升级

下面这些情况，不建议急着把 Skill 升级成 Plugin：

- 只有你自己在用
- 需求还在快速变化
- 任务边界还不稳定
- 只有一个 Skill，且没有接入需求
- 只是想“看起来更完整”

这类时候升级成 Plugin，收益通常不高，维护成本却会上升。

最常见的问题是：

- 多了目录
- 多了配置
- 多了入口维护
- 但实际能力没有变强

这就是典型的结构先行、需求滞后。

---

## 8. 什么时候应该升级

下面这些信号出现时，可以认真考虑升级成 Plugin：

- 你已经有 2 个以上相关 Skill 想放在一起
- 你需要统一的插件入口
- 你要挂 marketplace
- 你需要 hooks 或 MCP 配置
- 你要把能力交给团队或多个项目使用
- 你要把扩展做成正式产品形态

这时升级成 Plugin 就不是为了“看起来完整”，而是因为：

**Plugin 已经开始解决真实的分发和接入问题。**

---

## 9. 一个非常实用的演进决策表

| 当前状态 | 更适合的形态 | 原因 |
|---|---|---|
| 只是一个重复动作 | `Script` | 重点是稳定执行，不是任务推理 |
| 有稳定任务流程，但不需要安装入口 | `Skill` | 重点是工作流和约束 |
| 已有多个相关能力，需要统一组织 | `Plugin` | 重点是打包和管理 |
| 需要对接外部系统 | `MCP` + `Skill`，必要时再放进 `Plugin` | 重点是能力接入，不只是任务规则 |
| 个人探索期，边界不清晰 | 先别上 Plugin | 先验证需求，再定结构 |

---

## 10. 推荐你在项目里怎么落

如果你准备做一个新能力，可以按这个顺序推进：

1. 先写下任务边界
2. 把可重复的动作抽成脚本
3. 把判断逻辑和工作流沉淀成 Skill
4. 观察是否真的出现安装、打包、市场、统一入口的需求
5. 只有这些需求真实存在时，再升级成 Plugin

这个顺序的好处是：

- 不容易过度设计
- 更容易验证真实需求
- 返工成本更低
- 结构会更贴近实际使用方式

---

## 11. 一句话收尾

不要把 `Plugin` 当成“比 Skill 更高级”的版本。

更准确的理解是：

- `Skill` 解决“怎么做”
- `Plugin` 解决“怎么接进来、怎么打包、怎么分发”

真正好的工程路径，不是谁更高级，而是：

**先用最轻的结构解决当前问题，再在需求逼近时升级结构。**
