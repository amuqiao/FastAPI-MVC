# Plugin 与 Skill 区别及选型指南

> **文档职责**：讲清 `Plugin`、`Skill`、`MCP`、`Script` 分别解决什么问题，以及项目里到底该先做哪个。  
> **适用场景**：你准备扩展 Codex 能力，但不确定应该做成 Skill、Plugin，还是只写脚本。  
> **阅读目标**：看完后能直接判断“这件事该做成什么”。

## 1. 先说结论

可以先记住这句话：

**Skill 是让 Codex 更会做事，Plugin 是让 Codex 多一种可安装、可接入、可管理的扩展能力。**

也就是说：

- `Skill` 更偏任务方法、工作流、规则和领域知识
- `Plugin` 更偏能力打包、安装入口、结构容器和产品化扩展

如果你现在只想解决一个高频任务，通常先做 `Skill`。  
如果你想做一个正式扩展，里面可能带 `Skill`、脚本、hooks、MCP 配置，那通常做 `Plugin`。

---

## 2. Skill 到底是什么

`Skill` 本质上是一个“任务能力包”。

它通常包含：

- `SKILL.md`
- 可选的 `references/`
- 可选的 `scripts/`
- 可选的 `assets/`

它解决的问题是：

- 让 Codex 知道这类任务应该怎么做
- 哪些请求该触发这个 Skill
- 默认工作流是什么
- 约束是什么
- 输出应该长什么样

所以 `Skill` 更像：

- 专家操作手册
- 任务 SOP
- 一类问题的标准处理协议

典型场景：

- 知识库规划
- 某类代码审查
- 某类文档生成
- 某种固定分析流程

一句话理解：

**Skill 重点不是“多一个功能按钮”，而是“让模型在某类任务上更稳定”。**

---

## 3. Plugin 到底是什么

`Plugin` 本质上是一个“扩展能力容器”。

它至少会有：

- `.codex-plugin/plugin.json`

还可以带：

- `skills/`
- `hooks/`
- `scripts/`
- `assets/`
- `.mcp.json`
- `.app.json`

它解决的问题是：

- 把一组能力做成正式扩展
- 让能力能被安装、管理、展示
- 让能力在仓库或 marketplace 里有稳定入口

所以 `Plugin` 更像：

- 一个插件壳
- 一个扩展包
- 一个能力分发单元

典型场景：

- 做一个本地插件
- 做一个团队统一使用的扩展
- 想挂 marketplace
- 想把多个能力组织成一个产品化模块

一句话理解：

**Plugin 重点不是“教模型怎么思考”，而是“把能力装进去并接入 Codex”。**

---

## 4. 最直接的区别

可以直接用这张表判断：

| 对象 | 核心作用 | 主要内容 | 解决的问题 | 是否可单独用 | 典型场景 |
|---|---|---|---|---|---|
| `Skill` | 定义任务工作流 | `SKILL.md`、`references/`、`scripts/`、`assets/` | 让 Codex 更会做某类事 | 可以 | 知识库规划、代码审查、文档生成 |
| `Plugin` | 打包和接入扩展能力 | `.codex-plugin/plugin.json`，可带 `skills/`、`hooks/`、`scripts/`、`.mcp.json`、`.app.json` | 让能力可安装、可管理、可展示 | 可以 | 本地插件、市场插件、团队统一分发 |
| `MCP` | 接外部工具或服务 | MCP server 配置、工具定义 | 让 Codex 能调用外部系统 | 通常作为能力源存在 | 查数据库、调内部 API、接第三方系统 |
| `Script` | 执行确定性动作 | Python/Bash 脚本 | 把重复动作稳定自动化 | 可以 | 批处理、脚手架生成、格式转换 |

---

## 5. 它们之间是什么关系

最常见的关系是：

```text
Plugin
├── skills/
│   └── 一个或多个 Skill
├── scripts/
│   └── 可执行脚本
├── hooks/
├── .mcp.json
└── .codex-plugin/plugin.json
```

所以这几个概念不要混：

- `Skill` 是“怎么做”
- `Script` 是“具体执行”
- `MCP` 是“连外部能力”
- `Plugin` 是“把这些东西打成一个可安装扩展”

也就是说，`Plugin` 可以包含 `Skill`，但 `Skill` 不等于 `Plugin`。

---

## 6. 最常见的四种组合

### 6.1 只有 Skill

适合：

- 你只是想把某类高频任务沉淀下来
- 不需要 marketplace
- 不需要做成正式插件

例如：

- `sql-reviewer`
- `doc-structure-auditor`

这种场景下，做 `Skill` 就够了。

### 6.2 Skill + Script

适合：

- 既需要任务规则
- 又有一部分动作必须稳定执行

例如：

- Skill 负责告诉 Codex 如何审查 SQL
- Script 负责跑 explain、格式化 SQL、输出报告

这种场景很常见，也很实用。

### 6.3 Plugin + Skill

适合：

- 你希望把能力做成正式扩展
- 并且这个扩展里包含一个或多个任务型能力

例如：

- 一个团队内部“数据库助手插件”
- 插件里带 `schema-review`、`migration-check` 两个 Skill

### 6.4 Plugin + MCP + Skill

适合：

- 你不仅要有任务流程
- 还要接真实外部系统

例如：

- 插件接公司内部工单系统、数据库、知识库 API
- Skill 负责告诉 Codex 在什么情况下调用哪个工具

这已经不是“写一点提示词”了，而是完整扩展能力。

---

## 7. 什么时候该做 Skill

以下场景优先做 `Skill`：

- 你会反复做同一类任务
- 任务有稳定步骤
- 需要一些专有规则约束模型
- 不需要正式的插件安装入口
- 重点是“让模型更懂你的做法”

比如下面这些请求，明显更适合 Skill：

- 帮我做知识库主题规划
- 帮我审查 SQL 性能风险
- 帮我把需求文档整理成固定输出格式
- 帮我按团队规范生成接口设计文档

这类事情的关键不是“多接一个外部系统”，而是“让任务流程稳定”。

所以答案通常是：**先做 Skill。**

---

## 8. 什么时候该做 Plugin

以下场景优先做 `Plugin`：

- 你希望能力可安装、可管理、可展示
- 你需要 marketplace 入口
- 你要把多种能力放在一个统一扩展里
- 你需要 hooks、MCP、app 配置等插件结构
- 你想做成团队可复用的正式扩展包

比如下面这些请求，更适合 Plugin：

- 搭一个本地 Codex 插件
- 做一个团队内部数据库插件
- 给 Codex 增加一个可安装的业务扩展
- 把一组工具和 skill 打包成统一插件

这类事情的关键不是“只定义任务流程”，而是“定义扩展边界和接入形式”。

所以答案通常是：**做 Plugin。**

---

## 9. 什么时候根本不用 Plugin，也不用 Skill

这个判断也很重要。

下面这些情况，往往只要 `Script` 就够了：

- 只是一个一次性自动化动作
- 不需要任务推理
- 不需要触发逻辑
- 不需要模型理解上下文

例如：

- 批量重命名文件
- 导出某种固定格式
- 扫描目录生成索引
- 一键创建脚手架

这种事如果硬做成 Skill，反而会过度设计。

一句话：

**能用纯脚本解决的确定性动作，不要先上 Plugin。**

---

## 10. 一个最实用的决策清单

你可以按下面顺序问自己：

### 10.1 这是“任务方法问题”还是“扩展接入问题”

- 如果重点是任务规则、流程、输出格式：偏 `Skill`
- 如果重点是安装、接入、组合能力：偏 `Plugin`

### 10.2 是否需要外部系统能力

- 不需要：先看 `Skill` 或 `Script`
- 需要：看是否要接 `MCP`，再决定是否做 `Plugin`

### 10.3 是否需要可安装、可展示、可挂市场

- 不需要：通常不用 `Plugin`
- 需要：通常应该做 `Plugin`

### 10.4 是否只是一个确定性动作

- 是：先做 `Script`
- 不是：再看是否需要 `Skill`

---

## 11. 你可以这样快速判断

下面是最简判断法：

- 只想让 Codex 更会做某类任务：`Skill`
- 只想把一个动作自动化：`Script`
- 想接外部系统：`MCP`
- 想正式打包、安装、挂市场、组合多能力：`Plugin`

如果还拿不准，可以继续用这个一句话判断：

**偏任务方法和工作流沉淀，用 `Skill`；偏产品化扩展和能力打包接入，用 `Plugin`。**

---

## 12. 两个典型例子

### 例子 1：知识库规划助手

需求：

- 判断新主题放哪里
- 规划目录结构
- 给出文件职责

这个场景重点是：

- 任务方法
- 结构判断
- 规则约束

所以更适合：`Skill`

### 例子 2：数据库助手插件

需求：

- 能连接数据库
- 能做 schema 分析
- 能提供多个相关技能
- 能作为一个正式扩展交给团队使用

这个场景重点是：

- 能力组合
- 外部接入
- 安装和管理

所以更适合：`Plugin`

插件内部还可以继续带多个 `Skill`。

---

## 13. 最后一句话

不要一开始就把所有东西都做成 Plugin。

更稳的工程顺序通常是：

1. 先确认问题是不是高频且稳定
2. 能脚本化的先脚本化
3. 需要任务工作流时做 `Skill`
4. 需要正式接入、安装和组合能力时再升级成 `Plugin`

这样不容易过度设计，也更适合长期维护。
