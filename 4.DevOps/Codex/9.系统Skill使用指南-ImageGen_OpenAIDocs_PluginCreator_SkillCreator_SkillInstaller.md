# 系统 Skill 使用指南：Image Gen / OpenAI Docs / Plugin Creator / Skill Creator / Skill Installer

> **文档职责**：讲清 5 个常用系统 Skill 的定位、触发方式、标准用法和典型示例。  
> **适用场景**：你已经在使用 Codex，希望把图片生成、官方文档查询、插件脚手架、Skill 设计与 Skill 安装这几类工作稳定下来。  
> **阅读方式**：先看“怎么调用 Skill”，再按需跳到具体 Skill 的示例。

## 1. 先理解：Skill 不是普通提示词

在 Codex 里，Skill 可以理解为：

- 一套面向某类任务的固定工作流
- 一组已经约定好的工具使用规则
- 少量按需加载的参考资料或脚本

所以调用 Skill 的目标，不是“换个名字提问”，而是让 Codex进入更稳定的工作模式。

---

## 2. 这些 Skill 怎么触发

常见触发方式有两种：

### 2.1 直接点名 Skill

你可以在请求里直接写 Skill 名称，例如：

```text
使用 Image Gen 生成一张首页 hero 图。
```

或者：

```text
$skill-installer 列出当前可安装的 curated skills。
```

这种方式最稳，因为 Codex 能明确知道你要用哪一个 Skill。

### 2.2 不点名，但任务本身明显匹配

例如你说：

```text
帮我搭一个 Codex 插件脚手架，包含 marketplace 入口。
```

这类请求虽然没有写 `Plugin Creator`，但任务和 Skill 职责完全匹配，也会触发对应 Skill。

---

## 3. 一个高质量请求建议包含什么

不管你用哪个 Skill，最好都尽量补齐这 4 类信息：

- 目标：你到底要产出什么
- 范围：只做规划，还是要直接创建文件/执行命令
- 约束：不能改什么，必须保留什么
- 示例或上下文：已有文件、目标目录、参考风格、目标平台

一个过于模糊的请求：

```text
用 Plugin Creator 处理一下。
```

一个更好的请求：

```text
使用 Plugin Creator，在当前仓库创建一个 repo-local 插件 `mysql-inspector`，
需要生成 marketplace 入口，并额外包含 `skills/` 和 `scripts/` 目录。
先直接创建文件，不需要我二次确认。
```

## 4. Image Gen：图片生成与编辑

### 4.1 它适合做什么

`Image Gen` 适合处理位图类资产，例如：

- 网站 banner、hero 图、宣传图
- 游戏素材、角色图、道具图
- 产品图、场景图、概念图
- 对现有图片做编辑、替换背景、去元素、改光照

不适合它的场景：

- 仓库里已有 SVG 图标，只想小改
- 适合直接用 HTML/CSS/SVG 画出来的简单图形
- 明确要求代码原生输出而不是图片文件

### 4.2 默认使用方式

这个 Skill 默认走内置图片工具路径，不需要你手动指定 CLI。

只有在你**明确要求走 CLI fallback** 时，才会使用 `scripts/image_gen.py`。  
如果你没有特别说明，就按默认内置工具执行。

### 4.3 你应该怎么提需求

最好一次说清：

- 这是“新生成”还是“编辑已有图片”
- 图片用途是什么
- 风格、构图、尺寸倾向
- 必须保留或必须避免的内容

### 4.4 示例

示例 1：生成网站头图

```text
使用 Image Gen 为电商首页生成一张 hero 图。
主题是手工陶瓷咖啡杯，风格干净、偏产品摄影，画面右侧留白方便放标题。
不要 logo，不要文字，不要水印。最终图片需要可放进当前项目。
```

示例 2：编辑已有图片

```text
使用 Image Gen 编辑这张产品图，只替换背景为暖色夕阳渐变。
保持产品主体、边缘和颜色不变，不要加文字，不要改构图。
```

### 4.5 使用时要记住的点

- 如果是项目要落地使用的图片，最终应保存到项目目录，而不是只停留在默认生成目录
- 如果是编辑任务，约束要写得很明确，例如“只改背景，其他不变”
- 如果你只想先看效果，可以明确说“先出预览，不落库”

## 5. OpenAI Docs：查官方文档、做升级和选型

### 5.1 它适合做什么

`OpenAI Docs` 适合解决这几类问题：

- 想查 OpenAI 官方文档里的某个能力怎么用
- 想确认最新推荐模型
- 想做 GPT-5.4 升级评估
- 想把旧 prompt 或旧调用方式升级到新推荐写法

### 5.2 默认工作方式

这个 Skill 的核心原则是：

- 先查 OpenAI 官方开发者文档
- 优先使用 OpenAI 开发者文档的 MCP 工具
- 只有在 MCP 没法提供有效结果时，才退到官方域名的网页检索

也就是说，它不是“凭记忆回答 OpenAI 问题”，而是“先查官方，再回答”。

### 5.3 你应该怎么提需求

这类请求最好说清：

- 你问的是哪个产品：Responses API、Realtime、Apps SDK、Codex、Agents SDK 等
- 你是要查“用法”，还是要“选型/升级建议”
- 你是否需要引用、链接依据或兼容性判断

### 5.4 示例

示例 1：查接口用法

```text
使用 OpenAI Docs 查询 Responses API 里多轮对话和工具调用的推荐写法。
我需要基于官方文档的简明说明，并给出关键接口字段。
```

示例 2：做模型选型

```text
使用 OpenAI Docs 帮我判断：做代码生成和工具调用的 agent 系统，
当前更适合选什么模型。请基于官方文档给出建议，并说明理由。
```

示例 3：升级评估

```text
使用 OpenAI Docs 审查我们现在的 GPT 调用是否适合升级到 GPT-5.4。
请按“目标模型、推理强度建议、兼容性风险、prompt 是否要改”这几个维度输出。
```

### 5.5 使用时要记住的点

- 这个 Skill 适合“要权威依据”的问题
- 如果你要的是最新官方建议，最好明确写“基于官方文档”
- 如果你在做升级，最好直接说明现有模型、现有 prompt 和调用场景

## 6. Plugin Creator：创建 Codex 插件脚手架

### 6.1 它适合做什么

`Plugin Creator` 用来生成 Codex 插件目录和基础清单，典型场景包括：

- 新建一个本地插件
- 给插件补 `skills/`、`hooks/`、`scripts/` 等目录
- 生成或更新 `.agents/plugins/marketplace.json`

### 6.2 默认工作方式

它的核心是使用脚手架脚本创建规范结构。  
创建后，`plugin.json` 里通常会保留占位符，后续再按插件实际信息补齐。

### 6.3 你应该怎么提需求

最好明确这几个点：

- 插件是 `repo-local` 还是 `home-local`
- 插件名是什么
- 是否要带 marketplace 入口
- 是否要附带 `skills/`、`hooks/`、`scripts/`、`assets/`、`.mcp.json`、`.app.json`

### 6.4 示例

示例 1：最小插件

```text
使用 Plugin Creator，在当前仓库创建一个 repo-local 插件 `api-tester`。
只要最小结构，不需要 marketplace。
```

示例 2：完整插件脚手架

```text
使用 Plugin Creator，在当前仓库创建插件 `mysql-inspector`。
需要 marketplace 入口，并额外生成 `skills/`、`hooks/`、`scripts/`、`assets/`。
插件先保留默认占位符，不要替我编造元信息。
```

### 6.5 使用时要记住的点

- 插件名会被规范化为小写连字符形式
- marketplace 里的插件顺序会影响 UI 展示顺序
- 如果你没有明确说覆盖已有条目，通常不应该强行替换

## 7. Skill Creator：创建或更新一个 Skill

### 7.1 它适合做什么

`Skill Creator` 适合：

- 新建一个可复用 Skill
- 重构已有 Skill 的结构
- 补充 `agents/openai.yaml`
- 把大段说明拆到 `references/`
- 判断哪些内容该放 `SKILL.md`，哪些该放脚本或参考资料

### 7.2 它强调什么

这个 Skill 的核心思想不是“把说明写得越长越好”，而是：

- `SKILL.md` 要短而硬
- 只保留关键工作流、触发条件、约束和导航
- 大块细节拆到 `references/`
- 需要确定性执行时再放 `scripts/`

也就是说，它更像“Skill 架构设计器”，而不是“帮你堆很多文档”。

### 7.3 你应该怎么提需求

建议说清：

- Skill 解决什么任务
- 触发词或触发场景是什么
- 希望保留高自由度，还是希望强约束
- 需不需要附带脚本、references、assets

### 7.4 示例

示例 1：新建 Skill

```text
使用 Skill Creator，帮我设计一个 `sql-reviewer` skill。
它负责审查 SQL 查询的性能风险和索引问题。
先给出推荐目录结构、SKILL.md 应包含的核心部分，以及哪些内容应该拆到 references。
```

示例 2：重构已有 Skill

```text
使用 Skill Creator，检查当前 skill 是否过于臃肿。
重点看 SKILL.md 是否太长、哪些内容应该移到 references、是否需要增加 scripts 或 agents/openai.yaml。
```

### 7.5 使用时要记住的点

- 不要把 Skill 做成“给人看的说明书合集”
- 如果某段内容只有在少数场景才需要，优先拆出去
- 如果一个动作每次都要重写，而且步骤稳定，优先脚本化

## 8. Skill Installer：安装已有 Skill

### 8.1 它适合做什么

`Skill Installer` 主要负责：

- 列出可安装的 curated skills
- 安装某个现成 Skill
- 从 GitHub 仓库路径安装 Skill

### 8.2 默认工作方式

它依赖辅助脚本进行列表查询和安装。  
由于这类脚本通常涉及网络，请求执行时通常需要相应权限。

另外有一个重要边界：

- `.system` 下的系统 Skill 一般已经预装
- 通常不需要再次安装

### 8.3 你应该怎么提需求

你可以分 3 类提：

- 先列清单
- 直接安装 curated 里的某个 Skill
- 按 GitHub 仓库地址或路径安装

### 8.4 示例

示例 1：先列可安装清单

```text
$skill-installer 列出当前 curated skills，并标明哪些已经安装。
```

示例 2：安装指定 Skill

```text
使用 Skill Installer 安装 `knowledge-base-planner`。
安装完成后告诉我是否需要重启 Codex。
```

示例 3：从 GitHub 路径安装

```text
使用 Skill Installer，从某个 GitHub 仓库里的指定 skill 路径安装到本地。
如果下载失败，再尝试备用方式。
```

### 8.5 使用时要记住的点

- 安装完成后，通常需要重启 Codex 才能识别新 Skill
- 如果目标 Skill 已经存在，安装流程通常会中止，而不是默认覆盖
- 如果你要装的是系统内置 Skill，先确认是否其实已经预装

---

## 9. 五个 Skill 的快速选型

如果你分不清该用哪个，可以按下面判断：

- 要生成或编辑图片：`Image Gen`
- 要查 OpenAI 官方最新文档、做模型选型或升级：`OpenAI Docs`
- 要创建 Codex 插件目录和 marketplace 结构：`Plugin Creator`
- 要设计或重构一个 Skill：`Skill Creator`
- 要把已有 Skill 安装到本地环境：`Skill Installer`

---

## 10. 推荐提问模板

你后续可以直接复用下面这个通用模板：

```text
使用 <Skill 名称>，帮我处理 <任务目标>。
范围是 <只规划 / 直接执行 / 新建文件 / 更新现有文件>。
约束是 <不能改什么 / 必须保留什么 / 输出格式要求>。
相关上下文：<目录 / 文件 / 目标平台 / 示例>。
```

比如：

```text
使用 Plugin Creator，帮我处理新建 Codex 插件脚手架。
范围是直接执行并创建文件。
约束是保留 manifest 占位符，不要覆盖已有插件。
相关上下文：当前仓库需要一个 repo-local 的 `mysql-inspector` 插件，并带 marketplace 入口。
```

---

## 11. 一句话总结

这 5 个系统 Skill 分别解决的是：

- `Image Gen`：图片生成与编辑
- `OpenAI Docs`：官方文档查询与升级选型
- `Plugin Creator`：插件脚手架生成
- `Skill Creator`：Skill 设计与重构
- `Skill Installer`：Skill 列表查询与安装

你真正要学会的，不是死记它们的名字，而是：

**在请求里把目标、范围、约束和上下文说清楚。**  
这样 Skill 才能稳定工作，而不是退化成模糊对话。
