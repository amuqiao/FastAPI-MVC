# Codex CLI 开发进阶指导

> **文档职责**：只解决 Codex CLI 的长期协作能力建设，包括 `AGENTS.md`、审批与规则、复杂任务拆解、`codex exec`、团队协作。  
> **适用场景**：你已经能用 Codex 做基础开发，但开始追求更稳定、更可复用的工作流。  
> **不包含内容**：不重复安装配置，不重复基础 prompt 模板。  
> **阅读顺序**：先读 `3.Codex_CLI_代码开发快速上手.md`，再读本文。

## 1. 进阶到底在进阶什么

进阶不是“记更多命令”，而是让 Codex 从一次性助手，变成可长期协作的工程代理。

你要重点控制的，其实只有 5 个边界：

1. `工作区边界`：它能读哪、写哪
2. `权限边界`：它什么时候要问你
3. `上下文边界`：它这次到底知道哪些规则和文件
4. `任务边界`：这次它只允许完成什么
5. `验证边界`：改完后必须用什么方式证明没乱改

这 5 个边界一旦清楚，Codex 的稳定性会明显上升。

## 2. 工作区边界先立住

### 2.1 从最小目录开始

如果任务只在一个子模块，就直接：

```bash
codex --cd services/payments
```

不要默认从仓库根开始，把整棵树都丢给它。

### 2.2 跨目录时再加额外范围

```bash
codex --cd apps/frontend --add-dir ../shared --add-dir ../backend
```

原则很简单：

- 能只开一个目录，就别开两个
- 能加精确子目录，就别直接放大到整个仓库

### 2.3 把权限模式理解清楚

日常最常见的 3 种模式可以这样理解：

- `read-only`：看代码、排障、审查
- `workspace-write`：大多数本地开发任务
- `danger-full-access`：明确接受完全放权时才使用

如果你只是日常人工协作，本地默认优先选择工作区可写而非完全放权的模式。

### 2.4 审批策略不要一上来开到最大

常见审批策略可以理解成：

- `untrusted`：更保守
- `on-request`：默认平衡型
- `never`：不拦截，适合自动化

日常本地开发，优先用“工作区可写 + 需要越界时再问”的思路。

## 3. 把长期规则收进 `AGENTS.md`

### 3.1 为什么它比长 prompt 更重要

如果你每次都要重复说明这些信息：

- 测试命令
- lint 命令
- 模块职责
- 禁改区域
- 提交前检查

说明这些内容就不该继续留在 prompt 里，而应该进入 `AGENTS.md`。

### 3.2 它适合承载什么

一个实用的 `AGENTS.md` 至少应该包含：

- 仓库结构和关键目录
- 启动、测试、lint、format 命令
- 代码规范
- 禁止事项
- 完成标准
- 修改后需要同步更新的文档、测试或配置

### 3.3 一个推荐模板

```md
# AGENTS.md

## Repo Layout
- `app/`: 应用主代码
- `tests/`: 测试
- `scripts/`: 开发与运维脚本

## Commands
- Start: `uv run python main.py`
- Test: `pytest`
- Lint: `ruff check .`
- Format: `ruff format .`

## Working Rules
- 优先在现有模块内扩展，不要无必要新建抽象层
- 不要修改公共接口返回结构，除非任务明确要求
- 新增后端接口时，补最小必要测试

## Done When
- 相关测试通过
- 必要检查通过
- 变更说明清楚
- 没有越界修改无关模块
```

### 3.4 什么时候用 `/init`

如果仓库里还没有 `AGENTS.md`，先运行：

```text
/init
```

它适合生成脚手架，但不要直接提交。你仍然需要按仓库实际规范精简和补充。

### 3.5 子目录为什么也值得放说明

如果仓库很大，根目录 `AGENTS.md` 只解决全局规则，不够。

这时可以在更靠近实际工作目录的位置放局部说明，让某个子模块拥有自己的特殊规则，例如：

- 专用测试命令
- 该目录禁改项
- 修改这个模块必须同步更新的文件

## 4. 复杂任务要拆成 Codex 擅长的形状

### 4.1 不要给模糊大任务

反例：

```text
把支付系统彻底优化一下
```

这类任务的问题通常不是 Codex 不够聪明，而是验收条件根本不清楚。

### 4.2 推荐拆分法

把大任务拆成 4 层：

1. `调研层`：现状、链路、风险
2. `方案层`：最小改法、兼容性、回滚点
3. `实施层`：逐模块改动
4. `验证层`：测试、检查、审查

### 4.3 一个可执行示例

如果你要做“订单模块幂等性改造”，建议拆成：

1. 线程 A：分析订单创建链路和重复写入风险
2. 线程 B：设计最小幂等方案
3. 线程 C：实现 DB/缓存/锁相关改动
4. 线程 D：补测试并回归验证

### 4.4 什么时候用 `/fork`

如果你已经有一条可工作的线程，但想试另一种方案，不要覆盖原会话：

```text
/fork
```

适合：

- 对比两种实现路线
- 保留原始分析链路
- 快速试错而不污染主线程

---

## 5. 会话管理：让上下文别失控

### 5.1 一个线程只做一个任务

不要把下面这些混在一个线程里：

- 新功能开发
- Bug 修复
- 文档整理
- 部署排障

### 5.2 长线程及时压缩

```text
/compact
```

适合：

- 已经多轮讨论
- 准备从分析阶段进入实施阶段
- 你感觉 Codex 开始重复或遗忘重点

### 5.3 旧任务继续用 `/resume`

```text
/resume
```

适合隔天继续或中途切换后回来，不必重新解释全部背景。

### 5.4 新任务重新开局用 `/new`

```text
/new
```

新任务开新线程，通常比在旧线程里硬续更稳定。

## 6. 审批、规则与自动化

### 6.1 审批太频繁时，先别急着给满权限

先按这个顺序排查：

1. 当前目录是不是开大了
2. 当前模式是不是过于保守
3. 是不是某个高频命令适合规则化放行

### 6.2 适合逐步规则化的命令

这类命令通常适合沉淀规则：

- `pytest`
- `ruff check`
- `npm test`
- `go test`
- `gh pr view`

共同特点是：

- 高频
- 风险相对可控
- 命令结构稳定

### 6.3 不适合随便放行的命令

这类命令需要非常谨慎：

- 破坏性删除
- 广泛写入系统目录
- 凭据管理类操作
- 可能执行任意脚本的宽泛壳命令

### 6.4 规则文件的正确理解方式

规则控制的是“某个命令前缀应该被允许、提示还是禁止”，不是自然语言级别的模糊授权。

所以你应该：

- 从少量高频命令开始
- 先验证匹配范围
- 再逐步扩大

### 6.5 规则效果校验

```bash
codex execpolicy check --pretty --rules ~/.codex/rules/default.rules -- gh pr view 7888
```

在你准备沉淀团队规则之前，先用这个命令验证是否匹配到正确前缀。

### 6.6 用 `codex exec` 接入非交互任务

如果你希望接脚本或 CI，可以用：

```bash
codex exec --cd services/payments "审查当前目录的变更，输出风险和测试建议"
```

如果需要结构化或脚本友好的输出，再组合：

```bash
codex exec --json --output-last-message result.txt "总结当前工作区的风险"
```

如果要继续上一次非交互会话：

```bash
codex exec resume --last
```

## 7. 验证闭环：真正把质量拉起来

很多人能让 Codex 改代码，但做不到长期稳定，问题大多出在这里。

### 7.1 每次都要求 4 类验证

- 测试
- lint / format / type-check
- 行为确认
- 审查改动

### 7.2 把验证写成默认要求

你可以长期固定这样要求 Codex：

```text
改完后请：
1. 运行相关测试
2. 运行必要检查
3. 总结验证结果
4. 说明剩余风险
```

### 7.3 把 `/review` 纳入标准动作

```text
/review
```

适合在以下时机执行：

- 一轮改动完成后
- 提交前
- 你怀疑“修了 A，顺手破坏了 B”时

## 8. 团队协作时怎么落地

### 8.1 团队最先统一的不是提示词

优先统一这些东西：

- `AGENTS.md`
- 测试命令
- review 要求
- 禁改区域

这比统一一份超长提示词更有效。

### 8.2 适合固化在仓库里的内容

- `AGENTS.md`
- `code_review.md`
- 标准开发脚本
- 子模块本地说明

### 8.3 适合留在每次 prompt 里的内容

- 本次目标
- 本次边界
- 本次验收标准
- 本次特殊风险

长期规则进文件，任务差异留在 prompt。

## 9. 最后一条建议

如果你发现自己每次都在重复解释仓库规则、测试命令和禁改边界，说明你该补的是：

1. `AGENTS.md`
2. 会话拆分习惯
3. 审批与规则策略
4. 验证闭环

这 4 件事做好后，Codex 才会真正从“能帮你写代码”进化成“能稳定承担一部分开发工作”。

不要把“文档整理”“接口开发”“Bug 修复”“代码审查”混在一个线程。

### 8.2 长线程及时压缩

```text
/compact
```

适合：

- 来回多轮讨论后
- 已经完成阶段一，准备进入阶段二
- 你感觉 Codex 开始重复或遗忘重点

### 8.3 旧任务恢复

```text
/resume
```

适合：

- 隔天继续
- 中途切换分支后回来
- 想沿用旧上下文而不是重新解释

### 8.4 新任务重新开局

```text
/new
```

这比继续污染旧线程稳定得多。

## 9. 验证闭环：真正把质量拉起来

很多人到了“能改代码”就停了，真正的进阶在这里。

### 9.1 每次都要求 4 类验证

- 测试
- lint/格式化/类型检查
- 行为确认
- diff 审查

### 9.2 建议固定成默认口径

你可以长期对 Codex 使用这句：

```text
改完后请：
1. 运行相关测试
2. 运行必要检查
3. 自查 diff
4. 汇总改动、验证结果和剩余风险
```

### 9.3 把 review 也纳入标准动作

```text
/review
```

适合在以下时机执行：

- 一轮改动完成后
- 提交前
- 你怀疑“它修了 A，顺手破坏了 B”时

### 9.4 `/diff` 和 `/review` 的职责不同

- `/diff`：你看改了什么
- `/review`：让 Codex 帮你找问题

这两个动作不要互相替代。

## 10. 团队协作场景怎么用

### 10.1 团队最先统一的不是提示词，而是仓库说明

先统一：

- `AGENTS.md`
- 测试命令
- review 要求
- 禁改区域

比统一一份几百行提示词更有效。

### 10.2 哪些内容适合固化进仓库

- `AGENTS.md`
- `code_review.md`
- 常用任务脚本
- 子模块本地说明

### 10.3 哪些内容适合保留在当前 prompt

- 本次需求目标
- 本次边界
- 本次验收标准
- 本次特殊风险

长期规则进文件，任务差异留在 prompt。

## 11. 三类常见进阶误区

### 11.1 误区一：想靠一次巨型提示词解决全部问题

结果通常是：

- 上下文冗长
- 重点不突出
- 任务切换成本高

正确做法：仓库规则进 `AGENTS.md`，本次任务用短 prompt 补充。

### 11.2 误区二：为了省审批，直接给完全放权

这是最快失控的方式之一。

正确做法：优先用 `workspace-write`、`on-request`、精确目录范围和前缀规则。

### 11.3 误区三：把 Codex 当成同步逐行结对工具

Codex 更适合：

- 先吃上下文
- 自己执行一段完整子任务
- 再回来交付结果

你越把它当工程代理，收益越高。

---

## 12. 一套推荐的进阶工作流

下面是一套适合长期项目的实战工作流：

1. 在仓库根补齐 `AGENTS.md`
2. 子模块有特殊规则时，再加局部 `AGENTS.md` 或覆盖文件
3. 每个开发任务单独开线程
4. 复杂任务先 `/plan`
5. 需要文件聚焦时用 `/mention`
6. 改动完成后先 `/diff`
7. 再 `/review`
8. 确认测试和检查结果
9. 需要保留分支方案时用 `/fork`
10. 高频安全命令逐步沉淀到规则文件

---

## 13. 结论

Codex CLI 的进阶能力，本质上不在“更会写 prompt”，而在于你是否把它纳入了工程化协作体系：

**目录边界清晰、规则长期固化、任务拆分合理、验证闭环完整、自动化逐步放开。**

当这几件事做起来后，Codex 才会从“偶尔帮你写代码”变成“稳定承担一部分开发工作”。
