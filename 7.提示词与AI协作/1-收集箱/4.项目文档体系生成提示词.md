> 文档职责：沉淀一份可复用的“项目文档体系生成提示词”，用于指导 AI 为任意项目产出结构完整、职责清晰、可直接落地使用的文档。
> 适用场景：需要分析一个新项目，并一次性生成安装与使用教程、FAQ、5 分钟上手、源码分析等核心文档。
> 阅读目标：直接复制提示词给 AI，稳定生成低学习成本、分工明确、可实操的项目文档体系。
> 目标读者：需要给项目建立标准文档体系的开发者、维护者、知识库整理者。

# 项目文档体系生成提示词

## 先说结论

一个项目的最小核心文档集合，至少应包含 4 篇：

1. 安装与使用教程
2. 常见问题 FAQ
3. 5 分钟上手
4. 源码分析

这 4 篇文档必须同时满足两件事：

1. **分工明确**
2. **用户照着就能用**

也就是说，文档不能只是“介绍项目”，而要形成一套：

- 有主线
- 有操作
- 有边界
- 有排障
- 有源码依据

的完整体系。

## 可直接使用的主提示词

```markdown
请基于当前项目源码与实际运行行为，生成一套完整的项目文档体系。不要只写概念说明，要按“用户能直接配置、启动、验证、关闭”的标准输出。

你必须至少产出以下 4 篇文档：

1. 安装与使用教程
2. 常见问题 FAQ
3. 5 分钟上手
4. 源码分析

对整套文档，必须遵守以下总原则：

1. 先结论后展开，先全貌后细节。
2. 按场景组织内容，而不是只按抽象概念组织。
3. 关键前提、默认行为、边界条件，必须直接写进主流程，不允许依赖 FAQ 补洞。
4. 文档必须可实操。用户应能照着步骤完成配置、启动、验证、关闭。
5. 不同文档必须职责清晰，避免内容重复和边界混乱。
6. 不允许“补丁式文档”风格。要一次性把主线讲清楚。
7. 对源码相关结论，必须以实际代码行为为依据，不要凭空猜测。

每篇文档开头都必须使用引用格式，明确写出以下 4 项：

- 文档职责
- 适用场景
- 阅读目标
- 目标读者

每篇文档都必须满足以下写作约束：

1. 一节只讲一个问题。
2. 语言简洁，避免重复、堆砌和空话。
3. 对操作步骤，必须写清：
   - 为什么做这一步
   - 做完会发生什么
   - 如何验证
   - 如何关闭 / 回退
4. 对“默认行为”和“需要显式配置”的边界，必须写清楚。
5. 对“适合什么场景”和“不适合什么场景”，必须写清楚。
6. FAQ 只负责典型问题、误区、排障，不承担主流程缺失。
7. 源码分析只负责解释真实行为和读写路径，不承担新手上手教学。

请按以下结构生成文档：

### 1. 安装与使用教程

职责：
- 这是完整主线文档。
- 要覆盖项目定位、安装、配置、启动、验证、关闭、典型使用场景。

必须包含：
- 项目是什么，它解决什么问题
- 默认运行方式是什么
- 至少 3 个主要使用场景
- 每个场景都必须写清：
  - 适用目的
  - 是否需要额外配置
  - 配置什么
  - 如何启动
  - 启动后会发生什么
  - 如何验证
  - 如何关闭
- 服务管理命令
- 如何判断当前属于哪种运行状态
- 阅读路径：什么时候该看 FAQ，什么时候该看源码分析

### 2. 5 分钟上手

职责：
- 这是最短实操路径。
- 用户应能少看解释、先跑起来。

必须包含：
- 最关键的 3 到 5 条判断
- 最常用的 2 到 4 个场景
- 每个场景只保留最短命令、最小解释和验证方式
- 明确说明哪些是“试接入”，哪些是“完整使用”
- 明确说明默认推荐用法

### 3. 常见问题 FAQ

职责：
- 这是理解边界和处理疑惑的文档。
- 不负责主流程教学。

必须包含：
- 项目定位相关误区
- 默认行为相关误区
- hook / loop / memory / 配置边界相关问题
- 验证和排障相关问题
- 多项目使用相关问题
- “什么时候推荐用哪种模式”这类判断题

FAQ 中每个问题的回答都必须尽量直接，避免过度抽象。能用“先说结论”的方式就不要先绕概念。

### 4. 源码分析

职责：
- 这是行为真相和代码依据文档。
- 不负责新手操作教学。

必须包含：
- 关键入口文件
- 核心路径与状态文件
- 不同模式 / 不同场景下的读写行为
- 必要时用 Mermaid 画行为流转图
- 解释“为什么会这样”，而不只是“怎么做”

源码分析必须特别关注：
- 配置如何影响运行结果
- 哪些行为是默认行为
- 哪些行为依赖显式环境变量
- 不同组件之间是如何关联的

最后，请确保这 4 篇文档之间的职责分工如下：

- 安装与使用教程：完整主线
- 5 分钟上手：最短路径
- FAQ：边界、误区、排障
- 源码分析：真实行为与依据

不要让 FAQ 承担主流程缺失，不要让源码分析混成操作手册，不要让 5 分钟上手变成长篇教程。
```

## 单篇文档补充提示词

如果你不是一次性产出 4 篇，而是逐篇生成，可以追加下面的单篇约束。

### 安装与使用教程补充提示词

```markdown
请把这篇文档写成“完整场景手册”，而不是项目介绍。

要求：

1. 必须覆盖至少 3 个核心使用场景。
2. 每个场景都要写清：
   - 适用目的
   - 是否需要配置
   - 如何配置
   - 如何启动
   - 启动后会发生什么
   - 如何验证
   - 如何关闭
3. 对默认行为和需要显式配置的边界，必须写进正文主线。
4. 不允许把关键前提藏到 FAQ。
```

### 5 分钟上手补充提示词

```markdown
请把这篇文档写成“傻瓜式最短操作清单”。

要求：

1. 只保留最常用场景。
2. 解释要少，但关键判断不能漏。
3. 明确区分：
   - 试接入
   - 完整使用
4. 每个场景必须至少包含：
   - 目的
   - 命令
   - 验证方式
   - 关闭方式（如果需要）
```

### FAQ 补充提示词

```markdown
请把这篇文档写成“边界澄清 + 典型排障”。

要求：

1. 不要重复主教程步骤。
2. 每个问题先给结论，再展开。
3. 回答尽量直接，避免过度抽象。
4. 优先覆盖：
   - 用户最容易误解的地方
   - 配置是否必须
   - 默认行为是什么
   - 出现异常时如何判断
```

### 源码分析补充提示词

```markdown
请把这篇文档写成“行为依据文档”，不是操作手册。

要求：

1. 所有关键结论都要能在源码中找到依据。
2. 重点分析：
   - 关键入口
   - 核心状态文件
   - 读写路径
   - 配置如何改变行为
3. 对不同场景，必须讲清：
   - 会读哪些文件
   - 会写哪些文件
   - 为什么会这样
4. 必要时使用 Mermaid 图，但图前说明作用，图后简要解读。
```

## 文档分工检查清单

在文档产出完成后，用下面这份清单检查是否合格。

```markdown
- 是否至少包含：安装与使用教程 / FAQ / 5 分钟上手 / 源码分析
- 每篇文档开头是否都写了：文档职责 / 适用场景 / 阅读目标 / 目标读者
- 安装与使用教程是否覆盖完整主线，而不是只写概念
- 5 分钟上手是否足够短，但没有漏掉关键判断
- FAQ 是否只负责边界和排障，而不是补主流程缺失
- 源码分析是否只负责行为依据，而不是混入上手步骤
- 是否按场景组织，而不是只按概念组织
- 默认行为和显式配置边界是否写清
- 用户是否可以按步骤完成配置、启动、验证、关闭
- 不同文档之间是否有明显重复和职责混乱
```

## 最后一句话

好的项目文档体系，不是“把信息写全”，而是“把用户真正需要的主线、边界、操作和依据分开放对位置”。  
这份提示词的目标，就是让 AI 一次性把这套体系搭对，而不是靠后续反复补洞。
