# 项目分析二轮提示词

> 用途：第一轮项目分析已经完成后，不再重复全量分析，而是让 AI 基于已有认知与代码证据，给出下一轮最值得深挖的 4 个方向。
> 适用范围：通用 GitHub 项目。不要默认仓库是前端应用、服务端项目或 AI 应用。
> 使用方式：把整段提示词直接发给 AI；如果你已有特定上下文，也可在正文前补充“上一轮结论摘要”。

---

## 这版模板解决什么问题

这版不是让 AI “再给 4 个想法”，而是解决下面几个常见偏差：

1. 重复第一轮已经讲过的内容
2. 给出 4 个表面不同、实则重复的问题
3. 给出的方向只适合某一种项目形态
4. 一上来就强行补图，却没有先做 Mermaid 路由
5. 方向不绑定代码证据，导致建议泛泛而谈

---

## 主模板（复制此段使用）

```md
基于你对该项目上一轮分析的已有认知，不要重复做一遍完整项目分析。

这一次的目标只有一个：

请给我 **4 个真正值得继续深挖的方向**，帮助我决定下一轮该问什么。

---

## 执行要求

### 1. 以“上一轮结论 + 当前代码证据”作为基础

- 不要把第一轮已明确讲过的内容重写一遍
- 也不要只根据一般经验给泛化建议
- 这 4 个方向必须贴近该项目真实的架构重点、复杂边界或演进风险
- 如果你发现我当前认知里有明显偏差，可以先用 1～2 句话说明“最值得修正的一点”

### 2. 不要默认项目类型

请先判断该仓库更接近哪一类，再提出方向：

- 前端应用
- 后端服务
- CLI 工具
- SDK / Library
- AI / Agent 项目
- 基础设施 / 平台工程
- 多语言 monorepo
- 其他

后续给出的 4 个方向，必须服从这个判断，不要硬套固定分析框架。

### 3. 方向必须有区分度

4 个方向应尽量分散在不同分析维度中挑选。优先从下面几类里选与该项目最相关的 4 类，而不是机械全选：

- 主流程 / 主调度链 / 主执行路径
- 数据模型 / 核心抽象 / 协议边界
- 模块边界 / 子系统依赖 / 高耦合区域
- 状态、持久化、缓存、队列或资源管理
- 扩展机制 / 插件机制 / 注册机制
- 性能 / 成本 / 可靠性 / 并发边界
- 构建 / 部署 / 运行时环境 / 工程边界
- 多端 / 多实例 / 多租户 / 多人协作 / 演进瓶颈

不要给这种低区分度结果：

- “再看看技术栈”
- “再看看目录”
- “先把目录列一遍”但不说明会如何用 tree 目录图或模块骨架图建立结构认知
- “再看看主要模块”
- 同一个问题换 4 种说法

### 4. 每个方向必须回答 4 个问题

对每个方向，都必须给出：

1. **方向标题**
2. **为什么值得深挖**
3. **重点看哪些模块 / 文件 / 服务 / 包 / 子系统**
4. **继续深挖后，真正能回答什么问题**

### 5. 输出要有排序与判断

- 4 个方向必须编号 `1-4`
- 按“深挖价值”排序，而不是随意排列
- 排序依据要隐含体现在说明里：
  - 为什么它比别的方向更关键
  - 它能解释更核心的问题，还是暴露更大的风险

### 6. Mermaid 不是默认输出

如果某个方向只靠文字不够清楚，可以按需补 0～1 张 Mermaid 图。

但必须严格遵守：

- 先读取：
  - `@/Users/admin/Downloads/Code/AI-Engineering-Knowledge-Base/7.提示词与AI协作/Mermaid作图规范参考/mermaid-路引入口.md`
- 先输出 `Mermaid 路由声明`
- 先判断这张图是否真的有必要
- 再决定是否读取 `README.md` 和具体 `A/B/C/D/E` reference
- 不允许为了完整感强行出图

如果最终出图，必须满足：

- 图前写清：
  - 这张图回答什么问题
  - 使用哪份 Mermaid reference
  - 为什么选它
- 图前有 1～3 句前导语
- 图后有 2～4 条要点
- 一张图只回答一个主问题

---

## 输出要求

### 输出结构

请按下面结构输出：

1. `认知校准（可选）`
2. `4 个深挖方向`
3. `如果只能先看 1 个，最建议先看哪一个（可选）`

### 对“4 个深挖方向”的格式要求

每个方向使用下面结构：

#### 方向 N：{标题}

- **为什么值得深挖**：
- **重点看哪里**：
- **深入后能回答什么问题**：

### 语言风格

- 中文输出
- 术语保留英文原名
- 简洁，但判断要明确
- 不要替我展开完整分析
- 你的任务是“给出下一轮提问方向”，不是“提前把下一轮答案讲完”

### 内容边界

- 不要重复第一轮已经明显展开过的章节
- 不要给空泛建议
- 不要把 4 个方向都写成“可以继续看看……”
- 要体现你知道这个项目真正复杂在哪
```

## 精简版

### 极简版

```md
基于你对该项目上一轮分析的已有认知，不要重复做完整项目分析。

请只给我 4 个值得继续深挖的方向，帮助我决定下一轮问什么。

要求：
- 只给 4 个，不多不少
- 不要默认项目类型
- 4 个方向必须有区分度
- 每个方向都写：
  1. 标题
  2. 为什么值得深挖
  3. 重点看哪些模块 / 文件 / 包 / 子系统
  4. 深挖后能真正回答什么问题

如需出图：
- 先读取 Mermaid 路引入口
- 先输出 Mermaid 路由声明
- 再决定是否真的需要图

输出中文，术语保留英文，不要重复第一轮已明显讲过的内容。
```
