# 项目分析 Prompt 模块 · 使用说明

这个模块是“GitHub 项目通用分析 Prompt”的正式沉淀目录。

目标有两个：

1. 给任意类型仓库提供稳定、可复用的项目分析主 Prompt
2. 避免在 `1-收集箱/` 中继续把临时版本误当正式模板维护

---

## 文件说明

- `项目分析主Prompt.md`
  第一轮全量分析模板，适合从零开始分析一个仓库。
- `项目分析二轮-4个深挖方向Prompt.md`
  第二轮模板，适合在已有首轮分析基础上，筛选下一轮最值得追问的 4 个方向。
- `references/项目分析反例清单.md`
  专门约束模型不要误判项目形态、不要硬套分析框架、不要滥用 Mermaid。

---

## 使用顺序

### 场景 1：第一次分析一个项目

先用：

- `项目分析主Prompt.md`

适用：

- 还没有系统认知
- 想先建立项目全景
- 想得到结构、链路、数据模型、设计决策等基础判断

### 场景 2：第一轮做完，想决定下一轮问什么

再用：

- `项目分析二轮-4个深挖方向Prompt.md`

适用：

- 第一轮全量分析已经完成
- 不想再重复总览
- 想先筛出最值得继续追问的方向

---

## 使用边界

这两个 Prompt 都是“通用 GitHub 项目分析模板”，不是某一类项目的专用模板。

不要默认它们只适合：

- 前端应用
- 后端服务
- AI Web 平台

它们应该同样适用于：

- CLI 工具
- SDK / Library
- 编译器 / 解释器
- 基础设施项目
- 多语言 monorepo
- 插件型系统
- 脚本型仓库

如果你发现模型开始默认：

- 一定有 `App`
- 一定有 `context / store / service`
- 一定有典型请求-响应主链
- 一定有数据库 schema

请直接补充引用：

- `references/项目分析反例清单.md`

## 目录结构解读原则

项目分析里的“目录结构解读”默认优先使用：

1. 带职责注释的 tree 目录图
2. 或等价的文字树

原因是 tree 更适合先建立“目录空间感”和“模块落点”。

Mermaid 不应替代 tree 的工作。更合适的分工是：

- tree 目录图：负责回答“代码主要落在哪些目录、哪些是一二级主链模块”
- Mermaid：负责回答“这些模块如何分层、依赖、流转或变化”

因此，推荐顺序是：

1. 先用 tree 看目录骨架
2. 再按需要补 Mermaid 结构图或流程图

---

## Mermaid 使用原则

项目分析过程中如果需要 Mermaid：

1. 先读取 Mermaid 路引入口
2. 先输出 `Mermaid 路由声明`
3. 再决定是否需要图
4. 再决定是否读取 `README.md` 和具体 reference

不要把 Mermaid 当成默认装饰步骤。

---

## 和收集箱的关系

- `1-收集箱/` 用来放临时 Prompt
- `2-项目分析/` 用来放已经稳定、可复用、适合作为正式模板的 Prompt

如果后续继续优化这两个 Prompt，优先改这里，不要回到收集箱里继续漂移。
