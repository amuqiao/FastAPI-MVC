# 项目分析提示词模板

> 用途：对任意 GitHub 项目做“从宏观到细节”的结构化分析。
> 适用范围：前端应用、后端服务、CLI 工具、SDK / Library、AI 项目、基础设施项目、脚本型仓库、多语言 monorepo。
> 使用方式：替换 `{花括号}` 占位符后直接发送给 AI。
> 如果你已经做完第一轮分析，只想让 AI 给出下一轮最值得深挖的 4 个方向，请改用：
> `@/Users/admin/Downloads/Code/AI-Engineering-Knowledge-Base/7.提示词与AI协作/2-项目分析/项目分析二轮-4个深挖方向Prompt.md`

## 读者人设

本次分析默认面向一位 **Python 后端工程师** 输出：

- 熟悉 **FastAPI**、MVC / 分层架构，以及 `Router / Service / Repository / Model`
- 熟悉 CRUD、参数校验、业务处理、数据访问、典型请求响应链路
- 最关心 3 件事：**架构模式是什么、它和后端分层怎么对应、一个典型场景如何贯穿全系统**

因此输出要避免泛泛目录介绍，优先服务于“建立架构认知并具备二次开发能力”。

---

## 设计目标

这份模板重点解决四类常见问题：

1. 只看 README 和目录名，就开始凭经验猜架构
2. 默认把仓库当成“前端应用”或“服务端项目”，带入错误分析框架
3. Mermaid 没先做图型路由，直接硬画
4. 只描述“是什么”，不解释“为什么这样设计”

## 执行要求

请基于本地代码仓库做真实分析，不要只根据仓库名、README 标题或通用经验猜测。

### 1. 先识别项目类型，再进入分析

请先判断该仓库更接近哪一类：

- 前端应用
- 后端服务
- CLI 工具
- SDK / Library
- AI / Agent 项目
- 基础设施 / 平台工程
- 多语言 monorepo
- 其他

注意：

- 不要默认它是 Web 应用
- 不要默认它有 `App / context / store / service`
- 不要默认它一定有“请求-响应主链”
- 分析框架要服从项目类型，而不是反过来

### 2. 先做代码证据收集

优先阅读最能代表“项目骨架”的文件，再向细节展开。根据仓库类型灵活选择，不要硬套固定入口。

至少覆盖下面几类中的相关项：

- 入口与启动点：
  - 可执行入口
  - 主包 / 主模块
  - 服务启动文件
  - CLI 入口
  - 插件注册入口
- 构建与运行配置：
  - 构建脚本
  - 依赖清单
  - 环境配置
  - 容器 / 部署配置
- 核心抽象：
  - 关键类型 / 接口 / trait / protocol / schema
  - 核心类 / 核心模块 / 核心包
- 核心执行路径：
  - 主业务流程
  - 主调度链
  - 数据处理链
  - 推理 / 编译 / 渲染 / 导出 / 同步链
- 状态与数据：
  - 持久化层
  - 缓存
  - 队列
  - 本地状态
  - 远端依赖

结论必须尽量绑定到具体文件、函数、类型、模块或目录。

先读取：

### 4. 输出内容要强调“为什么”

  - 生成代码
  - 工程配置

- 视复杂度决定是否补图：
  - tree 目录图优先负责“空间结构和模块落点”
  - 如果还需要表达“模块如何分层”或“能力如何分区”，再走 Mermaid 路由后补图

### 4. 技术栈全景（必须）

- 列表形式输出：
  - 语言
  - 框架 / 运行时

### 7. 关键设计决策（必须）

如果不为空，请分析：

## 使用建议

### 推荐使用顺序

如果你是第一次分析一个陌生项目，建议按下面顺序使用：

1. 先用 **档位 S：最小可用版**
   目的：快速验证这个项目是否能被正确判定架构模式，并找到一个合格的主线场景
2. 如果输出已经足够清晰，就不必立刻升级到长 Prompt
3. 如果输出存在明显缺口，再升级到：
   - **主模板**：适合正式沉淀完整分析文档
   - **档位 B**：适合中等深度的架构深潜
4. 如果你只关心某个模块，再使用 **档位 C**

### 什么时候优先用短 Prompt

优先使用 `档位 S` 的场景：

- 你不确定项目到底是前端应用、后端服务还是混合架构
- 你担心长 Prompt 让模型开始机械对齐格式
- 你想先验证“架构模式判断”和“典型场景选择”是否靠谱

### 什么时候升级到长 Prompt

优先使用主模板或 `档位 B` 的场景：

- 你已经确认项目值得深挖
- 你需要沉淀成正式文档
- 你希望模型把模块依赖、抽象边界、FAQ 等内容补齐

## 对照实验方法

如果你想比较“短 Prompt”和“长 Prompt”哪个更稳，建议固定同一个项目，按下面方法做对照。

### 实验步骤

1. 先对同一个项目运行 **档位 S**
2. 再对同一个项目运行 **主模板** 或 **档位 B**
3. 不要更换项目，也不要更换你的任务目标
4. 把两份输出按同一组维度比较，而不是只看字数和图多不多

### 重点比较 4 个维度

#### 1. 架构模式判断是否准确

看下面几点：

- 是否先给出了明确的架构模式结论
- 是否能解释“为什么是这种模式，而不是另一种”
- 是否真的映射到了 `Router / Service / Repository / Model / Database`
- 是否避免了把项目硬套成 Web 后端

#### 2. 主线场景是否选得好

看下面几点：

- 是否真的选了“最能代表系统”的场景
- 这个场景能否覆盖主要模块
- 后文解释是否持续回扣这个场景
- 是否只是随便挑了一条链路凑流程图

#### 3. 图文是否一致

看下面几点：

- 3 张图是否都围绕同一个主问题
- 图中的层次和正文中的职责划分是否一致
- 图是不是在帮正文澄清结构，而不是做装饰

#### 4. 输出是否过度机械

看下面几点：

- 是否开始逐项打卡式回答小标题
- 是否出现大量正确但低价值的格式化复述
- 是否越长反而越弱化“架构模式”和“典型场景”这两个重点

### 如何判断哪一版更适合你

可以直接用下面这个判断标准：

- 如果 `档位 S` 已经能稳定给出：
  - 正确的架构模式判断
  - 合格的典型场景
  - 3 张真正有用的图
  那么优先继续用短版

- 如果 `档位 S` 常见问题是：
  - 模块解释太浅
  - 缺少设计权衡
  - 数据模型和扩展点讲不透
  那么再升级到长版

简化原则：

- **先用短版验证方向**
- **再用长版补深度**
- 不要默认长 Prompt 一定更好
