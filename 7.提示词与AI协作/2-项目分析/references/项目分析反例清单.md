# 项目分析反例清单

> 用途：约束模型不要把“通用 GitHub 项目分析”误做成某一类项目的专用分析。
> 这份清单不是让模型输出给用户看的正文，而是分析时的自检参考。

---

## 一、项目形态误判类

### 反例 1：默认仓库是前端应用

错误表现：

- 一上来就找 `App.tsx` / `pages/` / `components/`
- 默认从“页面、状态、接口调用”开始分析
- 把 CLI、SDK、基础设施项目也硬套成 UI 架构

正确做法：

- 先识别项目类型
- 再决定入口从可执行文件、主包、服务启动点、插件注册点还是核心模块开始看

### 反例 2：默认仓库是服务端项目

错误表现：

- 默认要找 controller / service / repository
- 默认认为核心主链一定是 HTTP 请求链
- 默认项目一定围绕数据库 CRUD 组织

正确做法：

- 有些项目的主链可能是编译、推理、渲染、导出、同步、调度、插件加载，而不是请求-响应

### 反例 3：默认项目一定有状态管理层

错误表现：

- 默认寻找 `context / store / reducer / state manager`
- 把库项目或工具型项目错误描述成“状态驱动系统”

正确做法：

- 先确认这个项目是否真的存在集中状态层
- 对库、SDK、编译器、CLI，更常见的是核心抽象、数据流和调用图，而不是应用状态树

## 二、分析框架硬套类

### 反例 4：固定按“目录 -> 技术栈 -> 请求链 -> 数据库”输出

错误表现：

- 不管项目类型，章节都长得一样
- 最终报告看起来完整，但没有贴合项目真正重点

正确做法：

- 保留通用章节骨架，但核心执行链路和数据 / 抽象分析必须服从项目类型

### 反例 5：把“数据模型”硬理解成数据库表

错误表现：

- 没数据库也硬写 schema
- 对 Library / SDK / 编译器类项目，只能写出空泛内容

正确做法：

- 数据模型要按项目类型理解
- 对工具、库、编译器项目，应分析核心抽象、协议模型、AST / IR / config model / interface hierarchy

### 反例 6：把“模块”硬理解成前端页面或后端服务

错误表现：

- 看到目录就按 controller / service / page 分类
- 忽略包、crate、plugin、adapter、runtime、engine 这类真实边界

正确做法：

- 模块可以是目录、包、crate、plugin、子系统、runtime、adapter、协议层，不要只用 Web 项目术语

## 三、Mermaid 滥用类

### 反例 7：没做路由就直接画图

错误表现：

- 先画图，再想图回答什么问题
- 图型完全凭直觉选

正确做法：

- 先读取 Mermaid 路引入口
- 先输出 `Mermaid 路由声明`
- 先判断是否真的需要图

### 反例 8：一张图混多个问题

错误表现：

- 把系统组成、依赖方向、消息流转、状态变化混在一张图里
- 图越画越大，但解释力反而下降

正确做法：

- 一张图只回答一个主问题
- 如果问题跨多个维度，先拆问题，再拆图

### 反例 9：选了 reference，但没继承风格

错误表现：

- 说自己用的是 `C / 时序图`
- 实际却没有 `autonumber`、没有阶段 `Note over`、没有时序约束表达

正确做法：

- 不是“像那个图型”就算通过
- 必须显式继承所选 reference 的语法和输出契约

### 反例 10：为了完整感强行补图

错误表现：

- 明明文字足够解释清楚，还硬塞一张图
- 图只是装饰，不增加信息量

正确做法：

- 图只在能显著提升理解时才出现
- 没必要时，直接文字说明即可

## 四、结论质量问题类

### 反例 11：只讲“是什么”，不讲“为什么”

错误表现：

- 只列目录职责
- 只列依赖清单
- 只描述执行步骤
- 没有设计判断

正确做法：

- 每个重要部分都尽量补：为什么这样设计、收益是什么、代价是什么、替代方案为什么没选

### 反例 12：不绑定代码证据

错误表现：

- 结论像经验判断
- 无法落到具体文件、函数、模块、类型

正确做法：

- 重要结论尽量绑定到代码证据
- 至少能指向入口文件、核心类型、关键执行路径或关键模块

### 反例 13：把 FAQ 写成总结复述

错误表现：

- FAQ 只是把前文换个问法再说一遍
- 没有真正覆盖原理、设计、实践、边界四类问题

正确做法：

- FAQ 要补前文不够展开、但读者最可能追问的部分
- 特别要覆盖失败路径、资源限制、扩展方式和取舍原因

## 五、最小自检清单

在输出项目前，至少自检这 8 条：

- 我有没有先识别项目类型？
- 我有没有把分析框架硬套成某一种项目？
- 我有没有把“数据模型”误写成“数据库表结构”？
- 我有没有把“模块”误写成“页面 / controller / service”？
- 如果我出了 Mermaid，我有没有先做路由声明？
- 每张图是不是只回答一个主问题？
- 重要结论有没有绑定代码证据？
- 我讲清楚“为什么这样设计”了吗？
