# 01-初次项目分析

> 用途：第一次接触一个项目时，先建立全貌，而不是直接钻某个局部。
> 目标：识别项目类型，建立技术栈与架构认知，梳理目录分层与核心链路，并给出下一轮可深挖的专题建议。
> 适用范围：全栈、前端、后端、AI 复合项目，以及 CLI / SDK / Infra / Monorepo。

---

## 使用说明

- 这是一份第一轮分析 Prompt。
- 它的职责是建立“整体认知”，不是一次性穷尽所有细节。
- 如果已经做过第一轮总览，不要重复使用它深挖单一问题，改用 `02-专题深挖分析.md`。

---

## 推荐替换变量

- `{项目名称}`
- `{本地路径}`
- `{仓库地址}`
- `{项目简介}`
- `{项目类型猜测}`
- `{已知技术栈}`
- `{当前目标}`
- `{重点模块}`
- `{补充上下文}`

可留空，但不要编造。

```md
## 分析目标

请你基于真实代码仓库做分析，帮助我一次性建立这个项目的整体认知。

这次不要把精力平均分散到所有细节，而要优先回答下面几件事：

1. 这个项目本质上是什么类型的系统
2. 它的整体架构和模块分层是什么
3. 它主要依赖哪些技术栈和运行时能力
4. 哪一条核心链路最能代表系统
5. 下一轮最值得继续深挖哪些专题

---

## 执行要求

### 1. 先识别项目类型，不要预设框架

请先判断这个仓库更接近哪一类：

- 全栈项目
- 前端应用
- 后端服务
- AI / Agent / RAG / Workflow 项目
- SDK / Library
- CLI 工具
- 基础设施 / 平台工程
- 多语言 monorepo
- 其他

不要默认：

- 一定有前后端分离
- 一定有标准 MVC
- 一定有数据库
- 一定有 HTTP 请求主链
- 一定是 Web 应用

分析框架必须服从项目类型。

### 2. 必须基于代码证据，不要只看表层信息

请优先阅读能代表项目骨架的内容，再向下展开。至少覆盖下列相关项中的有效部分：

- 入口与启动点
- 包管理和依赖清单
- 构建脚本与运行脚本
- 核心目录和主模块
- 关键配置文件
- 状态 / 存储 / 缓存 / 队列 / 文件系统 / 外部服务
- AI 模型、Agent、Prompt、Tool、Workflow、RAG 管线相关模块
- 部署 / 容器 / 环境变量 / 运行方式

结论尽量绑定到具体文件、目录、函数、类、接口、配置项或运行链路。

### 3. 输出重点不是“列目录”，而是“解释结构”

请不要只平铺目录名。你需要解释：

- 为什么目录这样拆
- 为什么模块边界这样划分
- 为什么某些能力在前端、后端、Worker 或 AI 编排层
- 为什么采用这种状态、存储或交互方式
- 这样设计的收益、成本和边界分别是什么

### 4. 必须区分结论强度

在关键判断里尽量区分：

- 已确认：有明确代码证据
- 高概率判断：有较强迹象，但还缺一锤定音
- 待验证：当前只能提出疑点或假设

### 5. Mermaid 只在必要时使用

如果出图，请遵守下面规则：

- 先做 Mermaid 路由判断
- 图前先说明这张图回答什么问题
- 一张图只回答一个主问题
- 图后补充文字解释
- 如果文字足够清楚，可以不出图

如果你认为需要图，第一轮最多输出 3 张图，优先用于：

1. 系统架构图
2. 分层能力图
3. 端到端核心链路图

---

## 输出结构

请严格按下面结构输出。

### 1. 项目定位

用 2 到 4 句话回答：

- 这个项目解决什么问题
- 目标用户或使用者是谁
- 它更接近什么项目类型
- 它与常见同类项目相比最重要的差异是什么

### 2. 架构模式判定

请按以下格式输出：

- **结论**
- **代码证据**
- **为什么这样判断**
- **不属于哪些常见模式**

如果它是全栈或 AI 复合项目，请明确拆开说明：

- 前端层
- 后端层
- AI / Workflow / Agent 层
- 基础设施或数据层

### 3. 目录结构与模块职责

请优先使用“带职责注释的目录树”或“文字树”来说明目录结构。

要求：

- 展开一到两级主目录
- 在树上直接标注每个目录的大致职责
- 对文档、脚本、测试、构建产物、样例、生成代码、历史遗留目录直接标明性质
- 重点展开运行主链相关目录，不要把无关目录机械铺满

### 4. 技术栈全景

请整理出这个项目的技术栈，并按类别归纳：

- 语言
- 框架 / 运行时
- 构建工具
- UI / 前端基础设施
- 服务端框架或通信协议
- 数据存储 / 缓存 / 队列 / 文件系统
- AI 相关依赖或模型调用方式
- 测试、质量、部署和运维相关工具

请指出任何值得注意的非主流选型、组合式选型或明显权衡。

### 5. 分层结构解读

请结合代码证据说明：

- 系统按哪些层或能力域划分
- 每层分别负责什么
- 层与层之间如何交互
- 哪些层耦合较高，哪些边界比较清晰

如果项目不是典型分层架构，也要明确说清：

- 它采用了什么替代组织方式
- 为什么这样组织更合理

### 6. 核心链路分析

请选 1 条最能代表系统的核心链路，不要贪多。

它可以是：

- 一次用户操作到结果展示
- 一次前端请求到后端响应
- 一次任务创建到执行完成
- 一次模型调用 / Agent 编排 / RAG 检索生成
- 一次数据同步、构建、渲染、导出或发布

请说明：

- 为什么选这条链路
- 这条链路覆盖了哪些关键模块
- 关键步骤如何衔接
- 哪些地方最值得后续深挖

### 7. 风险、复杂点与设计权衡

请指出当前仓库中最值得注意的 3 到 6 个点，例如：

- 模块边界不清
- 配置复杂
- 前后端契约松散
- 状态来源分散
- AI 调用链过长
- 工程层和业务层耦合
- 缓存 / 持久化 / 队列边界复杂
- 多模块协作成本高

每一点都要说明：

- 现象
- 影响
- 为什么值得关注

### 8. 下一轮建议深挖的专题

请给出 4 个下一轮最值得深挖的专题，按优先级排序。

每个专题都要写：

- **标题**
- **为什么值得深挖**
- **建议重点看哪些目录 / 文件 / 模块**
- **深入后能回答什么问题**

---

## 输出风格要求

- 中文输出
- 保留关键英文术语
- 不要空泛赞美
- 不要只做目录导览
- 优先给结论，再给证据，再解释原因
- 如果证据不足，要明确说不确定，而不是补脑
```
