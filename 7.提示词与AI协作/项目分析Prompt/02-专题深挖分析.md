# 02-专题深挖分析

> 用途：在已经建立项目全貌之后，围绕单个专题继续深挖。
> 特点：可重复多次使用，每次只分析一个主题，必要时结合 Mermaid 辅助说明。
> 目标：找到关键代码证据，解释设计原因、边界、权衡、风险与演进空间。

---

## 使用说明

- 这不是第二次“重做全量分析”。
- 它的职责是围绕一个主题向下钻。
- 你可以对同一个项目连续使用多次，每次换一个专题。

---

## 推荐替换变量

- `{项目名称}`
- `{本地路径}`
- `{仓库地址}`
- `{已有分析摘要}`
- `{本轮专题}`
- `{本轮目标}`
- `{重点模块}`
- `{补充上下文}`

---

## 四种专题分析框架

根据本轮专题选择最合适的一种，不要机械全用。

### 1. 架构与模块边界框架

适合：

- 系统架构
- 模块职责
- 子系统拆分
- 前后端边界
- Agent / Workflow / Tool 边界
- 插件机制 / 扩展机制

重点追问：

- 模块为什么这样拆
- 谁依赖谁
- 哪些边界清晰，哪些边界含混
- 是否存在跨层调用、职责漂移或循环依赖

### 2. 数据与状态流框架

适合：

- 状态管理
- 表结构 / Schema
- 缓存
- 文件存储
- 前端状态来源
- RAG 索引、向量库、上下文缓存

重点追问：

- 数据从哪里来，到哪里去
- 哪一层拥有状态
- 状态如何同步、刷新、失效、持久化
- 是否存在重复状态、状态漂移或一致性风险

### 3. 运行时链路与交互框架

适合：

- HTTP 请求链
- 任务调度链
- 消息 / 事件流
- SSE / WebSocket
- 异步任务
- Agent 执行链、工具调用链、推理链

重点追问：

- 关键步骤如何串联
- 调用边界在哪里
- 哪些步骤同步，哪些步骤异步
- 错误、重试、超时和回退如何处理

### 4. 工程化与运行环境框架

适合：

- 构建体系
- 配置体系
- 环境变量
- 部署拓扑
- 多包仓库协作
- CI/CD
- 可观测性、测试、质量控制

重点追问：

- 本地开发和生产运行如何组织
- 配置如何分层
- 发布、部署、构建和调试路径是否清晰
- 工程层是否反过来制约业务或 AI 能力

```md
## 本次任务边界

这一次不要重复做完整项目总览。

请围绕“{本轮专题}”做一次聚焦分析，只回答与该专题强相关的问题：

- 它的真实结构是什么
- 关键代码证据在哪里
- 为什么这样设计
- 设计的收益、成本、边界和风险分别是什么

如果你发现这个专题本身定义过大，请先主动缩窄范围，再展开分析。

---

## 执行要求

### 1. 先基于已有认知校准范围

请先用 3 到 6 句话说明：

- 这个专题在整个项目里的位置
- 它和哪些模块强相关
- 它为什么值得单独深挖
- 这次不打算展开哪些外围问题

### 2. 分析必须绑定代码证据

不要只给概念层解释。请尽量绑定到：

- 目录
- 文件
- 函数
- 类
- 接口
- 类型
- 配置项
- 环境变量
- 请求或任务链路

每个关键判断都尽量说明“你是从哪里看出来的”。

### 3. 不只解释“怎么做”，还要解释“为什么这样做”

至少回答下面几个维度中的相关项：

- 为什么模块边界这样划分
- 为什么状态放在这一层
- 为什么交互链路这样编排
- 为什么使用这种存储、缓存、消息或 AI 调用方式
- 为什么没有采用更常见的替代方案
- 这种设计更适合当前项目的哪些现实约束

### 4. 必须识别风险与边界

请明确指出：

- 哪些结论已经确认
- 哪些判断仍需验证
- 该专题中最容易出问题的边界在哪里
- 当前实现的主要收益和主要代价分别是什么

### 5. Mermaid 按需使用，不默认强制

只有当文字不足以清楚表达结构或链路时，才使用 Mermaid。

如果出图，先做 Mermaid 路由判断，并遵守：

- 图前说明“这张图回答什么问题”
- 一张图只回答一个主问题
- 图后补充文字说明
- 不要为了完整感强行出图

---

## 输出结构

请按下面结构输出。

### 1. 专题定位

回答：

- 这个专题在系统中的作用是什么
- 它与哪些模块直接相关
- 这次分析聚焦的边界是什么

### 2. 关键结论

先给 3 到 6 条高密度结论，每条都尽量明确，不要写成泛化废话。

### 3. 结构 / 链路 / 状态 / 交互分析

根据专题实际情况，自适应选择最合适的展开方式。

可选方向包括但不限于：

- 模块结构
- 调用链路
- 数据流
- 状态流
- AI 推理链
- Agent / Tool 编排链
- 配置与运行时边界
- 扩展点与插件机制

如果专题同时涉及多个维度，请按主次展开，不要平均铺开。

### 4. 关键代码证据

请列出最重要的代码证据点，并说明各自证明了什么。

### 5. 设计原因与权衡

请回答：

- 为什么是这种设计
- 这样做解决了什么问题
- 它引入了什么复杂度
- 哪些场景下这套设计会暴露边界或代价

### 6. 风险与待验证问题

请分成两类：

- **已发现风险**
- **仍需进一步验证的问题**

### 7. 如果继续深挖，下一步最值得看什么

请给出 1 到 3 个紧邻当前专题的后续问题，每个问题都写明：

- 为什么要继续追
- 建议重点看哪里

---

## 输出风格要求

- 中文输出
- 保留英文术语
- 以判断和证据为主
- 不要复写整份项目总览
- 不要把没有证据的猜测写成确定结论
```
