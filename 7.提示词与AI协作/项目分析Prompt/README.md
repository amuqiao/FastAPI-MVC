# 项目分析 Prompt

这组 Prompt 用于分析任意代码项目，目标不是“泛泛介绍仓库”，而是帮助你稳定建立系统认知、逐步专题深挖，并在最后做一次完整性审查。

适用范围：

- 全栈项目
- 前端项目
- 后端项目
- AI / Agent / RAG / Workflow 项目
- SDK / Library / CLI / Infra / Monorepo

---

## 目录

- `01-初次项目分析.md`
  第一次使用。建立项目全貌，识别项目类型、技术栈、架构模式、目录分层、核心链路，并给出后续值得深挖的专题建议。
- `02-专题深挖分析.md`
  第二次及以后使用。围绕某个专题继续下钻，可重复使用多次，按主题拆开分析，不重复全量总览。
- `03-查漏补缺审查.md`
  最后使用。拿着项目代码和你已经积累的所有分析结果，做完整性审查，补齐遗漏、修正误判、收口结论。

---

## 推荐使用顺序

### 1. 先做第一次全貌分析

使用：

- `01-初次项目分析.md`

适合：

- 刚接触一个项目
- 还不知道它究竟更像前端、后端、AI 编排还是多模块系统
- 需要先建立整体认知

输出重点：

- 项目定位
- 架构模式
- 目录结构和模块职责
- 技术栈全景
- 1 条核心链路
- 后续深挖主题建议

### 2. 再做专题深挖

使用：

- `02-专题深挖分析.md`

适合：

- 第一轮全貌已经建立
- 需要围绕某个方向深入
- 想逐个吃透复杂模块，而不是每次重复扫全仓

常见专题：

- 系统架构与模块边界
- 前后端交互链路
- 后端请求处理链
- 状态管理与数据流
- 数据存储与缓存
- AI 推理链、Agent 编排链、RAG 检索链
- 插件机制 / 扩展机制 / 任务调度
- 部署拓扑、配置体系、运行时边界

### 3. 最后做查漏补缺

使用：

- `03-查漏补缺审查.md`

适合：

- 已经完成第一轮和若干轮专题深挖
- 想判断哪些地方还没覆盖
- 想把分析结果收束成一份更完整、更可靠的认知

输出重点：

- 已覆盖内容盘点
- 缺失问题清单
- 误判与不确定项
- 应补充的最终分析

## 三份 Prompt 的分工

### `01` 负责“建立全貌”

核心目标：

- 识别项目是什么
- 它由哪些部分构成
- 为什么这样分层
- 哪条链路最能代表系统

不要在这一阶段陷入过深的子模块细节。

### `02` 负责“围绕一个问题深挖”

核心目标：

- 只盯住 1 个专题
- 找到关键代码证据
- 解释设计原因、边界、权衡、风险
- 必要时用 Mermaid 帮助说明

不要重新做一遍第一轮总览。

### `03` 负责“审查是否漏了什么”

核心目标：

- 检查分析覆盖面是否完整
- 找出被忽略的关键路径、关键模块、关键状态、关键依赖
- 对前面结论做修正和补洞

不要在这一阶段继续无限扩写细节。

## 建议输入变量

三份 Prompt 都建议按需填写这些变量：

- `{项目名称}`
- `{本地路径}`
- `{仓库地址}`
- `{项目简介}`
- `{项目类型猜测}`
- `{已知技术栈}`
- `{当前目标}`
- `{重点模块}`
- `{已有分析摘要}`
- `{本轮专题}`
- `{补充上下文}`

不知道的变量可以留空，但不要伪造信息。

---

## 分析原则

- 先看代码证据，再下结论，不要只看 README 或目录名猜。
- 先识别项目类型，再选择分析框架，不要默认一切项目都是 Web CRUD。
- 结论尽量绑定到具体目录、文件、函数、类、接口、配置或运行链路。
- 不只回答“是什么”，还要回答“为什么这样设计”。
- 每一轮分析都要区分：
  - 已确认
  - 高概率判断
  - 仍需验证

## Mermaid 使用约束

这组 Prompt 支持 Mermaid，但 Mermaid 不是默认装饰项，只在确实能帮助理解时使用。

统一要求：

- 先做 Mermaid 路由判断
- 先说明这张图回答什么问题
- 一张图只解决一个主问题
- 图后必须补文字解释
- 如果文字足够清楚，可以不出图

每份 Prompt 末尾都预留了：

`Mermaid 样式风格参考 {}`

它的作用是让你在实际使用时，按项目或文档风格补充样式偏好，而不是把样式规范硬编码进正文。

---

## 最短使用法

如果你只想快速上手：

1. 第一次分析项目，用 `01-初次项目分析.md`
2. 选一个重点主题，反复用 `02-专题深挖分析.md`
3. 最后用 `03-查漏补缺审查.md` 做收口

这三份文件配合使用，比“一份超长 Prompt 试图解决所有问题”更稳定。
