# Mermaid 作图规范参考 · 使用说明

> 这个目录不是让模型“一次性全读完”的，而是先做图型判断，再引用对应 reference。
> 适用目标：你在 Prompt 后引用整个目录，或不确定该引用哪一份文件时，用本说明做路由。

## 一、先决定引用方式

### 1.1 优先级建议

优先顺序如下：

1. 已经明确要画什么图时，**直接引用单个文件**
2. 只知道“要画 Mermaid 图”，但还没确定图型时，**先引用本 README，再按路由选择具体文件**
3. 只有在确实需要让模型自行选型时，才**引用整个目录**

### 1.2 为什么不要默认整目录引用

这个目录覆盖 10 类图，语法横跨：

- `flowchart`
- `stateDiagram-v2`
- `sequenceDiagram`
- `gantt`
- `timeline`
- `mindmap`

如果直接整目录注入而不给路由约束，模型常见偏差是：

- 把“结构图”和“流程图”混画
- 在 `stateDiagram-v2` / `sequenceDiagram` 里误用 `classDef`、`subgraph`
- 本来只需 1 张图，却为了“完整”补出多张装饰图

所以整目录引用时，建议显式加一句：

```text
先根据 README 的路由规则判断图型，只加载必要的 reference，不要把整个目录内容混合成一套统一语法。
```

## 二、五份文件怎么分工

| 文件 | 负责的问题 | 包含图型 | 典型触发词 |
|------|-----------|---------|-----------|
| `mermaid-A-系统认知层.md` | 系统怎么组成、能力怎么分层、请求主链怎么走 | 系统架构图 / 分层能力结构图 / 端到端流程图 | 架构、分层、全景、主链路、系统组成 |
| `mermaid-B-代码深潜层.md` | 数据怎么存、代码怎么组织 | 数据模型关系图 / 类层级关系图 | 表关系、领域模型、类图、继承、接口、Mixin |
| `mermaid-C-运行时行为层.md` | 状态怎么变化、跨组件消息怎么流转 | 状态机图 / 时序图 | 生命周期、状态流转、异步链路、回调、SSE、Celery |
| `mermaid-D-规划与演进层.md` | 怎么排期、怎么看演进 | 甘特图 / 时间线图 | 路线图、里程碑、演进、规划、排期 |
| `mermaid-E-知识结构层.md` | 技术栈怎么归类、模块如何分组、主题如何展开 | 思维导图 | 技术栈、模块构成、能力地图、概念分类、知识结构 |

## 三、最短路由规则

按你真正想回答的问题选：

```text
想看“系统由哪些层/服务/模块组成”       → A
想看“能力按哪些阶段或层次展开”         → A
想看“一个请求从头到尾怎么同步流转”     → A

想看“数据表/持久化结构怎么关联”         → B
想看“类/接口/抽象层次怎么组织”         → B

想看“实体状态如何变化”                 → C
想看“跨组件/跨进程消息如何交互”         → C

想看“项目如何排期”                     → D
想看“技术/产品如何沿时间演进”           → D
想看“技术栈/模块/主题如何分层归类”      → E
```

如果一句需求同时命中两类问题，先拆问题，再拆图：

- “系统怎么组成 + 请求怎么走” → A 中通常出 2 张图
- “数据怎么存 + 代码怎么组织” → B 中通常出 2 张图
- “状态怎么变 + 消息怎么传” → C 中通常出 2 张图

不要为了省事把两类语义硬塞进一张图。

## 四、整目录引用时的推荐 Prompt 写法

### 4.1 只做路由，不立即出图

```text
请先阅读 Mermaid 作图规范参考目录中的 README，判断当前问题最适合哪一类图。
先输出：
1. 这次要回答的核心问题
2. 推荐图型
3. 应引用的具体 reference 文件
如果无需画图，直接说明原因，不要为了完整感强行出图。
```

### 4.2 允许出图，但要求按需加载

```text
请先按 Mermaid 作图规范参考/README.md 做图型路由。
只复用必要的 reference 文件，不要混用不同语法。
每张图输出前先说明“这张图回答什么问题”，并控制图数量，只保留必要图。
```

### 4.3 已知图型时的更好写法

如果你已经知道要画哪类图，不要再引用整个目录，直接引用单文件，例如：

- 系统架构 / 分层 / 主链路：`references/mermaid-A-系统认知层.md`
- 表关系 / 类层级：`references/mermaid-B-代码深潜层.md`
- 状态机 / 时序：`references/mermaid-C-运行时行为层.md`
- 甘特 / 时间线：`references/mermaid-D-规划与演进层.md`
- 思维导图 / 技术栈归类：`references/mermaid-E-知识结构层.md`

## 五、使用约束

- 一次任务优先只选 1 个文件；确实跨问题时再选 2 个
- `C` 文件与 `A/B` 的语法体系不同，不能共享 flowchart 写法
- `D` 只负责规划与演进，不承接技术栈归类
- `E` 负责技术栈、模块地图、概念分类，不表达时间和依赖
- 如果文字已经足够解释问题，可以不出图

## 六、维护建议

后续如果这个目录继续扩展，建议保持两个原则：

1. 新文件必须先声明“回答什么问题”，而不是先声明语法
2. 本 README 始终作为整目录引用时的唯一入口，不把路由规则分散写到各文件里
