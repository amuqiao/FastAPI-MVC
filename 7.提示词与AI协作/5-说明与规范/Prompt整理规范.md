# Prompt整理规范

这一区域采用“收集箱 + 按主题编号的独立模块”模式维护 Prompt。

## 顶层目录职责

- `1-收集箱/`：平时随手新增 Prompt Markdown，只管收集，不要求立刻整理。
- `2-项目分析/`：项目分析相关 Prompt 模块，已作为正式沉淀目录使用。
- `3-Mermaid作图/`：Mermaid 作图相关 Prompt 模块。
- `4-知识库规划/`：知识库目录分析与规划 Prompt 模块。
- `5-说明与规范/`：放这套 Prompt 目录的组织规范、示例和说明材料。

顶层目录按主题或用途划分，而不是按“资产形态”划分。

---

## 两种正式 Prompt 资产形态

“专题模块型”和“单文件复用型”是资产形态，不一定直接体现在一级目录名里。

### 1. 专题模块型

适合：
- 一个主题要长期维护
- 需要主 Prompt、二轮 Prompt、极简版、使用说明
- 可能需要单独的 references
- 需要补边界、反例、使用顺序

当前示例：
- `2-项目分析/`
- `4-知识库规划/`

### 2. 单文件复用型

适合：
- 高频复用
- 单文件就能工作
- 常带少量占位符
- 不需要一整套复杂目录

当前示例：
- `3-Mermaid作图/通用Mermaid作图Prompt.md`

---

## `2-项目分析/` 的正式结构

`2-项目分析/` 已经不是收集态目录，而是正式模块，建议至少保持下面结构：

- `项目分析主Prompt.md`
  第一轮 GitHub 项目全量分析模板。
- `项目分析二轮-4个深挖方向Prompt.md`
  第二轮方向筛选模板。
- `使用说明.md`
  说明适用边界、使用顺序、与收集箱的关系。
- `references/*`
  放反例清单、补充约束、辅助说明，不把 reference 漂在主题目录之外。

如果某个主题已经沉淀为正式模块，后续优先在正式模块内维护，不要再回到 `1-收集箱/` 里继续改正文。

---

## 收集箱与正式模块的关系

`1-收集箱/` 的职责是：

- 临时收集
- 快速试验
- 尚未稳定的 Prompt 暂存

它不应该长期承担“正式模板维护”的职责。

如果某个 Prompt 已经升级为正式模块，收集箱里的同名文件建议改成：

- 简短说明
- 指向正式维护位置的跳转
- 不再维护正文

这样做的目的：

- 防止正式版和收集箱版分叉
- 防止后续误在旧文件上继续改
- 让使用者知道当前应以哪份文件为准

---

## reference 规则

reference 应尽量跟着它所属的主题模块走，不要单独漂在外面，也不要双份维护。

例如：
- Mermaid 风格参考统一放在 `Mermaid作图规范参考/`
- 项目分析模块自己的反例与边界说明，放在 `2-项目分析/references/`

规则是：

- 通用 reference 跟通用模块走
- 专题 reference 跟专题模块走
- 不要在多个地方复制同一份 reference 正文

---

## 日常使用方式

你平时只需要做一件事：

1. 把好用的 Prompt 新增为一个 `.md` 文件放进 `1-收集箱/`

如果一开始判断不出它以后应该变成专题模块型还是单文件复用型，也没关系，先放收集箱。

---

## 收集箱推荐最小格式

````md
# Prompt标题

## Prompt
```text
这里放 Prompt 原文
```

## 备注
- 这次主要解决什么问题
- 为什么这次比较好用
````

如果你懒得写备注，只写 `Prompt` 正文也可以。

---

## 定期整理方式

隔一段时间，把 `1-收集箱/` 整个目录交给大模型，按下面五类整理：

- 保留观察
- 合并去重
- 升级为专题模块型
- 升级为单文件复用型
- 删除

推荐直接这样提：

```text
请按 `7.提示词与AI协作/5-说明与规范/Prompt整理规范.md` 整理 `7.提示词与AI协作/1-收集箱/`。
把里面的 Prompt 分类为：保留观察、合并去重、升级为专题模块型、升级为单文件复用型、删除。
如果某些 Prompt 已经足够稳定，请按最合适的资产形态沉淀。
如果某个 Prompt 已经升级为正式模块，请把收集箱中的同名文件改成跳转说明，而不是继续维护正文。
```

---

## 升级判断

通常满足下面至少 3 条再升级：

- 已重复使用 3 次以上
- 不依赖某次临时上下文
- 不同场景下表现比较稳定
- 已经能说清输出边界和任务顺序
- 未来的你拿出来仍然能直接复用

如果已经升级，再额外满足下面 2 条，建议从“收集箱文件”转成“正式模块维护”：

- 已经出现主 Prompt / 二轮 Prompt / 使用说明 / reference 的组合需求
- 已经需要明确“哪份是正式版、哪份只是入口跳转”
