# 使用示例

这份文档是当前主题的唯一入口说明，面向刚开始使用 Prompt 和 skill 的初学者。

它主要回答 4 个问题：

- Prompt 和 skill 有什么区别
- `codebase-project-analysis` 怎么用
- `mermaid-analysis-visualizer` 怎么用
- 什么时候该用 skill，什么时候只用技术笔记模板

当前你主要会用到这三类能力：

- `codebase-project-analysis`
- `mermaid-analysis-visualizer`
- `技术笔记提示词-*`

## 一、先理解：Prompt 和 skill 是什么

### 1. Prompt 是什么

Prompt 可以理解成一段可复用的提问模板。

它适合：
- 你已经知道自己要 AI 做什么
- 任务结构比较自由
- 你想快速复制一段现成要求

典型例子：
- 讲解 Kafka
- 对比 Kafka 和 RabbitMQ
- 写一篇面向初学者的 Transformer 原理笔记

### 2. Skill 是什么

skill 可以理解成“带内置工作流的能力模块”。

它不只是告诉 AI “写什么”，还会约束 AI：
- 什么时候触发
- 先做什么判断
- 默认工作方式是什么
- 应该参考哪些资料
- 输出时要遵守哪些规则

典型例子：
- 先分析本地代码仓库，再决定看整体还是看模块
- 先判断该不该画图，再选择 Mermaid 图型

### 3. 初学者怎么选

可以直接记住这条：

- **看代码库**：优先用 skill
- **学技术主题**：优先用 Prompt 模板
- **需要画 Mermaid 图**：优先用 Mermaid skill

再简单一点：

- **任务有稳定工作流** → 用 skill
- **任务更像自由写作** → 用 Prompt

### 4. skill 不是替代 Prompt，而是增强 Prompt

这是初学者最容易混淆的一点。

可以这样理解：

- Prompt 负责说明：**这次具体要做什么**
- skill 负责说明：**做这件事时，启用哪套能力和规则**

两者不是二选一关系，而是分工关系。

例如这句话：

```md
使用 $mermaid-analysis-visualizer 画一张系统架构图。
```

里面有两层意思：

- `画一张系统架构图`：这是任务本身
- `$mermaid-analysis-visualizer`：这是启用的能力

如果你只写：

```md
使用 $mermaid-analysis-visualizer
```

那模型只知道你启用了这个 skill，但并不知道你到底要它：
- 画架构图
- 画流程图
- 画时序图
- 还是把一段分析转成图

所以最短的正确用法不是“只写 skill 名”，而是：

```md
使用 $skill名 + 具体任务
```

例如：

- `使用 $mermaid-analysis-visualizer 画一张系统架构图。`
- `使用 $mermaid-analysis-visualizer 把下面这段分析转成 Mermaid 图。`
- `使用 $codebase-project-analysis 分析本地项目 \`/path/to/repo\`。`

### 5. 只知道 skill 时，怎么开始用

如果你现在只记得 skill 名，不知道怎么写复杂 Prompt，也没关系。

先用最短模板即可：

#### 最短模板 1：Mermaid

```md
使用 $mermaid-analysis-visualizer 画一张系统架构图。
```

#### 最短模板 2：代码库分析

```md
使用 $codebase-project-analysis 分析本地项目 `/path/to/repo`。
```

然后如果你想更稳一点，再多补一行“重点关注什么”：

```md
使用 $mermaid-analysis-visualizer 画一张系统架构图。

重点关注：
- 系统分层
- 核心组件
- 调用关系
```

或者：

```md
使用 $codebase-project-analysis 分析本地项目 `/path/to/repo`。

我现在最关心：
- 项目定位
- 主模块
- 主链路
```

对初学者来说，先学会这种“短句 + skill”的写法就够了，不必一开始追求复杂 Prompt。

## 二、你当前已有的两个 skill

### 1. `codebase-project-analysis`

作用：
- 分析本地代码仓库
- 默认先快速扫描，再逐步深入
- 会先澄清你的分析目标
- 适合看项目定位、主模块、主链路、设计决策

适用场景：
- 初次看一个 GitHub 项目
- 想快速理解本地代码库
- 想从 scan 深入到 architecture 或 module

不适合：
- 讲解 Kafka、Transformer 这类知识主题
- 只想画图、不关心代码库本身

### 2. `mermaid-analysis-visualizer`

作用：
- 判断当前问题是否值得画图
- 选择最合适的 Mermaid 图型
- 输出 Mermaid 代码和配套说明
- 避免错用语法或乱画图

适用场景：
- 你已经知道想画架构图、流程图、时序图
- 你分析完项目后，想把结果画出来
- 你写技术笔记时，希望图更规范

不适合：
- 深度阅读整个代码库
- 独立承担完整项目分析

## 四、最重要的决策规则

如果你拿不准该用什么，先按下面判断。

### 场景 1：我要看一个本地项目

优先用：
- `$codebase-project-analysis`

如果你还想配图：
- 再加 `$mermaid-analysis-visualizer`

### 场景 2：我已经知道要画什么图

优先用：
- `$mermaid-analysis-visualizer`

例如：
- 画架构图
- 画时序图
- 画状态机
- 把已有分析转成图

### 场景 3：我要学一个技术主题

优先用：
- `技术笔记提示词-*`

必要时在 Prompt 里补一句：
- `如有必要，请使用 $mermaid-analysis-visualizer 输出合适的 Mermaid 图，并附配套说明。`

### 场景 4：我既想看代码库，又想沉淀图和文档

组合用：
- `$codebase-project-analysis`
- `$mermaid-analysis-visualizer`

这个组合是你后续最常用的方式。

## 五、最推荐的日常工作流

可以直接记成下面这 3 条：

1. 看代码库  
先用 `$codebase-project-analysis`

2. 需要图  
让它默认调用或显式补上 `$mermaid-analysis-visualizer`

3. 学技术主题  
直接用 `技术笔记提示词-*`

## 六、`codebase-project-analysis` 怎么用

### 1. 默认特点

这个 skill 的默认行为是：

- 先澄清你的分析目标
- 默认从 `scan` 开始
- 只抓主模块和主链路
- 先给简洁结论
- 图有帮助时默认会带

### 2. 三种深度模式

#### `scan`

适合：
- 第一次看项目
- 想快速建立整体认知

通常会输出：
- 项目定位
- 技术栈
- 主模块
- 最关键执行链路
- 2 到 3 个关键设计决策
- 1 到 2 张关键图

#### `architecture`

适合：
- 已经看过 scan
- 现在要深入整体架构

通常会输出：
- 模块依赖关系
- 核心执行链路
- 数据模型
- 扩展点机制
- 风险或高耦合区域
- 2 到 4 张图

#### `module`

适合：
- 已经知道要深挖哪个模块

通常会输出：
- 模块职责边界
- 入口函数或触发条件
- 模块内部主链路
- 依赖方向
- 核心数据结构
- 错误处理与边界情况
- 最小改动路径
- 1 到 3 张图

### 3. 可直接复制的示例

#### 示例 A：第一次看本地项目

```md
使用 $codebase-project-analysis 分析本地项目 `/path/to/repo`。

先澄清我的分析目标，默认从 scan 开始。
我现在最关心：
- 这个项目的整体定位
- 主模块和主链路
- 2 到 3 个关键设计决策

只抓主线，不要一开始展开过多细节。
如果图能明显降低理解成本，请默认配 Mermaid 图。
```

#### 示例 B：继续深入整体架构

```md
继续使用 $codebase-project-analysis 深入这个项目，切换到 architecture 模式。

这次重点看：
- 模块依赖关系
- 核心执行链路
- 数据模型
- 扩展点机制
- 风险或高耦合区域

继续使用 Mermaid 图辅助说明，但不要堆图，控制在 2 到 4 张内。
```

#### 示例 C：聚焦单个模块

```md
使用 $codebase-project-analysis 深入分析模块 `{模块名}`，采用 module 模式。

重点输出：
- 职责边界
- 入口函数或触发条件
- 模块内部主链路
- 依赖方向
- 核心数据结构
- 错误处理与边界情况
- 如果后续要改这个模块，最小改动路径是什么

如果图有帮助，请配 1 到 3 张 Mermaid 图并附说明。
```

## 七、`mermaid-analysis-visualizer` 怎么用

### 1. 什么时候单独用它

单独使用这个 skill，通常是因为：

- 你已经知道要画什么图
- 你已经有一段分析结果，想把它转成图
- 你在写笔记，希望图更规范

### 2. 它会帮你做什么

它会：
- 判断是否值得画图
- 选择合适图型
- 输出 Mermaid 代码
- 补配套说明
- 做图型和语法自检

### 3. 可直接复制的示例

#### 示例 A：直接画架构图

```md
使用 $mermaid-analysis-visualizer 输出一张系统架构图。

请先用 2 到 4 句话说明这张图回答什么问题，再给 Mermaid 代码，最后补一段图解说明。

如果信息不足，请明确指出缺口，不要编造不存在的组件关系。
```

#### 示例 B：把已有分析转成图

```md
基于下面这段分析结果，使用 $mermaid-analysis-visualizer 选择最合适的 Mermaid 图型表达。

要求：
- 优先选择最能降低理解成本的那一张图
- 先给简短分析
- 再给 Mermaid 代码
- 最后给图解说明

分析结果：
{粘贴你的分析内容}
```

#### 示例 C：画异步时序图

```md
使用 $mermaid-analysis-visualizer 描述这条异步链路。

我想重点看：
- 谁先触发
- 谁在等待
- 哪一步是异步回调
- 哪些节点最容易成为故障点

请优先考虑时序图或状态机图，并说明为什么这样选。
```

## 八、两个 skill 怎么组合用

这是最推荐的组合。

逻辑是：

1. 先用 `codebase-project-analysis` 理解项目
2. 再用 `mermaid-analysis-visualizer` 把关键结论转成图

也可以一次性在同一个请求里明确说出来。

### 可直接复制的组合示例

```md
使用 $codebase-project-analysis 分析本地项目 `/path/to/repo`，
并使用 $mermaid-analysis-visualizer 辅助可视化。

先澄清我的分析目标，默认从 scan 开始。
输出：
- 项目定位
- 技术栈
- 主模块
- 最关键执行链路
- 2 到 3 个关键设计决策

如果适合画图，优先输出 1 到 2 张最关键的 Mermaid 图，并给出配套说明。
```

当你的对象不是“代码仓库”，而是“知识主题”时，优先用模板。

例如：
- Kafka
- Transformer
- RAG

### 2. 模板怎么选

#### `技术笔记提示词-01-主题讲解`
适合：
- 快速讲清一个主题是什么
- 面向小白或初学者

#### `技术笔记提示词-02-原理深挖`
适合：
- 深入讲工作原理
- 看执行过程和设计动机

#### `技术笔记提示词-03-对比选型`
适合：
- Kafka vs RabbitMQ
- Transformer vs RNN

#### `技术笔记提示词-04-工程实践`
适合：
- 如何落地
- 怎么接入项目
- 常见坑和排查方式

### 3. 可直接复制的示例

#### 示例 A：学习 Kafka

```md
请使用 `技术笔记提示词-01-主题讲解.md` 的风格，讲解 Kafka。

要求：
- 面向初学者
- 先补基础概念
- 用类比帮助理解
- 解释为什么 Kafka 这样设计
```

#### 示例 B：深入理解 Transformer

```md
请使用 `技术笔记提示词-02-原理深挖.md` 的风格，深入讲解 Transformer。

重点关注：
```

```md
- 适合什么场景
```

## 十、初学者最容易犯的错误

### 错误 1：什么都想用 skill

不是所有任务都适合 skill。

如果你只是：
- 学 Kafka
- 学 Transformer
- 做一篇知识型笔记

那通常直接用模板更轻。

### 错误 2：什么都想自己写 Prompt

如果任务已经有稳定工作流，例如：
- 看代码库
- 逐层分析项目
- 选择 Mermaid 图型

那 skill 往往更省脑力，也更稳定。

### 错误 3：一上来要求 AI 输出过长文档

更好的方式通常是：
- 先 scan
- 再 architecture
- 再 module

不要一开始就要求“全量深度分析”，否则很容易又长又乱。

### 错误 4：只要图，不要说明

对你现在这个阶段，图必须配说明。

因为你真正要的是：
- 看懂重点
- 知道先看哪里
- 知道图没覆盖什么

而不是只堆 Mermaid 代码。
