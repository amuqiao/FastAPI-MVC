# 目录结构图

> 文档职责：定义目录结构图的用途、边界、必要信息要素和参考图。
> 适用场景：需要快速说明源码、文档或资源目录如何组织时使用。
> 阅读目标：判断何时使用这张图，并理解它与模块依赖图、整体架构图的边界。
> 目标读者：需要先建立项目文件组织认知的人。

## 1. 标准定位

- 上位标准：`Tree View / Annotated Tree`
- Mermaid 常见写法：不使用 Mermaid，使用文本树

## 2. 这张图回答什么问题

- 项目的目录按什么层级组织
- 哪些目录是核心入口
- 关键路径分别承载什么职责

不回答：

- 服务之间如何调用
- 模块之间的依赖方向
- 业务请求如何流转

## 3. 必要信息要素

- 1 个根目录
- 3-6 个一级目录
- 每个关键目录补 1 句职责说明
- 不展开到过细文件级别

## 4. 节点表达规则

- 应写：目录、子目录、关键路径及必要职责注释。
- 不应写：服务调用、数据库实体、代码继承关系、接口路径或部署区域。
- 禁止混入：运行时链路、模块依赖箭头、类层次结构。

## 6. 参考图 2：Annotated Tree

```text
project/
├── app/                  # 核心后端代码
│   ├── api/              # 接口入口
│   ├── services/         # 业务服务
│   └── repositories/     # 数据访问
├── web/                  # 前端页面与组件
├── infra/                # 部署与运维脚本
├── docs/                 # 设计文档与规范
└── tests/                # 测试用例
```

## 7. 使用边界

- 该图适合作为项目认知入口，不用于替代架构图。
- 如果重点是模块调用关系，应改用模块依赖图。
- 如果重点是系统边界和外部依赖，应改用系统上下文图或整体架构图。
