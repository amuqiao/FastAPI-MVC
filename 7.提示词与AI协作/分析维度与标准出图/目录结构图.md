# 目录结构图

> 文档职责：定义目录结构图在项目分析中的用途、边界和最小输出要求。
> 适用场景：需要快速说明源码、文档或资源目录如何组织时使用。
> 阅读目标：区分“目录树”与“模块依赖”或“系统架构”的职责边界。
> 目标读者：需要先建立项目文件组织认知的人。

## 1. 标准定位

- 上位标准：`Tree View / Annotated Tree`
- 表达形式：使用文本树或带职责标注的目录树
- 与现有 Mermaid 参考的关系：不属于 Mermaid 图种，属于项目认知入口类表达

## 2. 这张图回答什么问题

- 项目的目录按什么层级组织
- 哪些目录是核心入口
- 关键路径分别承载什么职责

不回答：

- 服务之间如何调用
- 模块之间的依赖方向
- 业务请求如何流转

## 3. 最小出图要求

- 1 个根目录
- 3-6 个一级目录
- 每个关键目录补 1 句职责说明
- 不展开到过细文件级别

## 4. 参考示例 1：Tree View

```text
project/
├── app/
│   ├── api/
│   ├── services/
│   └── repositories/
├── web/
│   ├── pages/
│   └── components/
├── infra/
│   ├── docker/
│   └── scripts/
└── docs/
    ├── architecture/
    └── prompts/
```

## 5. 参考示例 2：Annotated Tree

```text
project/
├── app/                  # 核心后端代码
│   ├── api/              # 接口入口
│   ├── services/         # 业务服务
│   └── repositories/     # 数据访问
├── web/                  # 前端页面与组件
├── infra/                # 部署与运维脚本
├── docs/                 # 设计文档与规范
└── tests/                # 测试用例
```

## 6. 使用边界

- 这张图适合作为项目认知入口，不适合替代架构图
- 如果重点变成模块调用关系，应改画模块依赖图
- 如果重点变成系统边界和外部依赖，应改画系统上下文图或整体架构图
