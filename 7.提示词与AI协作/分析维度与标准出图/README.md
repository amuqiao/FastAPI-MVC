# 分析维度与标准出图

> 文档职责：定义项目分析场景下的标准图名、所属维度、建模标准、Mermaid 常见写法和一句话用途，并作为本目录的唯一入口。  
> 适用场景：为任意项目快速决定“该画哪些图”时使用。  
> 阅读目标：先确定分析阶段，再路由到对应图种、图文档和 Mermaid 样式规则。  
> 目标读者：需要维护项目分析方法和出图标准的人。

## 1. 目录内容

本目录维护三类内容：

- 一张总表：说明 `图名 / 维度 / 建模标准 / Mermaid 常见写法 / 一行说明`
- 对应图文档：每个文档只负责一种图
- 一张 Mermaid 样式风格速查：按 Mermaid 类型集中维护可复用样式规则

## 2. 核心概念

### 2.1 图名、建模标准、Mermaid 写法的分工

- 图名是唯一索引，不用同义词并列做主名称
- 只有总表里的 `图名` 一栏保留 `中文名（English Name）` 形式
- `建模标准` 决定“这张图表达什么”，`Mermaid 常见写法` 决定“这张图怎么落地”
- `建模标准` 一栏优先写正式标准名；如果没有正式标准，则写通用业界名称

### 2.2 C4 Model Level 与 Mermaid 的关系

`C4 Model Level` 和 `Mermaid` 不是同一层概念。

- `C4 Model Level 1` 是系统上下文图，Mermaid 常见落地可用 `flowchart`
- `C4 Model Level 2` 是容器图，Mermaid 常见落地可用 `flowchart`
- `C4 Model Level 3` 是组件图，Mermaid 常见落地可用 `flowchart`
- `C4 Model Level 4` 是代码级结构图，Mermaid 可用 `flowchart` 或 `classDiagram`

一句话区分：

- `建模标准` 回答“画什么”
- `Mermaid 常见写法` 回答“怎么画”

因此：

- `C4 Model Level 2` 不是 `flowchart`
- `flowchart` 也不是一种建模标准
- 它们的关系是“标准层”和“语法层”的关系

## 3. 使用顺序

推荐按下面的顺序使用本目录：

1. 先看“标准图总表”，确认当前问题属于哪一类图。
2. 再看对应图文档，理解该图的边界、推荐结构和参考图。
3. 最后按需查看 `Mermaid样式风格速查.md`，决定具体语法和轻样式。

如果当前问题是“该画哪张图”，优先看总表；如果当前问题是“这张图该怎么写”，优先进入图文档；如果当前问题是“这种 Mermaid 语法怎么排版更稳”，再看样式速查。

## 4. AI 路由入口

本节用于让 `README.md` 既能给人读，也能被其他 Prompt 直接当作“项目分析路由入口”引用。

如果其他 Prompt 中写了类似：

```text
项目分析相关的图种选择、出图规范和 Mermaid 样式，请查看
@/Users/admin/Downloads/Code/AI-Engineering-Knowledge-Base/7.提示词与AI协作/分析维度与标准出图/README.md
```

则应按下面的顺序执行。

### 4.1 AI 执行顺序

1. 先判断当前问题属于 `首轮全貌`、`二轮深挖` 还是 `三轮补充`。
2. 再从“标准图总表”中选择最能直接回答当前问题的图。
3. 对每一张已选图，继续读取对应单图文档，而不是直接凭空画图。
4. 如果该图需要 Mermaid，再读取 `Mermaid样式风格速查.md` 中对应语法的小节。
5. 输出时先给“路由决策”，再给分析正文，最后按需出图。

### 4.2 路由规则

- 不为“显得完整”而机械补图，只选当前问题真正需要的图。
- 单张图只回答一个主问题；若一个问题需要多张图，先拆问题再拆图。
- 如果重点是“系统怎么组成”，优先选系统上下文图、整体架构图、接口地图等结构图。
- 如果重点是“事情怎么一步步发生”，优先选核心业务链路图，而不是强行使用结构图。
- 如果重点是“状态如何变化”，优先选状态机图；如果重点是“谁和谁交互、谁在等待谁”，优先选时序型核心业务链路图。
- 如果重点是“时间规划或历史演进”，优先选甘特图或时间线图，而不是 `flowchart`。

### 4.3 单图文档引用协议

对每一张已选图，必须继续读取对应图文档，并至少提取以下信息：

- 这张图回答什么问题
- 必要信息要素
- 节点表达规则
- 参考图
- 使用边界

禁止做法：

- 跳过单图文档，直接按记忆画图
- 把一种图的节点规则套到另一种图上
- 用“看起来像”代替建模标准判断

### 4.4 Mermaid 样式引用协议

如果已选图需要 Mermaid：

1. 先确定这张图对应的 Mermaid 语法
2. 再读取 `Mermaid样式风格速查.md` 中对应语法的小节
3. 只复用该 Mermaid 类型对应的样式规则，不跨语法乱套

常见对应关系：

- `flowchart`：系统上下文图、整体架构图、接口地图、分层能力结构图、核心组件图、模块依赖图、部署图、构建与发布流程图、流程型核心业务链路图
- `sequenceDiagram`：时序型核心业务链路图
- `classDiagram`：代码图中的 UML Class 表达
- `stateDiagram-v2`：状态机图
- `erDiagram`：数据模型图中的 ERD 表达
- `mindmap`：技术栈图中的总览表达
- `gantt`：甘特图
- `timeline`：时间线图

### 4.5 推荐输出顺序

如果本 README 被上层 Prompt 当作路由入口引用，推荐输出顺序如下：

1. 路由决策
2. 结论摘要
3. 分析正文
4. 按需输出标准图
5. 图后一句话解读

推荐的“路由决策”至少包含：

- 当前阶段
- 已选图种
- 每张图引用了哪个单图文档
- 如果用了 Mermaid，引用了哪种 Mermaid 语法和样式规则
- 哪些图没选，以及为什么没选

## 5. 标准图总表

项目分析不应把所有图杂糅在一起，而应按理解顺序逐层展开。

### 5.1 首轮全貌

这一阶段的目标是：从 0 到 1 建立项目整体认知，优先回答“项目是什么、由什么组成、主链路怎么走”。

| 顺序 | 图名 | 维度 | 建模标准 | Mermaid 常见写法 | 一行说明 |
|------|------|------|----------|------------------|---------|
| 1 | 目录结构图（Directory Structure Diagram） | 认知维度 | Tree View / Annotated Tree | 不使用 Mermaid，使用文本树 | 使用文本树展示源码、文档或资源目录层级，并标注关键模块职责。 |
| 2 | 技术栈图（Tech Stack Diagram） | 认知维度 | Tech Stack Diagram | `mindmap` / `flowchart` | 按技术域分类展示前端、后端、数据存储、中间件、治理和基础设施等技术组件。 |
| 3 | 系统上下文图（System Context Diagram） | 上下文维度 | C4 Model Level 1 | `flowchart` | 展示系统与用户、外部系统和业务环境之间的关系。 |
| 4 | 整体架构图（High-Level Architecture Diagram） | 结构维度 | C4 Model Level 2 | `flowchart` | 展示系统边界、核心子系统、外部依赖及主要交互协议。 |
| 5 | 接口地图（Interface Map / API Surface Map） | 接口维度 | Interface Map / API Surface Map | `flowchart` | 展示系统对外提供的 API、CLI、事件入口或 webhook 接口面。 |
| 6 | 核心业务链路图（Key Business Flow Diagram） | 动态维度 | UML Sequence / Flow Diagram | `sequenceDiagram` / `flowchart` | 选取一个典型端到端业务场景，展示关键节点、数据流向和异常分支。 |

### 5.2 二轮深挖

这一阶段的目标是：在已经建立全貌后，继续深入结构、数据、状态和部署等局部问题。

| 顺序 | 图名 | 维度 | 建模标准 | Mermaid 常见写法 | 一行说明 |
|------|------|------|----------|------------------|---------|
| 1 | 分层能力结构图（Layered Capability Map） | 能力维度 | Layered Capability Map | `flowchart` | 按用户层、应用层、服务层、数据层、基础平台层组织关键能力。 |
| 2 | 核心组件图（Component Diagram） | 结构维度 | C4 Model Level 3 | `flowchart` | 深潜某个核心容器内部的组件划分与协作关系。 |
| 3 | 数据模型图（Data Model Diagram） | 数据维度 | Data Model / ERD | `flowchart` / `erDiagram` | 展示核心实体、表关系和关键引用字段。 |
| 4 | 模块依赖图（Module Dependency Diagram） | 结构维度 | Module Dependency Diagram | `flowchart` | 展示模块依赖方向、公共模块边界和循环依赖风险。 |
| 5 | 状态机图（State Machine Diagram） | 动态维度 | UML State Machine | `stateDiagram-v2` | 展示核心实体或异步任务的状态迁移和触发条件。 |
| 6 | 部署图（Deployment Diagram） | 部署维度 | C4 Deployment | `flowchart` | 展示运行环境、节点区域、网络入口和部署边界。 |
| 7 | 构建与发布流程图（Build & Release Flow Diagram） | 工程维度 | Build & Release Flow Diagram | `flowchart` | 展示本地构建、测试、打包、发布和部署的工程执行流程。 |

### 5.3 三轮补充

这一阶段的目标是：补充代码、规划和演进类信息，用于专项分析，而不是默认主线。

| 顺序 | 图名 | 维度 | 建模标准 | Mermaid 常见写法 | 一行说明 |
|------|------|------|----------|------------------|---------|
| 1 | 代码图（Code Diagram） | 代码维度 | C4 Model Level 4 / UML Class | `flowchart` / `classDiagram` | 深潜某个核心组件内部的代码结构、接口层次和实现关系。 |
| 2 | 甘特图（Gantt Chart） | 规划维度 | Gantt | `gantt` | 展示项目分阶段排期、任务依赖和并行关系。 |
| 3 | 时间线图（Timeline Diagram） | 演进维度 | Timeline | `timeline` | 展示技术、产品或架构在时间轴上的演进节点。 |

## 7. 维护约定

### 7.1 当前图文档结构约定

- 不再拆维度子文档
- 文档名直接与图名对齐
- 每个图文档只负责一种图名，不混入别的图种
- 当前大多数图文档采用如下主结构：`标准定位`、`这张图回答什么问题`、`必要信息要素`、`节点表达规则`、`参考图`、`使用边界`
- 对高频核心图，可在 `节点表达规则` 与 `参考图` 之间补充 `最佳实践速查`
- 如果一个图名存在多种标准表达或多种 Mermaid 写法，可在同一文档下维护 `参考图 1 / 参考图 2 / ...`
- 多张参考图时，标题应直接写出差异来源，例如标准差异或写法差异
- `标准定位` 统一只保留两项：`上位标准`、`Mermaid 常见写法`
- `节点表达规则` 统一作为约束层编写，只回答“应写 / 不应写 / 禁止混入”，不围绕某一张参考图写图注
- `参考图` 统一作为高质量样本维护，优先选择经典场景，不再保留过弱的演示型示例
- Mermaid 示例默认只保留最小可读样式；跨图复用的完整样式体系统一维护在 `Mermaid样式风格速查.md`
- 颜色、边框、子图层装饰不属于标准本身，优先保证语义正确和结构清晰
- 如确需补充样式，优先复用 `Mermaid样式风格速查.md`；当前图文档只保留与该图强相关的局部规则
- 不使用过重的 `themeVariables`、大段 `classDef` 或复杂装饰性样式作为标准的一部分
- 允许参考 Mermaid 规范中的轻样式思路：`subgraph + layerStyle`、业务节点分类配色、`note` / `noteStyle` 提示块
- `note` 不是必备元素，只在整体架构图、核心业务链路图、状态机图等高信息密度图中按需保留
- 说明能在图文解释部分说清楚时，优先放在正文，不强行占用图面
- `sequenceDiagram`、`stateDiagram-v2` 如需补充关键提示，优先使用原生 `Note` / `note` 语法
- 如果后续新增图种，先更新总表，再新增对应图文档

### 7.2 例外与对齐规则

- `必要信息要素` 目前仍是目录内图文档的共同结构之一；若后续个别图种不再需要这一节，应先统一调整模板，再回写本 README
- `最佳实践速查` 不是所有图的必备章节，只在高频、高信息密度或确有稳定经验可沉淀时补充
- `Mermaid样式风格速查.md` 负责“按 Mermaid 类型复用的样式规则”；单图文档负责“该图表达什么、不能混入什么、参考图长什么样”
- 若 `README.md`、图文档、`Mermaid样式风格速查.md` 三者出现冲突，应优先修正文档定义层，再修正样式层

### 7.3 维护原则

- 先维护“建模边界”，再维护“参考图质量”，最后才处理“样式一致性”
- 优先保持总表、图文档、样式速查三者口径一致，不允许出现互相冲突的定义
- 如果某一条规则只适用于单个图种，应写进对应图文档；如果适用于某种 Mermaid 语法，应写进 `Mermaid样式风格速查.md`
- 新增图种时，先补总表定义，再补图文档，最后按需补样式速查
