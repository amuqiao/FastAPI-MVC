# 知识库构建与维护规范

> 定位：这是一份独立于具体主题的通用规范，用于指导任意知识库的创建、扩展、维护和更新。

---

## 1. 文档目标

本文解决四类问题：

1. 如何从一个主题出发，搭建可长期维护的知识库结构
2. 如何区分入口页、总纲文档、子主题文档的职责
3. 如何在新增材料时只更新最小必要文件，而不是反复重写整库
4. 如何通过 prompt 稳定驱动知识库构建和维护

---

## 2. 适用范围

适用于以下场景：

- 用 Markdown 管理个人或团队知识库
- 使用 AI / prompt 协助构建知识文档
- 主题会持续新增资料，需要长期迭代
- 希望避免文档重复、结构混乱、结论失控

不适用于以下场景：

- 一次性草稿
- 不需要长期维护的临时笔记
- 强依赖数据库或专业知识管理系统的复杂内容平台

---

## 3. 核心原则

### 3.1 单一职责

一个文件只承担一种主要职责，不把入口、总纲、细节、变更记录混写在一起。

### 3.2 稳定结构

目录一旦建立，后续优先在既有结构内更新，不轻易改目录名、文件名和层级。

### 3.3 最小修改

新增资料时，只修改最小必要文件，而不是整套文档重写。

### 3.4 分层表达

知识内容至少区分为：

- 基础概念
- 核心原理
- 工程实践
- 平台差异或场景差异
- 待核验项

### 3.5 冲突隔离

不同来源之间有冲突时，先标记冲突，不强行混为一谈。

### 3.6 稳定结论优先

只有相对稳定、可复用的内容才进入总纲。版本相关、平台相关、未核实内容，单独放在差异或待核验区域。

### 3.7 可导航

知识库必须有清晰入口，用户应能快速知道：

- 先看什么
- 不同问题该看哪里
- 哪些部分已稳定，哪些仍待核验

---

## 4. 标准结构

### 4.1 小型知识库

适用于单一主题、内容量不大时：

```text
主题目录/
├── README.md
└── 主题-主文档.md
```

### 4.2 中型知识库

适用于会持续扩展的主题：

```text
主题目录/
├── README.md
├── 主题-主文档.md
├── 基础概念/
├── 核心原理/
├── 工程实践/
└── 平台差异/
```

### 4.3 大型知识库

适用于内容持续增长、来源复杂、更新频繁时：

```text
主题目录/
├── README.md
├── 主题-主文档.md
├── 基础概念/
├── 核心原理/
├── 工程实践/
├── 平台差异/
├── 案例/
├── 附录/
└── sources/
```

说明：

- `sources/` 只在需要保留原始材料时使用
- 如果知识库已经高度规范，不必强制保留 `sources/`

---

## 5. 文件职责规范

### 5.1 `README.md`

`README.md` 只负责入口和导航，不负责承载大段知识正文。

应该包含：

- 主题简介
- 阅读顺序
- 按问题找文档
- 当前知识状态
- 维护规则
- 当前目录结构

不应该包含：

- 大段定义说明
- 详细平台差异
- 细碎案例展开

### 5.2 `主文档`

主文档负责主题总纲，是整个主题的稳定摘要。

应该包含：

- 文档目标
- 主题定义
- 核心原理
- 关键实践结论
- 子主题索引
- 重复内容合并说明
- 待核验项入口

不应该包含：

- 大量具体示例
- 频繁变化的平台细节
- 和子文档完全重复的展开内容

### 5.3 `子文档`

子文档只负责单一主题展开。

例如：

- 基础概念：定义、边界、术语
- 核心原理：机制、设计逻辑、为什么这样做
- 工程实践：步骤、规范、模板、维护方式
- 平台差异：不同平台、不同时期、不同版本的差别

### 5.4 `待核验项`

待核验项不是失败内容，而是尚未进入正式结论的暂存区。

适合放这里的内容：

- 来源之间存在冲突
- 平台版本可能变化
- 当前只有经验结论，没有稳定依据
- 需要后续补充官方口径

## 6. 命名规范

### 6.1 目录命名

目录名优先使用稳定分类词，不使用模糊名称。

推荐：

- `基础概念`
- `核心原理`
- `工程实践`
- `平台差异`
- `案例`
- `附录`

避免：

- `其他`
- `杂项`
- `补充1`
- `临时`

### 6.2 文件命名

推荐统一格式：

- `主题-主文档.md`
- `主题-基础概念.md`
- `主题-核心原理.md`
- `主题-工程实践.md`
- `主题-平台差异.md`
- `主题-案例汇总.md`

命名原则：

- 文件名能直接看出主题和职责
- 不频繁重命名
- 不混用多套命名风格

---

## 7. 文档状态规范

建议在 `README.md` 或索引页中维护状态表。

推荐状态：

- `已稳定`
- `待补充`
- `待核验`
- `已废弃`

定义如下：

- `已稳定`：可作为当前正式结论使用
- `待补充`：方向正确，但内容尚不完整
- `待核验`：暂不能写成正式结论
- `已废弃`：保留历史意义，但不应继续作为现行依据

## 8. 知识库构建流程

### 8.1 新建主题

当你要从零开始构建一个主题时，按这个顺序操作：

1. 明确主题名称
2. 收集原始材料
3. 识别重复、冲突和层级
4. 确定知识库规模
5. 搭建目录结构
6. 先写 `README.md`
7. 再写主文档
8. 最后拆出子文档

判断知识库规模的简单规则：

- 内容少：`README + 主文档`
- 内容中等：加 `基础概念 / 工程实践 / 平台差异`
- 内容大：再加 `核心原理 / 案例 / 附录 / sources`

### 8.2 新增材料

新增材料时，不要先问“要不要重写整套文档”，而要先问：

- 这份材料属于哪个已有主题
- 它属于哪个层级
- 它会影响哪个最小文件

推荐流程：

1. 判断材料归属
2. 判断材料类型
3. 只更新对应文档
4. 如影响整体认知，再同步主文档
5. 如只是导航变化，再更新 `README`

### 8.3 修订现有结论

如果新增材料不是新增主题，而是修正旧结论，按这个流程：

1. 找到原结论所在文档
2. 判断是局部修订还是全局影响
3. 优先局部修订
4. 只有影响总纲时才改主文档
5. 如果修订改变阅读方式，再改 `README`

### 8.4 待核验项转正

把待核验内容升级为正式结论时，建议这样做：

1. 核实依据是否足够稳定
2. 将结论写入对应正式文档
3. 从待核验区删除或标记已转正
4. 如该结论影响主题总纲，再同步更新主文档

---

## 9. 更新判断规则

后续维护时，可直接用下表判断该改哪里：

| 新内容类型 | 优先修改位置 |
|-----------|-------------|
| 定义、术语、边界 | 基础概念 |
| 机制、设计逻辑、原理说明 | 核心原理 |
| 目录、模板、方法、流程 | 工程实践 |
| 平台特性、版本差异、产品行为 | 平台差异 |
| 影响整体认知的稳定结论 | 主文档 |
| 阅读顺序、入口、状态变化 | README |
| 尚未确认的信息 | 待核验区 |

## 10. Prompt 驱动的使用规范

### 10.1 新建知识库 prompt

```text
请围绕主题《XXX》构建一套适合长期维护的知识库。

目标：
1. 识别该主题的基础概念、核心原理、工程实践、平台差异、待核验项
2. 设计适合该主题规模的目录结构
3. 生成 README、主文档，以及必要的子文档
4. 保证各文档职责清晰，不重复堆叠
5. 对冲突内容和不确定内容单独标注，不写成稳定结论

要求：
- README 只负责导航、阅读顺序、状态和维护规则
- 主文档只负责总纲、稳定结论、子主题索引
- 子文档按单一主题展开
- 保持命名统一，适合长期维护
- 如果主题规模不大，不要过度拆分
```

### 10.2 更新现有知识库 prompt

```text
请基于新增材料更新现有知识库，不要重写整套文档。

更新目标：
1. 判断新增内容属于哪个现有文件
2. 只更新最小必要文件
3. 保持原有目录结构和命名稳定
4. 将稳定结论写入正文
5. 将不确定、版本相关、来源冲突内容写入待核验区或平台差异区
6. 不要把相同内容重复写入 README、主文档和子文档

输出要求：
- 先说明本次应该更新哪些文件，为什么
- 再输出更新内容
- 如果某些文件无需修改，明确说明“不需要修改”
```

### 10.3 维护优化 prompt

```text
请检查当前知识库的结构是否适合长期维护。

检查目标：
1. 是否存在 README 与主文档职责重叠
2. 是否存在子文档之间内容重复
3. 是否存在应拆分但尚未拆分的主题
4. 是否存在不应写死却被写成正式结论的内容
5. 是否存在可以转正的待核验项

输出要求：
- 先给出结构诊断
- 再给出最小修改建议
- 非必要不要建议重构整个目录
```

## 11. 维护操作建议

### 11.1 日常新增

适合当你看到新资料、想到新结论时使用。

操作原则：

- 不立刻改主文档
- 先找最贴近的子文档
- 如无法归类，再放入待核验区

### 11.2 周期性整理

适合每隔一段时间做一次。

建议检查：

- `README` 的导航是否仍准确
- 主文档是否被塞入过多细节
- 子文档之间是否出现重复
- 平台差异是否混入稳定结论
- 待核验项是否能转正或删除

### 11.3 大版本调整

只有在以下情况才建议调整目录结构：

- 主题规模明显扩大
- 单个子文档过长
- 子主题边界已经清晰但仍混在一起
- 现有结构已明显影响检索与维护

大版本调整时应遵循：

- 优先增量拆分
- 尽量不改已有文件名
- 先拆子文档，再改主文档索引，再改 README

---

## 12. 常见错误

### 12.1 把所有内容都写进主文档

后果：

- 主文档越来越长
- 后续更新成本快速上升
- 读者很难判断哪里是总纲，哪里是细节

### 12.2 让 README 变成第二份主文档

后果：

- 导航页与总纲页重复
- 更新时容易两边不同步

### 12.3 每次更新都重写整套知识库

后果：

- 结构不断漂移
- 文件命名和目录反复变化
- 历史结论难以追踪

### 12.4 把待核验内容写成正式结论

后果：

- 知识可信度下降
- 后续纠错成本高

### 12.5 过度拆分

后果：

- 文件太多，导航成本上升
- 读者需要频繁跳转

---

## 13. 最小可执行规则

如果你不想记太多，只记住这 8 条：

1. 先建 `README`，再建主文档，最后建子文档。
2. `README` 只做导航，不讲大段正文。
3. 主文档只写总纲和稳定结论。
4. 子文档按单一主题展开。
5. 新资料优先更新最小必要文件。
6. 平台差异和版本细节不要污染主文档。
7. 不确定内容先放待核验区。
8. 非必要不要重构目录和改文件名。

---

## 14. 推荐使用方式

如果你希望把这份规范真正用起来，建议这样操作：

### 场景一：创建新主题

直接把“新建知识库 prompt”发给 AI，并附上你的原始材料。

### 场景二：给已有知识库新增内容

直接把“更新现有知识库 prompt”发给 AI，并附上新增材料与现有目录。

### 场景三：定期检查知识库是否失控

直接把“维护优化 prompt”发给 AI，让它只给最小修改建议。

### 场景四：你自己做人工判断

先问自己三个问题：

1. 这是稳定结论，还是待核验信息
2. 这属于总纲，还是属于某个子主题
3. 这次更新是否真的需要改目录

如果这三个问题能答清，知识库通常就不会失控。
