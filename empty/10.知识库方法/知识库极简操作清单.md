# 知识库极简操作清单

> 定位：这是 [知识库构建与维护规范.md](/Users/admin/Downloads/Code/empty/skill开发规范/知识库构建与维护规范.md) 的日常执行版，用于快速判断“现在该怎么做”。

---

## 1. 先判断你现在属于哪种场景

只需要先回答一个问题：

### 场景 A：我要新建一个主题

适用情况：

- 这是一个全新的知识主题
- 还没有目录和文档结构

你该做的事：

1. 判断主题规模
2. 建目录
3. 建 `README.md`
4. 建主文档
5. 按需建子文档

你该用的 prompt：

- 使用“新建知识库 prompt”

### 场景 B：我拿到了新的材料

适用情况：

- 不是新主题
- 是对现有主题的补充

你该做的事：

1. 先判断这份材料属于哪个主题
2. 再判断它属于哪个层级
3. 只更新最小必要文件

你该用的 prompt：

- 使用“更新现有知识库 prompt”

### 场景 C：我发现旧结论需要修正

适用情况：

- 旧文档里有错误
- 旧文档不完整
- 旧文档表述已经不准确

你该做的事：

1. 找到原结论所在文件
2. 局部修订
3. 只有影响整体认知时才同步主文档
4. 只有影响导航时才同步 `README`

你该用的 prompt：

- 使用“更新现有知识库 prompt”

### 场景 D：我想整理这套知识库是否已经混乱

适用情况：

- 文件开始变多
- 内容开始重复
- 不知道是否该拆目录
- 不确定某些内容是不是写错位置了

你该做的事：

1. 检查 `README` 是否还只是入口
2. 检查主文档是否塞入过多细节
3. 检查子文档之间是否重复
4. 检查待核验项是否能转正

你该用的 prompt：

- 使用“维护优化 prompt”

## 2. 新内容来了，先判断该改哪里

| 新内容类型 | 优先修改位置 |
|-----------|-------------|
| 定义、术语、边界 | 基础概念 |
| 原理、机制、为什么这样设计 | 核心原理 |
| 方法、流程、模板、目录规范 | 工程实践 |
| 平台差异、版本差异、产品行为 | 平台差异 |
| 影响整体认知的稳定结论 | 主文档 |
| 阅读顺序、入口、状态变化 | README |
| 暂时无法确认的内容 | 待核验区 |

如果判断不出来，默认先放：

- 子文档，而不是主文档
- 待核验区，而不是正式结论

---

## 3. README、主文档、子文档怎么分

### README

只做这些：

- 入口说明
- 阅读顺序
- 按问题找文档
- 当前知识状态
- 维护规则

不要做这些：

- 写长篇正文
- 写详细平台差异
- 写大量具体案例

### 主文档

只做这些：

- 主题总纲
- 稳定结论
- 核心认知
- 子主题索引
- 待核验入口

不要做这些：

- 承担所有细节
- 重复子文档正文
- 写大量版本细节

### 子文档

只做这些：

- 展开单一主题
- 承载细节
- 承载方法
- 承载差异说明

## 4. 每次维护时的最小动作

你每次更新时，按这个顺序判断：

1. 这是新主题还是旧主题补充
2. 这是稳定结论还是待核验内容
3. 它属于哪个子主题
4. 是否真的影响主文档
5. 是否真的影响 `README`

只要前 3 步能答清，通常就不会失控。

---

## 5. 什么情况下才改主文档

只有符合以下情况，才建议改主文档：

- 新增内容改变了对主题的整体理解
- 某条旧结论已不再成立
- 某个子主题已经成为核心组成部分
- 待核验内容已经转正，且影响总纲

其他情况下，优先改子文档。

---

## 6. 什么情况下才改 README

只有符合以下情况，才建议改 `README.md`：

- 阅读顺序改变了
- 目录结构改变了
- 某个模块状态变化了
- 新增了重要子文档，需要加入导航

其他情况下，不要动 `README`。

---

## 7. 什么情况下才改目录结构

只有符合以下情况，才建议改目录结构：

- 单个文档已经明显过长
- 一个子文档中混入了多个独立主题
- 当前结构已经影响检索和维护
- 新增内容是长期存在的稳定大类

不符合这些条件时，不要为了“看起来更完整”而新增目录。

---

## 8. 哪些事不要做

### 不要每次都重写整套知识库

正确做法：

- 只更新最小必要文件

### 不要让 README 变成第二份主文档

正确做法：

- README 只保留导航和状态

### 不要把平台细节写进总纲

正确做法：

- 平台差异放平台差异文档

### 不要把不确定内容直接写成正式结论

正确做法：

- 先写到待核验区

### 不要轻易改文件名和目录名

正确做法：

- 保持结构稳定，优先增量维护

## 9. 三条最常用 prompt

### 9.1 新建主题

```text
请围绕主题《XXX》构建一套适合长期维护的知识库。

目标：
1. 识别该主题的基础概念、核心原理、工程实践、平台差异、待核验项
2. 设计适合该主题规模的目录结构
3. 生成 README、主文档，以及必要的子文档
4. 保证各文档职责清晰，不重复堆叠
5. 对冲突内容和不确定内容单独标注，不写成稳定结论

要求：
- README 只负责导航、阅读顺序、状态和维护规则
- 主文档只负责总纲、稳定结论、子主题索引
- 子文档按单一主题展开
- 保持命名统一，适合长期维护
- 如果主题规模不大，不要过度拆分
```

### 9.2 更新现有主题

```text
请基于新增材料更新现有知识库，不要重写整套文档。

更新目标：
1. 判断新增内容属于哪个现有文件
2. 只更新最小必要文件
3. 保持原有目录结构和命名稳定
4. 将稳定结论写入正文
5. 将不确定、版本相关、来源冲突内容写入待核验区或平台差异区
6. 不要把相同内容重复写入 README、主文档和子文档

输出要求：
- 先说明本次应该更新哪些文件，为什么
- 再输出更新内容
- 如果某些文件无需修改，明确说明“不需要修改”
```

### 9.3 检查结构是否失控

```text
请检查当前知识库的结构是否适合长期维护。

检查目标：
1. 是否存在 README 与主文档职责重叠
2. 是否存在子文档之间内容重复
3. 是否存在应拆分但尚未拆分的主题
4. 是否存在不应写死却被写成正式结论的内容
5. 是否存在可以转正的待核验项

输出要求：
- 先给出结构诊断
- 再给出最小修改建议
- 非必要不要建议重构整个目录
```

---

## 10. 最后只记住这 10 句

1. 新主题先建 `README`，再建主文档，最后建子文档。
2. `README` 只做导航，不写长正文。
3. 主文档只写总纲和稳定结论。
4. 子文档只负责单一主题展开。
5. 新材料先判断归属，再判断层级。
6. 优先更新最小必要文件。
7. 不确定内容先放待核验区。
8. 平台差异不要污染主文档。
9. 非必要不要改目录和文件名。
10. 非必要不要让 AI 重写整套知识库。
