# Skill 平台差异对照：Claude / Codex / Copilot / Cursor

## 1. 先看结论

当前更稳妥的判断是：

- Claude / Codex / Copilot 的部分 Skill 实践围绕 `SKILL.md` 目录包展开
- Cursor 使用独立的规则体系，核心载体通常是 `.cursor/rules/*.mdc` 或 `AGENTS.md`
- 因此，`SKILL.md` 不应直接等同于 Cursor Rule

## 2. 适合长期维护的跨平台策略

推荐采用“主源 + 派生”的组织方式：

1. 以 `SKILL.md` 相关内容维护主源规范
2. 对支持相同结构的平台尽量直接复用
3. 对 Cursor 单独派生 `.mdc` 或 `AGENTS.md`

这样做的好处是：

- 主体知识只维护一份
- 平台差异集中管理
- 不会把不同规范体系混写在一个文件里

## 3. 平台差异表

| 维度 | `SKILL.md` 体系 | Cursor 规则体系 |
|------|-----------------|----------------|
| 核心载体 | `SKILL.md` | `.cursor/rules/*.mdc` 或 `AGENTS.md` |
| 关注点 | 任务流程封装 | 规则注入与规则匹配 |
| 复用方式 | 适合围绕主源文件复用 | 更适合作为平台派生层 |
| 维护建议 | 作为知识主干维护 | 作为适配文件维护 |

## 4. 当前不应写死的内容

以下内容容易随版本变化，建议单独核实后再进入正式规范：

- 各平台的最新安装路径
- 各平台的扫描范围和作用域
- `/skills`、`$skill` 等显式调用入口
- `agents/openai.yaml` 在各平台中的实际行为

## 5. 实务建议

如果目标是长期维护，而不是单次迁移，推荐采用这个原则：

- 主规则写在可抽象、可复用的主源文档里
- 平台差异只保留在平台目录中
- 不把平台特有字段反向污染到总纲文档中
