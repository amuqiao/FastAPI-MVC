## 合并后的正式版本

### 一、基础概念

Skill 是一类任务的可复用工作流包。它不是新模型，也不是独立工具，而是把高频、相对稳定、但仍需要模型理解的任务过程整理为可反复调用的标准说明。

与一次性 prompt 相比，Skill 的关键不在于更长，而在于：

- 有清晰触发边界
- 有稳定工作流结构
- 可以附带参考资料、脚本和模板
- 可以在多个相似任务中重复使用

一个实用类比是：

| 概念 | 可理解为 |
|------|----------|
| 普通提示词 | 一次性交代 |
| Skill | 标准作业说明书 |
| `scripts/` | 说明书附带的小工具 |
| `references/` | 说明书引用的背景资料 |
| `assets/` | 产物直接会用到的模板和样例 |

### 二、核心原理

Skill 的核心价值是把高频任务中的稳定部分沉淀下来，把不稳定、需要理解和判断的部分继续交给模型。

一份可维护的 Skill 通常具备这些特征：

- 聚焦单一任务或单一任务簇
- 输入输出边界明确
- 步骤有顺序，但不过度依赖特定上下文
- 机械步骤与判断步骤分层处理

Skill 的加载逻辑适合按三层理解：

1. 先读取轻量元信息，用于发现和匹配
2. 命中后再加载完整 `SKILL.md`
3. 必要时再读取 `references/`、执行 `scripts/`、使用 `assets/`

### 三、工程实践

长期维护时，建议遵循“先轻量、后升级”的策略。

最小形态：

```text
my-skill/
└── SKILL.md
```

扩展形态：

```text
my-skill/
├── SKILL.md
├── references/
├── scripts/
├── assets/
└── agents/
    └── openai.yaml
```

是否升级到生产级，主要看四个信号：

- `SKILL.md` 已经过长
- 需要拆出长规格或清单到 `references/`
- 出现稳定可执行流程，适合写入 `scripts/`
- 需要复用模板和样例，适合沉到 `assets/`

### 四、扩展讨论

跨平台时，最重要的不是记住所有细节，而是先判断自己在写哪一类规范：

- 如果目标是 Skill 目录包，优先围绕 `SKILL.md`
- 如果目标是 Cursor 规则，优先围绕 `.cursor/rules/*.mdc` 或 `AGENTS.md`

当前更稳妥的维护策略是：

1. 以 `SKILL.md` 体系维护主源内容
2. 对支持同类结构的平台尽量复用
3. 若需要支持 Cursor，再派生 `.mdc` 规则文件

### 五、冲突与待确认项

以下内容不宜在主文档中写成绝对结论，建议作为版本相关事项继续核实：

- Codex CLI、ChatGPT、Copilot CLI 的准确安装路径和作用域
- `agents/openai.yaml` 的真实职责边界
- `/skills`、`$skill` 等显式调用方式是否为通用能力
- 平台发布日期、兼容数量、互通范围等事实性陈述

## 重复内容合并说明

本主题原始笔记中的高频重复点，已统一收敛为以下稳定结论：

1. `SKILL.md` 是 Skill 的核心入口
2. Skill 的本质是可复用工作流包
3. `description` 更接近路由规则，而不是宣传文案
4. 轻量级优先，生产级按需升级
5. `references/`、`scripts/`、`assets/` 的职责必须明确区分
6. Cursor 规则与 `SKILL.md` 体系相关，但并非同一套规范
