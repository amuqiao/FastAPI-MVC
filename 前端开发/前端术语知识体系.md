## 一、语言基础

### JavaScript (JS)

- **定义**：浏览器和 Node.js 的原生脚本语言，动态类型，基于原型继承
- **文件后缀**：`.js`
- **现代特性（ES6+）**：箭头函数、`async/await`、模块 `import/export`、解构赋值、可选链 `?.`
- **运行环境**：浏览器 / Node.js / Deno / Bun
- **关键点**：所有前端生态的根基，TypeScript 是其超集

---

### TypeScript (TS)

- **定义**：微软出品，JavaScript 的静态类型超集，编译后输出纯 JS
- **文件后缀**：`.ts`
- **核心价值**：编译时类型检查、IDE 智能提示、大型项目可维护性
- **重要概念**：
  - `interface` / `type`：定义类型结构
  - `enum`：枚举
  - `generics <T>`：泛型
  - `as` / `satisfies`：类型断言
- **注意**：TS 不改变运行时行为，只在开发阶段检查类型
- **编译器**：`tsc`（官方，做类型检查）/ esbuild / SWC（只转译，不检查类型）

---

### JSX

- **定义**：JavaScript XML，允许在 JS 中写类 HTML 的模板语法
- **文件后缀**：`.jsx`
- **本质**：语法糖，`<div>Hello</div>` 会被编译为 `React.createElement('div', null, 'Hello')`
- **与 HTML 的区别**：
  - 属性用 camelCase：`className` 而非 `class`，`onClick` 而非 `onclick`
  - 必须有根元素，或用 `<>...</>` Fragment 包裹
  - `{表达式}` 插入动态内容
- **需要转译**：Babel / esbuild / Vite 均支持

---

### TSX

- **定义**：TypeScript + JSX 的组合
- **文件后缀**：`.tsx`
- **用途**：React + TypeScript 项目的标准文件格式
- **示例**：

```tsx
interface Props {
  name: string;
  age?: number;
}

const UserCard = ({ name, age }: Props) => (
  <div className="card">
    <h2>{name}</h2>
    {age && <p>{age} 岁</p>}
  </div>
);
```

## 二、结构与样式

### HTML

- **定义**：HyperText Markup Language，网页的骨架，定义语义结构
- **文件后缀**：`.html`
- **HTML5 语义标签**：`<header>` `<main>` `<article>` `<section>` `<footer>` `<nav>`
- **与 JSX 关系**：React/Vue 组件最终渲染为真实 DOM（HTML）
- **重要概念**：DOM（文档对象模型）、虚拟 DOM（React/Vue 的优化中间层）

---

### CSS

- **定义**：Cascading Style Sheets，控制视觉呈现
- **文件后缀**：`.css`
- **现代核心能力**：
  - Flexbox：一维布局
  - Grid：二维布局
  - CSS Variables（`--color-primary`）
  - 媒体查询 `@media`：响应式设计
  - 动画 `@keyframes`
- **现代方案**：
  - **Sass/SCSS**：CSS 预处理器，支持变量、嵌套、混入
  - **PostCSS**：CSS 的"Babel"，Tailwind / autoprefixer 都基于它
  - **CSS Modules**：作用域隔离，避免类名冲突（`.module.css`）
  - **CSS-in-JS**：如 styled-components，在 JS 中写 CSS
  - **Tailwind CSS**：原子化 CSS 框架（见框架章节）

---

### Tailwind CSS

- **定义**：原子化 CSS 框架，通过组合小工具类构建 UI，无需手写 CSS 文件
- **核心理念**：`utility-first`，每个类只做一件事
- **示例**：`<div class="flex items-center gap-4 p-6 text-gray-600 rounded-lg">`
- **优点**：构建时 tree-shake 未使用类，生成极小 CSS 包；设计一致性强
- **配置文件**：`tailwind.config.js` / `tailwind.config.ts`
- **搭配**：PostCSS、shadcn/ui、Headless UI、Radix UI

---

## 三、框架与库 — React 系

### React

- **定义**：Meta 出品的 UI 库（不是完整框架），专注于组件化渲染
- **核心概念**：
  - **组件**：函数组件（主流）或类组件
  - **虚拟 DOM**：React 维护一棵虚拟树，diff 后最小化真实 DOM 操作
  - **单向数据流**：数据从父组件流向子组件（props）
- **核心 Hooks**：
  - `useState`：局部状态
  - `useEffect`：副作用（数据请求、订阅）
  - `useContext`：跨层级传值
  - `useRef`：引用 DOM / 持久化值
  - `useMemo` / `useCallback`：性能优化
- **生态**：
  - 路由：React Router / TanStack Router
  - 状态管理：Zustand / Redux Toolkit / Jotai
  - 数据请求：TanStack Query / SWR
  - 组件库：shadcn/ui / Ant Design / MUI

---

### Next.js

- **定义**：基于 React 的全栈框架，Vercel 出品
- **核心能力**：
  - **文件路由**：`app/` 目录下的文件自动成为路由
  - **SSR**（服务端渲染）：每次请求在服务端渲染 HTML
  - **SSG**（静态生成）：构建时生成 HTML，性能最佳
  - **ISR**（增量静态再生）：静态页面按需重新生成
  - **API Routes**：在同一项目中写后端接口
  - **Server Components**：默认在服务端运行，减少客户端 JS
- **配置文件**：`next.config.js` / `next.config.ts`
- **部署**：Vercel（一键）/ 任意 Node.js 环境 / Docker
- **竞品**：Remix、Astro

---

## 四、框架与库 — Vue 系

### Vue

- **定义**：尤雨溪（Evan You）出品的渐进式 UI 框架
- **当前版本**：Vue 3（主流），Vue 2 已于 2023 年底停止维护
- **两种 API 风格**：

**Options API**（Vue 2 风格，简单直观）：
```js
export default {
  data() { return { count: 0 } },
  methods: {
    increment() { this.count++ }
  }
}
```

**Composition API**（Vue 3 推荐，与 React Hooks 理念相似）：
```vue
<script setup>
import { ref } from 'vue'
const count = ref(0)
const increment = () => count.value++
</script>
```

- **核心特性**：
  - 响应式系统：`ref()` / `reactive()` 自动追踪依赖
  - 模板语法：`v-if` / `v-for` / `v-model` / `v-bind` / `v-on`
  - 单文件组件（SFC）：`.vue` 文件，`<template>` + `<script>` + `<style>` 三段式
  - `<script setup>`：Vue 3 语法糖，简化 Composition API 写法
- **与 React 对比**：
  - Vue 模板语法更接近 HTML，上手门槛较低
  - Vue 有双向绑定 `v-model`，React 需手动处理
  - Vue 官方提供路由（Vue Router）和状态管理（Pinia），React 依赖社区生态

---

### Vue Router

- **定义**：Vue 官方路由库
- **核心概念**：
  - `<RouterView>`：渲染当前路由对应的组件
  - `<RouterLink>`：声明式导航，类似 `<a>` 标签
  - 动态路由：`/users/:id`
  - 路由守卫：`beforeEach` 做权限控制
- **使用方式**：

```js
const routes = [
  { path: '/', component: Home },
  { path: '/about', component: About },
  { path: '/users/:id', component: UserDetail }
]
```

---

### Pinia

- **定义**：Vue 官方推荐的状态管理库，Vue 3 时代替代 Vuex
- **核心优势**：比 Vuex 更简洁，完整 TypeScript 支持，DevTools 集成
- **三大概念**：`state`（数据）、`getters`（计算值）、`actions`（方法）
- **示例**：

```ts
export const useUserStore = defineStore('user', {
  state: () => ({ name: '', isLoggedIn: false }),
  getters: {
    displayName: (state) => state.name || '游客'
  },
  actions: {
    login(name: string) { this.name = name; this.isLoggedIn = true }
  }
})
```

---

### Nuxt.js

- **定义**：基于 Vue 的全栈框架，相当于 Vue 生态的 Next.js
- **当前版本**：Nuxt 3（基于 Vue 3 + Vite + TypeScript）
- **核心能力**：
  - **文件路由**：`pages/` 目录自动生成路由
  - **SSR / SSG / SPA**：三种渲染模式灵活切换
  - **自动导入**：组件、Composables、工具函数无需手动 import
  - **Server Routes**：`server/api/` 目录下自动成为 API 接口
  - **Nitro**：Nuxt 3 的服务器引擎，支持部署到 Vercel / Netlify / Node / Edge
  - **Layers**：可复用的 Nuxt 配置层，适合多项目共享代码
- **配置文件**：`nuxt.config.ts`
- **目录结构**：

```
├── pages/          # 文件路由
├── components/     # 自动导入的组件
├── composables/    # 自动导入的组合式函数
├── server/
│   └── api/        # 服务端 API 接口
├── public/         # 静态资源
└── nuxt.config.ts
```

- **与 Next.js 对比**：

| 对比项 | Nuxt 3 | Next.js 14 |
|--------|--------|------------|
| 基础框架 | Vue 3 | React |
| 服务器引擎 | Nitro | Node.js |
| 自动导入 | ✅ 内置 | ❌ 需手动 |
| 文件路由 | ✅ | ✅ |
| 边缘部署 | ✅ Nitro | ✅ Vercel Edge |
| 学习曲线 | 较低 | 中等 |

---

### Nuxt 核心 Composables

Nuxt 3 内置了一套自动导入的组合式函数：

- `useFetch(url)`：SSR 友好的数据请求，服务端和客户端都能运行
- `useAsyncData(key, fn)`：更底层的异步数据获取
- `useState(key, init)`：跨组件共享的 SSR 安全状态
- `useRoute()`：获取当前路由信息
- `useRouter()`：编程式导航
- `useCookie(name)`：读写 Cookie，SSR 兼容
- `useRuntimeConfig()`：读取运行时环境变量

## 五、构建工具

### Vite

- **定义**：尤雨溪出品的新一代前端构建工具
- **开发模式**：利用浏览器原生 ES Modules，按需编译，冷启动极快（毫秒级）
- **生产构建**：基于 Rollup 打包，输出优化的静态资源
- **配置文件**：`vite.config.ts`
- **已取代**：Create React App（CRA）
- **插件生态**：`@vitejs/plugin-react`、`@vitejs/plugin-vue` 等
- **内部使用**：esbuild（依赖预构建）+ Rollup（生产打包）

---

### Webpack

- **定义**：老牌模块打包器，将 JS / CSS / 图片等所有资源视为模块
- **核心概念**：Loader（处理非 JS 文件）、Plugin（扩展功能）、Code Splitting
- **配置文件**：`webpack.config.js`
- **现状**：配置复杂，启动慢，但生态完善；大量遗留项目仍在使用
- **竞品**：Vite / Rollup / Parcel / Turbopack

---

### Babel

- **定义**：JavaScript 编译器，将 ES6+ / JSX / TS 转换为兼容旧浏览器的 JS
- **配置文件**：`babel.config.js` / `.babelrc`
- **现状**：在 Webpack 生态中仍广泛使用（`babel-loader`）；Vite 项目中已被 esbuild 替代
- **被替代趋势**：esbuild（Go 实现）和 SWC（Rust 实现）速度快 10-100 倍

---

### esbuild

- **定义**：Go 语言编写的极速 JS 打包/转译工具
- **速度**：比 Babel + Webpack 快 10-100 倍
- **使用场景**：Vite 用它做依赖预构建；也可独立作为构建工具
- **注意**：支持 TS / JSX 转译，但**不做类型检查**（类型检查仍需 `tsc`）

---

### Rollup

- **定义**：专为库打包设计的模块打包器
- **特点**：输出体积小，Tree-shaking 能力强
- **使用场景**：Vite 生产构建的底层；开发 npm 库时首选

---

### SWC

- **定义**：Rust 编写的超快 JS/TS 编译器
- **使用场景**：Next.js 13+ 用它替代 Babel（速度提升 17 倍）
- **功能**：转译 TS / JSX，不做类型检查

---

### Turbopack

- **定义**：Vercel 用 Rust 编写的下一代打包器，Webpack 的继任者
- **使用场景**：Next.js 13+ 内置（开发模式），正逐步成熟
- **目标**：在超大型项目中比 Vite 更快

---

## 六、包管理器

> 三者功能相同，都是"安装和管理项目依赖"，区别在于速度、磁盘占用和特性。

### npm

- **定义**：Node Package Manager，随 Node.js 一起安装，最早的 JS 包管理器
- **核心文件**：
  - `package.json`：项目元数据和依赖声明
  - `package-lock.json`：锁定精确版本，保证团队一致
  - `node_modules/`：实际安装的包
- **常用命令**：
  ```bash
  npm install          # 安装所有依赖
  npm install react    # 安装单个包
  npm install -D vite  # 安装开发依赖
  npm run dev          # 运行 package.json 中的 dev 脚本
  npm publish          # 发布包到 npmjs.com
  ```

---

### pnpm

- **定义**：高性能 npm 替代品
- **核心优势**：
  - 全局内容寻址存储（硬链接）：相同版本的包只存一份，节省大量磁盘空间
  - 安装速度比 npm / Yarn 更快
  - 默认阻止"幽灵依赖"（phantom dependencies）
  - Monorepo 支持极佳（`pnpm workspaces`）
- **锁文件**：`pnpm-lock.yaml`
- **新项目推荐选择**

---

### Yarn

- **定义**：Facebook 出品，解决早期 npm 速度慢和一致性问题
- **两个版本**：
  - Yarn Classic（v1）：与 npm 类似，仍广泛使用
  - Yarn Berry（v2+）：引入 Plug'n'Play（PnP），彻底去掉 `node_modules`，改用 `.pnp.cjs`
- **锁文件**：`yarn.lock`
- **常用命令**：`yarn add react` / `yarn dev`

---

## 七、运行时与环境

### Node.js

- **定义**：基于 V8 引擎的服务端 JavaScript 运行时
- **核心作用**：让 JS 脱离浏览器运行（前端工具链全运行在 Node 上）
- **版本策略**：偶数版本为 LTS（长期支持），如 Node 18、20、22
- **内置模块**：`fs`（文件系统）、`path`、`http`、`crypto` 等
- **后端框架**：Express / Fastify / NestJS / Hono
- **版本管理**：nvm / fnm / Volta

---

### nvm

- **定义**：Node Version Manager，在同一台机器上管理多个 Node.js 版本
- **常用命令**：
  ```bash
  nvm install 20          # 安装 Node 20
  nvm use 20              # 切换到 Node 20
  nvm alias default 20    # 设置默认版本
  nvm ls                  # 列出已安装版本
  ```
- **Windows**：使用 nvm-windows 或 fnm
- **更快替代**：fnm（Rust 实现）

---

### fnm

- **定义**：Fast Node Manager，Rust 编写，比 nvm 更快，跨平台
- **特点**：支持 `.nvmrc` / `.node-version` 自动切换版本
- **推荐场景**：新机器配置，Windows 用户

---

### Deno

- **定义**：Ryan Dahl（Node.js 原作者）重新设计的 JS/TS 运行时
- **特点**：
  - 原生支持 TypeScript，无需配置
  - 默认安全（需显式授权访问文件/网络）
  - 内置测试、格式化、lint 工具
  - Deno 2 已与 npm 生态兼容
- **云平台**：Deno Deploy

---

### Bun

- **定义**：用 Zig 编写的全能 JS 工具，集运行时 + 包管理器 + 构建工具于一身
- **速度**：运行时比 Node 快 3-5×，包安装比 npm 快 25×
- **兼容**：Node.js API 和 npm 生态
- **内置**：`bun install`（包管理）/ `bun build`（打包）/ `bun test`（测试）
- **现状**：最受关注的 Node.js 替代品，生产环境正逐步成熟

---

| **SFC** | Single File Component，Vue 的 `.vue` 单文件组件 |
| **Nitro** | Nuxt 3 的服务器引擎，支持多平台部署 |
| **Turbopack** | Vercel 用 Rust 写的新一代打包器 |
| **SWC** | Rust 编写的 JS/TS 编译器，Next.js 使用 |
| **Vitest** | 基于 Vite 的单元测试框架，与 Jest 兼容 |
| **Playwright** | 现代端到端（E2E）测试框架 |
| **Storybook** | 组件隔离开发和文档工具 |

## 十、常见混淆点

### JS vs TS
TypeScript 只在开发阶段存在，编译后全是 JS。**运行时没有 TypeScript**，Node.js 和浏览器都不认识 `.ts` 文件。

### JSX vs HTML
JSX 长得像 HTML，但本质是 JS 函数调用。属性用 camelCase（`className`），可以插入 JS 表达式（`{变量}`），必须有根元素。

### React vs Next.js
React 是 UI 库，只管组件渲染。Next.js 是基于 React 的完整框架，加了路由、SSR、API、打包配置等一切。**学 React 是基础，Next.js 是生产方案。**

### Vue vs Nuxt
同上。Vue 是 UI 框架，Nuxt 是基于 Vue 的完整全栈框架。

### npm vs pnpm vs yarn
功能相同，都是"安装依赖"。**新项目推荐 pnpm**：更快、节省磁盘、防幽灵依赖。

### Vite vs Webpack
Vite 是新时代选择，开发启动极快（毫秒级）。Webpack 老牌强大，大量遗留项目在用。**新项目用 Vite，维护老项目才用 Webpack**。

### esbuild / SWC vs tsc
esbuild 和 SWC 只做**转译**（速度极快），不检查类型。`tsc` 才做**类型检查**（较慢）。在 CI 中通常两者都跑：先 `tsc --noEmit` 检查类型，再用 esbuild/SWC 打包。

### SSR vs SSG vs CSR
- **CSR**：浏览器下载空 HTML + JS，JS 运行后渲染内容。首屏慢，SEO 差。
- **SSR**：每次请求服务端实时渲染 HTML，首屏快，SEO 好，服务器压力大。
- **SSG**：构建时生成 HTML，性能最佳，适合内容不常变化的页面。
- **ISR**：SSG 的进阶版，可以设置页面定期重新生成。
