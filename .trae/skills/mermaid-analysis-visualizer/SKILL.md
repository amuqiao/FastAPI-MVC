# Mermaid Analysis Visualizer

## Purpose

这个 skill 用于分析驱动的 Mermaid 可视化。

它负责：
- 判断当前问题是否值得画图
- 当主图类型仍模糊时，先澄清图型
- 选择最合适的 Mermaid 图型
- 输出最小充分图集
- 给出简短分析、Mermaid 代码、图解说明
- 在输出前执行语法与表达自检

它不负责完整项目理解，也不负责非 Mermaid 图形工具。

## When To Use

以下情况使用本 skill：

- 用户明确要求 Mermaid、架构图、流程图、时序图、状态机图、甘特图、时间线图、思维导图或其他图示表达
- 任务是项目分析、流程梳理、架构设计、模块关系说明、异步链路分析、状态变化分析，且图能明显降低理解成本
- 用户已经指定图种，希望直接按规范输出

## When Not To Use

以下情况不应由本 skill 主导：

- 深度代码审计
- 完整架构评审
- 长篇技术报告撰写
- 信息严重不足且无法安全表达

若任务重点是理解代码库，应由分析类 skill 主导；本 skill 只负责可视化表达。

## Collaboration Boundary

与 `codebase-project-analysis` 等分析类 skill 的边界：

- 分析类 skill 负责读取代码、理解结构、提炼结论
- `mermaid-analysis-visualizer` 负责判断是否需要图、澄清主图类型、选择图型、输出 Mermaid 和说明

## Clarify Before Drawing

这个 skill 不默认做完整需求访谈，但当用户只说“画一张图”“画个图”“画个架构图”这类表述，仍需要先澄清主图类型，再出图。

默认只澄清“要哪一种主图”，不扩展为完整分析问答。

默认只在这些明显不同的主图类型之间做判断：

- `流程图`
- `技术架构图`
- `平台分层图`
- `分层能力结构图`

当主图类型不清时，优先只问一句：

1. 你更想看 `流程图`、`技术架构图`、`平台分层图`，还是 `分层能力结构图`？

如果用户没有补充信息，按以下默认值处理：

- 优先出一张最能建立共识的主图
- “事情怎么走”默认落到流程图
- “系统怎么组成”默认落到技术架构图
- “系统分几层、每层做什么”默认落到平台分层图
- “系统按阶段或能力域怎么分层展开”默认落到分层能力结构图

图内的表达重点由 skill 自动补足，不再让用户额外选择：

- 流程图默认突出主流程，必要时补关键分支
- 技术架构图默认突出关键服务协作
- 平台分层图默认突出职责边界和层间协作
- 分层能力结构图默认突出能力分组、阶段分层和主线推进

## Output Contract

默认按以下顺序输出：

1. 主图类型判断或澄清结果
2. 简短分析
3. Mermaid 代码
4. 图解说明
5. 补图提示（仅在必要时）

如果主图类型已经明确，可以省略“主图类型判断或澄清结果”这一项。

图解说明固定覆盖：
- 这张图回答什么问题
- 应该先看哪里
- 哪些节点或链路最关键
- 这张图没覆盖什么

## Diagram Routing

当用户没有说清主图类型时，先在以下四种主图之间判断：

### 流程图

适合回答：
- 创作、生成、审核、发布如何串起来
- 谁触发谁，前后顺序是什么
- 哪一步产出什么

默认优先图型：
- `flowchart LR`
- 必要时补一张 `sequenceDiagram`

### 系统技术架构图

适合回答：
- 系统由哪些服务和基础设施组成
- 服务之间如何调用、解耦和持久化
- 哪些组件处在关键路径上

默认优先图型：
- `flowchart TB`

### 平台分层架构图

适合回答：
- 平台通常分为哪些层
- 每层职责边界是什么
- 上下层如何协作

默认优先图型：
- `flowchart TB`

### 分层能力结构图

适合回答：
- 一个系统或方案按哪些阶段、能力域或生产环节分层展开
- 每层中有哪些并列能力块
- 层与层之间的主线如何自上而下推进

默认优先图型：
- `flowchart TB`

详细定义、与平台分层图/技术架构图的区别、以及标准示例，读取 `references/mermaid-A-系统认知层.md`。

只有当用户明确指定时，才直接进入其他图型，例如时序图、状态图、甘特图、时间线、脑图等；这些不作为默认澄清分支。

## Diagram Selection

只读取当前真正需要的 reference，不要默认全量加载。

- 系统结构、分层职责、同步流程：读取 `references/mermaid-A-系统认知层.md`
- 数据模型、类关系、模块内部设计：读取 `references/mermaid-B-代码深潜层.md`
- 异步链路、跨进程消息、状态变化：读取 `references/mermaid-C-运行时行为层.md`
- 项目排期、技术演进、主题展开：读取 `references/mermaid-D-规划可视化层.md`

需要总览导航时再读取 `references/mermaid-使用说明.md`。

读取 reference 后，不只复用“图种选择”，还要复用对应 reference 的视觉语法和表达习惯。

- A 类系统架构图 / 分层能力结构图 / 流程图：默认沿用 reference 中的 `classDef` 配色、`subgraph` 分层、`NOTE` 注记、`%% 边索引` 注释和 `linkStyle` 写法
- 若当前问题不需要完整样式，可简化节点数量，但不要脱离 reference 的整体风格另起一套视觉语言
- 生成面向具体主题的新图时，优先“替换领域节点”，不要“更换整套图面风格”

## Quantity Strategy

图数量由问题决定，遵循最小充分图集原则：

- 优先输出最能降低理解成本的那一张图
- 当两张图组合明显更合理时，可以输出两张
- 不为了完整而堆图
- 若存在更多有价值的视角，但当前信息不足或收益不高，只给补图提示

默认风格偏保守：
- 主输出只画最确定的图
- 额外价值通过补图提示说明

## Insufficient Information Handling

当关键信息不足时：

- 明确指出缺失信息
- 不把推测写成事实
- 不输出高风险误导图
- 如仍可安全表达，只输出高确定性的局部图
- 如不足以安全成图，只给分析和缺口提示

## Syntax Safety Rules

严格区分 Mermaid 语法边界：

- Flowchart 系列：架构图、流程图、数据模型关系图、类层级关系图
  - 常见语法：`flowchart TB`、`flowchart LR`、`classDef`、`subgraph`、`linkStyle`
- State / Sequence 系列：状态机图、时序图
  - 常见语法：`stateDiagram-v2`、`sequenceDiagram`
  - 不使用 `classDef`、`subgraph`、`linkStyle`
- Planning 系列：`gantt`、`timeline`、`mindmap`
  - 不套用 flowchart 风格规则
  - `mindmap` 换行使用 `<br/>`

## Self-Check

输出前检查：

- 图型是否匹配问题
- 图数量是否过多
- 是否混用了不兼容语法
- 节点命名是否过长或过密
- 关键关系是否表达准确
- 图解说明是否覆盖阅读重点
- 是否把不确定信息写成了确定事实
- 是否有内容更适合作为补图提示而不是直接输出
