# 资源目录说明

> 文档职责：定义 `9.资源/` 目录的用途、入口文件分工、拆分边界与维护规则。
> 适用场景：需要整理外部学习资料、开源项目、工具链接，或判断资源导航是否该拆分时使用。
> 阅读目标：快速判断资源应放在哪份文件、当前是否需要拆新文件、后续如何低成本增量维护。
> 目标读者：需要长期维护该知识库导航内容的作者，或第一次进入本目录的阅读者。

## 1. 当前目录定位

`9.资源/` 负责沉淀“站外资源入口”，而不是代替本仓库内部专题正文。

这里优先放：

- 官方文档、教程、论文、知识库、开源仓库、在线工具、社区入口
- 适合作为“继续阅读”或“外部检索起点”的资源

这里不优先放：

- 已经在本仓库中系统沉淀过的长篇主题正文
- 需要频繁更新细节的临时资讯流
- 纯粹一次性收藏、没有长期复用价值的链接

## 2. 当前最小可维护结构

现阶段保持三层分工最稳：

```text
9.资源/
├── README.md
├── AI 在线资源导航.md
└── GitHub开源项目导航.md
```

原因：

- 主题型主导航继续保留，整体阅读路径不变
- GitHub 项目已经形成相对稳定的独立维护需求，适合单独抽出
- GitHub 子导航仍按主题组织，不会破坏现有按主题查找的习惯

## 3. 现在是否要拆新文件

当前判断：

- `AI 在线资源导航.md`：保留，继续作为主入口
- `GitHub开源项目导航.md`：已拆出，独立维护
- `AI 学习教程导航.md`：暂不新建

当前只拆 GitHub 的原因：

- GitHub 仓库更适合形成独立索引，便于集中补充和筛选
- 但教程类资源仍和文档、论文、知识库强相关，暂时不值得单独拆
- 当前拆 GitHub 不会破坏主题维度，因为新文档仍按主题组织

## 4. 以后何时再拆

满足下面任一情况，再考虑新增子导航：

### 4.1 已拆 `GitHub开源项目导航.md`

- 现在由这份文档承接 GitHub 仓库类资源
- 主导航中不再重复维护 GitHub 条目正文
- 后续继续新增 GitHub 项目时，优先补到该文档

### 4.2 拆 `AI 学习教程导航.md`

- 教程 / 课程 / 系列指南条目明显增多
- 已经形成稳定学习路径，需要按“入门 -> 进阶 -> 实战”组织
- 教程型资源开始和“工具 / 论文 / 项目”混在一起，影响阅读效率

## 5. 当前维护规则

- 主导航优先按主题组织
- GitHub 仓库独立放在 `GitHub开源项目导航.md` 中，但仍按主题维护
- `AI 在线资源导航.md` 主要维护学习资料、工具平台、社区入口
- 同一条资源只保留一个主落点，避免多处重复维护
- 描述控制在一句话内，只写判断价值，不展开成评测或教程
- 条目不足时保留空位说明，不为了完整感硬凑链接
- 只有当阅读入口、文件职责或拆分边界变化时，才修改 `README.md`

## 6. 推荐维护顺序

新增资源时，按下面顺序判断：

1. 这是不是值得长期保留的外部入口
2. 它属于哪个技术主题
3. 如果它是 GitHub 仓库，优先放到 `GitHub开源项目导航.md`
4. 如果它不是 GitHub 仓库，再判断属于“学习资料 / 工具与平台 / 社区资讯”哪一类
5. 如果多个主题都能归类，选择最主要的那个主题落点
