## 项目架构分析提示词（基础版 vs 详细版）

本文提供两套可直接复用的提示词模板，用于分析任意项目并输出架构说明文档。

- **基础版**：重点输出“模块 + 功能架构”，适合快速理解项目全貌
- **详细版**：重点输出“组件 + 调用关系 + 数据流 + 关键实现”，适合技术深入分析

---

## 一、使用方法（通用）

在使用任一提示词前，先准备以下输入信息（越完整越好）：

1. 项目名称、目标、主要用户
2. 代码仓库结构（目录树）
3. 关键入口文件（启动文件、路由入口、任务入口）
4. 主要依赖（框架、中间件、数据源、消息/任务系统）
5. 运行模式（CLI/Web/API/定时任务/Worker 等）

推荐将输入放在提示词最后：

```text
【项目信息】
项目名：...
项目简介：...
主要目录结构：...
关键文件：...
运行方式：...
补充说明：...
```

## 二、基础版提示词（模块/功能架构）

> 目标：生成“可快速阅读”的项目架构说明，突出模块、职责和模块关系，不下钻实现细节。

```text
你是一名资深架构分析师。请基于我提供的项目信息，输出《项目架构图基础版说明文档》。

【分析目标】
1) 用业务视角和模块视角解释项目，不做源码级细节展开。
2) 明确“这个项目做什么、由哪些功能模块组成、模块如何协作”。
3) 输出一张基础版 Mermaid 架构图（模块级，不展开到过多文件）。

【输出要求】
- 使用中文输出。
- 信息不足时先基于已有信息给出“合理推断”，并明确标注“假设项”。
- 不要编造不存在的技术栈或模块；不确定时写“待确认”。
- 节点命名使用“英文标识/模块名 + 中文说明”格式，例如：`API Gateway<br/>统一接口入口`。

【文档结构】
1. 项目定位
   - 项目目标
   - 解决的问题
   - 适用场景
2. 核心模块（4-8个）
   - 每个模块：名称、职责、输入/输出
3. 模块关系
   - 谁调用谁
   - 谁依赖谁
   - 关键边界（如 API 边界、数据边界）
4. 核心流程（2-3条）
   - 用步骤列出主流程（如“请求进入 -> 核心处理 -> 结果输出”）
5. 技术栈（按层分类）
   - 前端 / 后端 / 数据 / 中间件 / 部署
6. Mermaid 架构图（基础版）
   - 图中仅保留模块级节点
   - 包含主要调用链路
7. 适用读者与阅读建议

【Mermaid约束】
- 使用 `flowchart LR` 或 `flowchart TD`。
- 使用分层 subgraph（如 应用层/API层/服务层/数据层）。
- 节点数量控制在 12-20 个。
- 图要“少而全”：覆盖主模块但避免实现细节。

现在开始生成《项目架构图基础版说明文档》。
```

## 三、详细版提示词（组件/实现关系）

> 目标：生成“可落地到开发协作”的详细架构说明，强调组件职责、调用路径、数据流和关键实现点。

```text
你是一名资深系统架构师兼技术文档专家。请基于我提供的项目信息，输出《项目架构图详细版说明文档》。

【分析目标】
1) 在模块级基础上，下钻到关键组件/关键文件级别（避免无意义穷举）。
2) 解释系统如何运行：入口、编排、核心处理、存储、对外输出。
3) 输出多张 Mermaid 图：总览关系图 + 关键流程图。

【输出原则】
- 使用中文输出，术语准确，面向技术读者。
- 不确定的信息必须标注“待确认/推断”。
- 每个关键节点必须有中文语义说明（可用 `<br/>` 换行）。
- 重点讲“为什么这样设计、关键风险在哪里、如何扩展”。

【文档结构】
1. 模块分解（按层）
   - 应用层、接口层、服务层、领域/核心层、数据层、基础设施层（按实际项目调整）
   - 每层列出关键组件（名称 + 职责 + 关键依赖）
2. 总体关系图（Mermaid）
   - 展示跨层调用关系和主链路
3. 关键组件说明
   - 每个组件说明：职责、输入、输出、依赖、异常处理策略
4. 核心流程（至少3条）
   - 例如：主业务流程、异步任务流程、配置变更流程、导入导出流程
   - 每条流程包含：触发条件、执行步骤、关键数据、失败处理
5. 数据流与状态流
   - 关键数据从哪里来、如何转换、落哪里、如何被消费
6. 技术实现要点
   - 关键设计模式/分层策略/并发策略/容错策略
7. 风险与改进建议
   - 性能、可用性、可维护性、安全性
8. Mermaid 图集
   - 1张总体架构图
   - 2-4张关键流程图（每图聚焦1条流程）

【Mermaid约束】
- 总览图建议 `flowchart TD`，流程图建议 `flowchart TD`。
- 每个节点使用“英文标识 + 中文说明”：
  - `api/app.py<br/>FastAPI应用入口`
  - `task_queue.py<br/>异步任务调度`
- 使用 subgraph 体现分层边界。
- 对不同链路使用 linkStyle 分组（入口链路、核心处理、数据链路）。

【质量检查（输出前自检）】
1) 是否能让新读者在 5 分钟内理解系统主链路？
2) 是否覆盖了关键模块，且没有陷入无关细节？
3) Mermaid 节点是否都有中文说明？
4) 是否明确了风险和可演进方向？

现在开始生成《项目架构图详细版说明文档》。
```

## 四、两种版本的选择建议

- **先用基础版**：用于陌生项目首轮理解、项目评估、汇报展示
- **再用详细版**：用于技术评审、方案设计、开发交接、重构评估
- **最佳实践**：同一项目先产出基础版，再在其基础上扩展详细版，保证叙述一致

---

## 五、核心区别（最终结论）

- **基础版**：聚焦模块与功能架构，强调“看全局、懂边界、知主链路”
- **详细版**：聚焦组件与实现关系，强调“可落地、可协作、可优化”
